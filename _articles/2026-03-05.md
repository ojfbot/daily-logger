## What shipped

### [purefoy] #5 — transcription pipeline v2.0.0

The anchor merge of the day. Four areas of change, all in `transcribe_episodes.py` and `aws_runner.py`:

**Multi-run directory layout** (`transcript/runs/{run_id}/` + `run_manifest.json`): Every transcription run writes to its own subdirectory. A `run_manifest.json` at the root tracks the canonical pointer and full run history. The schema looks like this:

```json
{
  "canonical": "run_20260305_001",
  "runs": [
    {
      "run_id": "run_20260305_001",
      "created_at": "2026-03-05T14:22:00Z",
      "diarization_config": "pyannote-4.0.4/default",
      "status": "complete",
      "episode_count": 12
    }
  ]
}
```

The `canonical` field is the pointer the retrieval layer will read from. A human (or another agent) promotes a new run by updating `canonical` after verification. Re-runs and A/B comparisons between model versions or diarization configs are non-destructive — prior output stays on disk.

**Speaker diarization via pyannote 4.0.4 + speechbrain 1.0.3** (pinned in `requirements-transcription.txt`, commit `31236b7`). A diarized output segment looks like:

```
[SPEAKER_02] 00:04:13 --> 00:04:41
"The way I approach that is really through the lens of available light..."
```

Each speaker turn is labeled and timestamped. `aws_runner.py` now queues undiarized segments for re-processing rather than silently dropping them (commit `230389d`). Security group auto-update and a `--limit` flag for controlled batch runs shipped in the same commit.

**Intro/outro tagging**: The pipeline identifies bookend segments using a timestamp-heuristic — fixed-window rules applied to the first and last N seconds of each episode. This is intentionally coarse. Deakins' format isn't rigidly consistent across 200+ episodes, and a misclassification silently skips substantive content. The heuristic is flagged internally as a correctness boundary: before the retrieval layer reads from this corpus, the tagging logic needs an explicit validation pass against episodes with non-standard openings. The downstream win is real — the agent skips Deakins' standard pre-roll and gets straight to the cinematography content — but the mechanism is brittle and that's acknowledged.

**ADR-005 accepted + docs sweep** (commits `5eaf75e`, `0c4f019`): `docs/` now carries ADR-004 and ADR-005 as accepted records. The audit commit (`0c4f019`) resolved a zero-label speaker conflict. The specific failure: pyannote's default first-speaker label is `SPEAKER_00`, and the undiarized fallback label in the downstream parser was also hardcoded as `SPEAKER_00`. When a segment failed diarization and got assigned the fallback label, the parser couldn't distinguish it from a legitimately diarized first-speaker segment — producing silent attribution errors. The fix renamed the undiarized fallback to `SPEAKER_UNRESOLVED` throughout the parser. This was caught by a human reading transcript output, not by an automated test — which is itself a gap worth noting.

### [purefoy] #6 — Claude Code GitHub workflow

PR #6 added the Claude Code GitHub Actions integration. The follow-on commits document a real debugging sequence:

- `c2c89be` — switched from `GITHUB_TOKEN` to `CLAUDE_CODE_OAUTH_TOKEN` after permission denials. The reason: `GITHUB_TOKEN` is a repo-scoped installation token issued by GitHub Actions — it can read and write repo contents, but it doesn't carry the OAuth credential the Claude Code GitHub App integration requires to call the Anthropic API. They're different auth layers. `CLAUDE_CODE_OAUTH_TOKEN` is the user-level OAuth token that the App is registered against.
- `afe737b` — explicit `allowed-tools: Write,Read` added to the Claude Code action config (`.github/workflows/claude-review.yml`); output surfaced in logs
- `3d1c793` — `Write` and `Read` added to the GitHub Actions runner permissions block (`permissions: contents: write`). These are two separate allowlists: the Claude Code action has its own `allowed-tools` field that controls what tools the model can invoke inside the action; GitHub Actions has its own `permissions` block that controls what the runner token can do in the repo. Setting one without the other leaves a gap — the model can call `Write` but the runner can't commit, or the runner can commit but the model's tool calls are blocked.
- `f70ca05` — scoped to manual dispatch on `main` only; no auto-trigger on PRs yet
- `26784c6` — addressed all findings from the code review pass Claude Code ran on its own first output
- `4e6b684` — **critical fix**: `terminate_instances` was scoped to *all project instances*, not just the current run. Scoped to `run_id` now. See Decision 3.

## The decisions

### Decision 1: Multi-run layout over overwrite-in-place

The previous pipeline wrote transcripts to a flat directory. Re-running with a different diarization config overwrote the prior output. This looks harmless until you want to compare pyannote's output against a future model, or recover from a bad run that completed but produced garbage speaker labels.

The `transcript/runs/{run_id}/` layout treats each run as an immutable artifact. `run_manifest.json` is the pointer store — it knows which run is canonical and the full history. This is the same pattern cv-builder's visual regression pipeline uses for screenshot runs: never overwrite, always compare.

**Why does this matter for the Frame agent?** When purefoy becomes a podcast AI agent inside Frame, the agent will retrieve transcript segments by speaker and timestamp. A flat overwrite model makes that retrieval fragile — a re-run silently changes what the agent sees. The multi-run layout means the agent always reads from a pinned canonical pointer, and a human (or another agent) can promote a new run when it's verified.

This demonstrates **Pillar 2 — Tooling for iteration**: prompts and evals are versioned artifacts. So are transcription runs.

### Decision 2: Pin pyannote and speechbrain; treat undiarized as re-queue, not drop

`requirements-transcription.txt` pins `pyannote==4.0.4` and `speechbrain==1.0.3` explicitly. Diarization models are sensitive to minor version changes in the embedding models that speechbrain ships. An unpinned install on a fresh EC2 runner could produce different speaker clusters than the local dev run that validated the output — a silent correctness regression.

The undiarized re-queue (commit `230389d`) addresses a different failure mode: pyannote sometimes fails to assign a speaker to short segments at turn boundaries. The threshold for triggering a re-queue pass is a minimum segment duration of 0.5s on the first pass; the re-queue pass relaxes this to 0.2s. The re-queue is bounded — a segment that fails both passes is written as `SPEAKER_UNRESOLVED` and does not loop. The `run_manifest.json` canonical pointer updates only after the queue fully flushes, not mid-run; partial re-queue state is tracked in a `requeue_log` field in the run's subdirectory manifest.

**What would break if we dropped instead of re-queued?** The purefoy knowledge base is Roger Deakins answering specific questions. If the question gets dropped and only the answer survives, the AI agent has an answer with no retrievable context. RAG over that corpus would return confident-sounding but unanchored responses. Dropped segments in a transcript are the worst kind of bug — they're invisible in output quality metrics until a human reads the transcript and notices a missing exchange.

### Decision 3: `terminate_instances` scoped to `run_id` — not project-wide

This is the most operationally significant fix in the batch (`4e6b684`), and it's both a trust boundary decision and a cost-control decision — in that order, because the blast-radius framing is what makes the cost consequence obvious.

The original implementation called `terminate_instances()` with a project-level tag filter. On AWS EC2, that terminates every instance carrying that project tag — including any instance from a concurrent run. The failure scenario is concrete: a manual dispatch and a scheduled job fire at overlapping times; the first job's cleanup call kills the second job's running instance mid-transcription. That instance runs until the cleanup call fires or until you notice it on your AWS bill. Neither is acceptable.

The fix stamps `run_id` onto the instance at launch and passes it through to the cleanup call. Two parallel transcription runs now clean up only their own instances.

**The blast-radius audit is a NOW item, not a suggestion.** The article originally listed an `/investigate` action to audit remaining project-scoped tag filters in `aws_runner.py`. That audit has not been completed as of this writing, which means the `4e6b684` fix may be incomplete. Until the audit runs, the confidence level on v2.0.0's AWS safety is "better than before, not fully verified." That's the honest status. The `/techdebt` action below is the first priority for tomorrow morning.

**On cost:** per-episode EC2 diarization cost hasn't been benchmarked against the full Deakins back-catalog. That number needs to exist before parallel runs are treated as a normal operating mode. The Claude Code review cost is pennies per trigger and correctly handled by the manual-dispatch constraint. The transcription pipeline cost is the one that needs a number.

This demonstrates **Pillar 4 — Security as emergent UX**: high-impact agent actions (terminate EC2 instances) must be scoped to exactly the context that triggered them. The fix isn't just operational hygiene — it's the same thinking that scoped MrPlug's frame-agent routing to dev builds only and locked the CORS allowlist to known extension origins.

### Decision 4: Claude Code CI — `CLAUDE_CODE_OAUTH_TOKEN` over `GITHUB_TOKEN`, manual dispatch only

The four CI commits (`c2c89be`, `afe737b`, `3d1c793`, `f70ca05`) document a real debugging sequence. `GITHUB_TOKEN` doesn't carry the permissions Claude Code needs to call the Anthropic API through the GitHub App integration — it's a repo-scoped token, not an OAuth credential. Switching to `CLAUDE_CODE_OAUTH_TOKEN` unblocked the runner.

Manual dispatch on `main` is a deliberate constraint, not a limitation to be removed later. Auto-triggering Claude Code review on every PR in purefoy — a repo that runs paid AWS jobs — would introduce unbounded cost and latency into the PR loop. The right shape is: Claude Code review fires when a human decides a PR warrants it, not on every push.

This is **Pillar 3 — Model behavior as design discipline**: the decision about *when* the model runs is as important as the decision about *what* it does when it runs.
