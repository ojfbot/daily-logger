### Babylon.js migration opens as a draft against a fully-closed Three.js Phase 0 — the existing src/scene/ files become behavioral specs, not migration targets — *model-behavior-as-design*

Draft PR #28 begins the pure-Babylon migration after core #94 recommended Babylon over Three.js for web-native 3D. The critical architectural principle: every existing Three.js scene file is treated as a behavioral specification for what the Babylon equivalent must reproduce — not as code to port line-by-line. A line-by-line port would import Three.js concepts (Object3D, MeshStandardMaterial, OrbitControls) that have no Babylon equivalent, producing a hybrid that is harder to maintain than either pure implementation. Operationally, this means src/scene/ stays in the repo during the migration as a reference corpus — it is not moved or deleted until each file has a corresponding Babylon behavior that passes the same assertions. The first concrete task once #27 is dispositioned: open #28, add a checklist mapping each Three.js scene file to a behavioral assertion it must satisfy, and treat that checklist as the migration's definition of done. The Three.js playability-pass PR (#27) sits in an ambiguous state: its heightAt(x,z) bilinear-blend fix documents what correct terrain sampling looks like, which is exactly the behavioral spec the Babylon migration needs. This principle should be in a standalone ADR — if it currently lives only in the PR description of #28, that is the gap to close.
