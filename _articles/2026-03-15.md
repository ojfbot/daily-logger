## The decisions

**Decision under review: merge order for [shell] #31 and [shell] #32**

These two PRs were opened the same day (2026-03-12) and are functionally complementary, but they're not symmetric in risk.

[shell] #32 — the cross-domain fan-out path — is the deeper change. It extends `MetaOrchestratorAgent` to fan out a single natural-language query across cv-builder, TripPlanner, and BlogEngine in parallel, then synthesize isolated responses into a single reply. This is the architectural heart of the hero demo: `"I'm applying for Berlin jobs, how does this affect my trip plan and my resume?"` resolves across three agents simultaneously.

> **Why fan-out instead of sequential chaining?**
> `max(agent_latencies)` vs. `sum(agent_latencies)`. If each agent takes 2s, sequential chaining gives you 6s minimum; fan-out gives you roughly 2s regardless of how many domains you hit. More importantly, sequential chaining creates implicit data dependencies that don't exist here: the trip plan response doesn't need the resume response to form. Isolated synthesis means each domain answers from its own context, and `MetaOrchestratorAgent` stitches the three payloads at the end — concretely, by passing all three responses to a final synthesis LLM call that produces a single unified reply. Reviewing #32 before #31 merges is how you confirm that synthesis call's response envelope is what the connected wrappers expect to consume.

[shell] #31 — the wiring PR — is safer but has a subtle dependency: it imports `HeaderConnected` and `HomeScreenConnected`, which wrap the extracted `@ojfbot/shell` components merged in [shell] #29 and [shell] #30. Those are already merged, so #31 has no outstanding code dependency. The question is whether the connected wrappers correctly resolve Redux state across the Module Federation boundary — the failure mode tracked in [shell] #5.

> **The Redux boundary failure mode — and why it's the first thing to watch after #31 merges:**
> Module Federation loads sub-apps as separate JavaScript bundles at runtime. If `HeaderConnected` imports Redux selectors from the shell's store while the sub-app has its own store instance, you get two independent store trees — state updates in one don't propagate to the other. The symptom is specific and observable: the header renders correctly on initial load but doesn't reflect navigation changes driven by the sub-app. This is exactly the failure mode [shell] #5 was opened to document.
>
> **[shell] #5 disposition:** It is open as an issue, not a PR, and it has no proposed code fix. The posture is deliberate: [shell] #31's first deploy is the live test. If the boundary is clean, #5 closes. If it isn't, the symptom surfaces immediately — the header won't update on sub-app navigation — and the remediation is to patch the store singleton export so both the shell and sub-apps reference the same instance rather than initializing independently. That patch is a targeted change to the store composition layer; it doesn't require rolling back #31. **This is a speed bump, not a Sunday-into-Monday blocker** — but "worth watching on first deploy" is now an explicit test, not a hope: trigger a sub-app navigation event after merge and confirm the header reflects it.

**Why [shell] #32 has been open since Thursday without review:**

The Sunday merge window is intentional, not drift. The last two weeks of this project have run a Thursday-open / Sunday-merge cadence for architectural PRs — Thursday gives the PR time to stabilize, Saturday is the reading and audit day, Sunday is the merge window with a clear pre-review checklist. [shell] #32 is on that cadence. It hasn't moved because Sunday hasn't happened yet.

**What a passing review of [shell] #32 looks like:**

Three specific checks: (1) the synthesis response envelope includes a top-level `reply` field that `HeaderConnected` can render without transformation; (2) the timeout boundary on the parallel agent calls is explicit — if one downstream agent exceeds the budget, the synthesis step proceeds with the available responses rather than failing the whole request; (3) the error envelope has a `partial: true` flag so the UI can signal degraded results rather than a hard error. A fail on any of these means a contract amendment before #31 merges. A pass means #31 can merge immediately after.

**Decision: don't merge #31 before #32 is reviewed**

The dependency flows in one direction: #32 is server-side (frame-agent), #31 is client-side (shell-app). They're independent at the code level. But if #32's review surfaces a contract change — a different response shape, a revised timeout boundary, an error envelope amendment — #31's connected wrappers may need to be updated to match. Reviewing #32 first costs nothing and eliminates that risk.

The underlying discipline here: define the API contract before wiring the UI. This is the same engineering posture as specifying a browser API before shipping an extension SDK — the contract surface is what both sides of the boundary program against, and changing it after one side is shipped creates a correction cycle that costs more than the review would have.
