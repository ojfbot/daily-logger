## The decisions

### Decision 1: Per-domain history scoping in `fanOut()` (ADR-0019)

The core architectural bet in `frame-agent` is that a single ShellAgent can orchestrate multiple sub-app agents in parallel and synthesize their responses. The danger is **context bleed**: if the MetaOrchestrator shares one history buffer across all domain agents, responses from cv-builder's session can silently pollute TripPlanner's synthesis window.

The fix in `72fb2e4` and `c1dc80e` enforces that `fanOut()` calls `historyFor(domain)` for each agent invocation, inside `packages/frame-agent/src/fanOut.ts`. Mechanically, `historyFor(domain)` filters the agent's conversation store by domain key — each domain maintains its own array of message objects, scoped by the domain identifier passed at invocation time. The MetaOrchestrator sees only what's relevant to its synthesis task.

The integration tests in `72fb2e4` are assertions at the orchestrator boundary: they fire a real cross-domain `fanOut()` call (e.g., cv-builder + BlogEngine in parallel) and assert that the MetaOrchestrator's synthesis window contains zero messages from the non-target domain. The fallback case from `c1dc80e` is covered by a separate fixture that forces the edge-case code path and makes the same assertion. When these tests are green, the isolation guarantee holds end-to-end — not just for the happy path.

> **Why not just clear history between calls?** Clearing history would lose session continuity — a user who says "update my resume AND add it to my trip packing list" needs both agents to have their respective prior context. Isolation, not erasure, is the right primitive.

> **What breaks without this?** The failure is subtle: the model produces a plausible-sounding answer that mixes facts from two unrelated app sessions. There's no exception, no visible error. The user just gets wrong information. This is the class of LLM bug that's hardest to catch without explicit integration tests — which is exactly why `72fb2e4` ships them alongside the fix.

This directly demonstrates the **security as emergent UX** pillar: the isolation boundary isn't enforced at the UI layer, it's enforced at the agent history layer, where the actual trust boundary lives.

### Decision 2: lean-canvas port assignment (3025/3026, not 3020/3021)

The port collision between lean-canvas and purefoy was a real sequencing risk. Both apps were being registered around the same time, and the initial lean-canvas registration (`c19b6f8`) grabbed 3020/3021 — purefoy's ports. `7fecb6a` corrected this across the entire cluster config.

> **Why does port assignment matter in a dev environment?** Module Federation remotes are registered by URL. If two remotes share a port, the second one to start silently wins, and the first app becomes unreachable without a restart. In a multi-app Frame OS session this is a hard-to-debug failure — the shell connects to the wrong bundle without any error.

### Decision 3: ADR-0007 distinct per-tool endpoints across all sub-apps

BlogEngine and TripPlanner both shipped `GET /api/tools` with per-tool endpoint descriptors Monday. This isn't cosmetic — it's the contract that lets ShellAgent discover each sub-app's capabilities at runtime without hardcoded knowledge.

A minimal response from a compliant `GET /api/tools` endpoint looks like this:

```json
[
  {
    "name": "createPost",
    "description": "Create a new blog post draft",
    "endpoint": "/api/tools/createPost",
    "method": "POST",
    "parameters": { "title": "string", "body": "string" }
  }
]
```

When ShellAgent needs to route a user request, it calls `GET /api/tools` on each registered remote and receives a manifest of what that app can do, what endpoint handles it, and what parameters it expects — the same pattern as Claude's tool-use protocol. Without distinct per-tool endpoints, ShellAgent would have to hardcode routing logic for every sub-app, which breaks the moment a new app is registered.

**cv-builder has not yet shipped this contract.** Until it does, ShellAgent's runtime capability discovery for Resume Builder falls back to hardcoded routing. This is a live gap, not a cleanup item — if you're trying to use ShellAgent to route a resume-related prompt through capability discovery, that path does not work today. The ADR-0007 rollout is 2 of 4 apps complete; cv-builder and shell are the remaining surfaces. Neither is blocked on external dependencies — they are prioritization items for the next sprint.

This is **assistant-centric architecture**: the shell doesn't know what apps can do at compile time. It discovers capabilities at runtime through a standard contract.

### Decision 4: purefoy TypeScript layer as a read-only knowledge browser first — and what the overnight batch fixed

purefoy's Phase 1 UI is deliberately read-only. The Python corpus (347 episodes, scraped forum data, embeddings) is the source of truth. The TypeScript layer adds navigation and search without touching ingestion.

> **Why read-only first?** The corpus pipeline is the expensive, slow-moving part — spot EC2 transcription jobs, SSO retry loops, embedding generation. Keeping the UI layer completely decoupled from the write path means the knowledge browser can ship and iterate on its own cadence. The eventual Frame OS integration (purefoy as a podcast AI agent inside Frame, Phase 7) connects to this read layer, not to the Python pipeline directly.

The overnight 347-episode batch ([purefoy] #10) exposed two failure classes worth documenting, because Phase 7 will depend on this pipeline being reliable:

1. **Detached execution** — The original batch script blocked the SSH session. If the session dropped mid-batch, the job died. The fix detaches the process so it survives session termination.
2. **SSO retry logic** — AWS SSO tokens expire during long-running batches. Without retry logic, the job would silently fail partway through when the token expired, leaving a partial corpus with no error surfaced. The retry loop re-authenticates and resumes from the last successful checkpoint.

These aren't edge cases — they're expected failure modes for any multi-hour spot instance job. They're documented here because the person running the next batch needs to know what the failure looks like if the retry logic is absent.

The overnight batch cost is not tracked in this article — spot `g4dn.xlarge` pricing is approximately $0.526/hr per instance; three instances over an estimated 6-hour batch is roughly $9.50. This is a rough estimate. Actual cost accounting is not currently instrumented.

### Decision 5: 'Start Fresh' and why that label matters

`'My Resume'` → `'Start Fresh'` in `991efd9` is not rename sweep cleanup. It reflects a deliberate onboarding framing: the empty state of a new Resume Builder instance should communicate that the user is beginning something, not that they already have a document named "My Resume." The decision originated in [cv-builder] #117 and was applied consistently across shell and agent system prompts. The label signals intent at the moment of first interaction — this is model behavior as design, applied to micro-copy.

### On [shell] #5 and why it has been open this long

[shell] #5 (Frame-wide Redux store composition) has been open throughout Phase 1 while other Phase 1 work shipped around it. The honest answer is that it was a sequencing call: the hero-demo PR ([shell] #32) needed to prove that cross-domain fan-out worked before it was worth investing in a shared store architecture. If fan-out had required a different isolation model, the store composition approach might have changed. With ADR-0019 now locked down and the fan-out pattern proven, the store composition question has a stable foundation to answer against. It wasn't blocked by a technical dependency — it was held back by epistemic caution about specifying store boundaries before the orchestration model was settled.
