## The decisions

### Deterministic action IDs from content hash, not wall-clock timestamp — *tooling-for-iteration*

Actions now derive their ID from a hash of their content (command + description + repo + date) rather than a random suffix generated at write time. The hash function is applied in `daily-logger` at ID-generation time; the specific algorithm is SHA-based and stable across runs given identical field values — whitespace normalization is applied before hashing to avoid encoding drift. This means the same action, re-generated in a subsequent run before it has been resolved, produces the same ID, enabling the closure pipeline to match against it reliably. One known edge case: if two actions legitimately share all four fields, they hash to the same ID. In practice this hasn't occurred — action commands are specific enough to be unique within a repo+date — but it's a known collision surface. An ADR for this scheme is not yet written; that is an open gap acknowledged below. What happens when an action's content legitimately changes before resolution? It generates a new ID and is treated as a new action; the old ID drifts open until manually closed. That behavior is intentional: a changed action is a different commitment.

### Suggestion-ignored telemetry closes the forward half of the feedback loop — *model-behavior-as-design*

The suggest-skills engine already logged when a suggestion was followed. Monday's `e1a4d8d` added the complementary `ignored` disposition so we can distinguish 'surfaced but irrelevant' from 'never surfaced'. Without the ignored signal, a 0% adoption rate is indistinguishable from 0% suggestion coverage — two very different failure modes with very different remedies. The telemetry currently writes to `~/.claude/suggestion-log.jsonl`, a local dotfile. This is the right scope for a single-developer workflow. The aggregation story for multiple users is explicitly unresolved: the log path would need to be configurable or forwarded to a shared sink. That's a known limitation, not a hidden assumption.

Tags: `assistant-centric`, `model-behavior-as-design`, `core`

### jq --slurp required for multi-line JSONL aggregation in pr-skill-audit — *tooling-for-iteration*

The `pr-skill-audit` hook was invoking `jq` on a JSONL file without `--slurp`. Without that flag, `jq` treats each newline-delimited JSON object as a separate, independent input document and runs the filter against each one independently — so an aggregation filter like `map(.field)` only ever sees one object at a time and silently produces wrong output. Adding `--slurp` wraps all lines into a single array before the filter runs, matching the intended aggregation semantics. The failure was silent because `jq` exited 0 and produced output — just wrong output. There was no error signal.
