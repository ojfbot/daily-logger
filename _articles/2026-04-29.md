## The decisions

### Discard ~2,500 lines of Three.js and migrate to Babylon.js — *model-behavior-as-design*

Core PR #94 evaluated three axes: web-native constraints (Three.js requires manual WebGPU shimming; Babylon ships with it), physics integration depth (Babylon's Havok plugin offers deterministic rigid-body physics with less glue code than Rapier-over-Three), and MCP tooling support (Babylon's scene inspector exposes a structured API that an MCP server can query without custom instrumentation). Three.js failed on all three counts under the project's constraints. The recommendation was accepted and beaverGame #28 opened as a pure rewrite. The decision record lives in core #94, which has not yet merged — that is the single most urgent merge in the project right now. Without it, the rationale for discarding 2,500 lines of working code exists only in a draft PR description and will need to be reconstructed in future sessions.

### Treat the existing Three.js scene as a test oracle, not legacy code — *tooling-for-iteration*

Rather than porting code line-by-line, the migration uses each existing Playwright snap rig (testFellNearestTree, testBuildDam, etc.) as a pass/fail oracle for the Babylon equivalent. This sidesteps the port-vs-rewrite dilemma: the behavioral contract is already machine-checkable. One mechanical tension to name explicitly: window.__beaver is now gated behind import.meta.env.DEV (see next decision), and the snap rigs drive those same window.__beaver handles. The rigs must run in a DEV build — they are not intended for production — so the gating is consistent. What the rigs actually assert: they call window.__beaver methods (e.g., testFellNearestTree()) which execute scene logic and return structured JSON that Playwright then validates against a snapshot. They are not pixel-diff tests. This means the oracle captures behavioral state (tree count, stump presence, particle spawn flag) rather than visual appearance — the feel of the felling mechanic, camera shake timing, and particle density are not covered by the current rig contract. That gap is real and should be tracked as a follow-on concern for the Babylon port.

### Gate window.__beaver debug rigs behind import.meta.env.DEV — *security-as-emergent-ux*

The test scaffolds that Playwright drives are only useful in development. Shipping them in production exposes internal scene state (tree positions, player coordinates, dam construction state) to the browser console. The threat model is not a sophisticated attacker — it is unintentional information leakage in a multiplayer or leaderboard context and the cost-of-reversal problem if the API surface grows before anyone notices. A single if (import.meta.env.DEV) guard keeps the snap-and-diff CI pipeline working while eliminating the production surface. This is hygiene, not a hardened security boundary — calling it security-as-emergent-ux is accurate because the guard is invisible to the user but structurally correct.

### Extract gateValidation as a pure function for direct unit testing — *tooling-for-iteration*

The validator in asset-foundry previously mixed I/O and validation logic, making the rejection-path branches untestable without a full orchestration run that required a live Blender process and a real file system. Extracting gateValidation as a pure function in src/validator/index.ts means each of the four rejection paths — missing validation JSON, rejected status, malformed JSON, schema mismatch — can be exercised with a single argument in a Vitest unit test. No Blender process. No file system. The fix also improves the design: the I/O boundary is now explicit, and future callers can call the validation logic without owning the orchestration lifecycle.

### asset-foundry evolves from build script to portable MCP platform (PR #16) — *assistant-centric*

PR #16 proposes four phases that take asset-foundry from a single-purpose Cozy Beaver pipeline to a game-agnostic asset platform with three front doors: CLI (existing), Frame MF browser app (asset-foundry sub-app at :3035, registered via shell #76), and MCP server exposing domain-level verbs. The hybrid MCP shape means asset-foundry both consumes Anthropic's first-party Blender MCP (for mesh generation and scene manipulation) and exposes its own domain verbs (e.g., validate_asset, query_manifest, get_palette) that a game-tooling agent can call without knowing Blender internals. This is the canonical pattern for future game-tooling agents in the ecosystem. What this does that a plain REST API does not: MCP verbs are discoverable and composable by the assistant at runtime without hardcoded endpoint knowledge — the agent can inspect the server's capability list and construct multi-step workflows. The dependency chain is: shell #76 (one-line host config) must land before the Frame MF front door is visible in dev; core #92 (asset-foundry editor spec) defines the interaction model for the browser app. Neither is blocking PR #16 from being authored, but both must merge before Phase 1 of the platform roadmap can close.

### Cross-repo asset imports: sibling-repo path dependency, not a package — *tooling-for-iteration*

beaverGame now imports the world manifest, fixtures, and palettes directly from the sibling asset-foundry repo via a TypeScript path alias that assumes both repos are checked out side-by-side in the same parent directory. This is not a published package, a workspace dependency, or a symlink — it is a path alias in tsconfig.json that resolves to ../asset-foundry/src/.... This works in local dev with a standard side-by-side checkout and in CI where both repos are cloned into the same workspace. What breaks: any checkout where the repos are not siblings (e.g., beaverGame checked out alone, asset-foundry at a different relative path, or a consumer running beaverGame in isolation). This is a deliberate tradeoff — it avoids a publishing pipeline while the two repos are co-evolving rapidly — but it is a real coupling that should be revisited when asset-foundry's public API stabilizes.
