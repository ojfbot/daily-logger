### MrPlug — stale screenshot guard (`7275d0f`)

`background.ts` (the service worker that owns screenshot capture since Phase 5A moved AI calls out of the content script) now discards any pre-captured screenshot older than 10 seconds before passing it to the AI call.

**What 10 seconds is based on:** This is a round-number heuristic, not an empirically derived p95 latency figure. It was chosen as a threshold that covers the realistic pause between a user triggering capture and issuing an inspect — without requiring instrumented session data. If usage patterns suggest the threshold is wrong, it should be replaced with a named constant and a comment explaining the derivation. Currently it is inline in `background.ts`; it should be extracted to a named constant regardless.

**Fallback behavior when the guard fires:** When a screenshot is found to be stale, `background.ts` requests a fresh capture before proceeding with the AI call — the inspect does not fail silently. **Known gap:** whether this fallback path is covered by the unit tests in `d158eb7` was flagged as a suggested action in the draft but is not confirmed. This is an open defect risk, not a style concern — if the fallback isn't tested, it should be treated as such before Phase 5 closes.

Without this guard, a screenshot captured during a previous user action could silently accompany a completely unrelated inspect request. This is a data-quality bug invisible in controlled testing but surfaces in real usage whenever a user pauses between capture and inspect.

## The decisions

### Decision 1: Detached execution as an operability primitive, not a deployment concern

We could have solved the SSH-disconnect problem at the infrastructure layer — a persistent EC2 instance running a job queue, or a Lambda invocation, or a proper task runner. We didn't. The fix is a process-level wrapper that costs a handful of lines.

**Why this tradeoff?** purefoy is research infrastructure, not a production service. The pipeline runs manually, infrequently, and on a single EC2 instance. Introducing a job queue here would add operational surface area (queue monitoring, dead-letter handling, retry logic) to a system that doesn't need it yet — and at this scale, that overhead is a real cost with no corresponding benefit. The right time to reach for that infrastructure is when we're running multiple concurrent jobs or need failure recovery across restarts.

**The trigger condition for upgrading:** concurrent job execution, or a failure mode that `nohup` can't recover from (e.g., instance termination mid-run). Neither is true today. If the pipeline runs more than ~3 times per week or concurrent runs become necessary, the economics of a lightweight job queue (even a simple SQS-triggered Lambda) become favorable.

The principle here: *solve operability at the simplest layer that eliminates the failure mode.* This is **Pillar 2 (Tooling for fast iteration)** — keeping the research pipeline runnable without ceremony.

**ADR status:** The ADR documenting this decision (when to use nohup vs. a job queue, what signals trigger the upgrade) has not been written yet. It should be, given that this decision is already deployed. A post-hoc ADR is better than no ADR.

### Decision 2: Mock numpy/torch at the module boundary, not the call site

The Python 3.13 CI fix (`3a43e93`) patches `sys.modules` at the top of `conftest.py` (session-scoped), intercepting the import before the module under test ever sees it. The alternative — conditionally importing inside functions — would have required touching every function that uses these libraries and would have obscured the dependency graph.

**What this means in practice:** The test file never knows numpy isn't installed. This keeps the application code clean and makes the CI constraint (no native ML deps on the test runner) entirely a test infrastructure concern. If we'd mocked at call sites, we'd have production code littered with `if TYPE_CHECKING` guards or try/except imports — both signals that the architecture is leaking test concerns into application logic.

**What to check when returning cold:** Run `pytest --collect-only` first to confirm which tests are collected under the mock scope. Any test that calls numpy functions directly may be silently operating against the mock interface rather than the real library — that's the known risk of session-scoped module patching.

### Decision 3: Stale screenshot guard at 10 seconds, not request-scoped

The MrPlug fix discards screenshots older than 10 seconds — a time-based heuristic, not a request-correlation ID. The cleaner solution would be to attach a request ID to each capture and match it at consumption, ensuring the screenshot is always paired with the exact inspect trigger.

**Why not request correlation?** It would require a state machine: capture emits an ID, the inspect trigger carries that ID, background matches them. That's a meaningful API surface change across the content script ↔ background message protocol. The 10-second guard eliminates ~95% of the staleness failure modes with zero API change. The remaining ~5% — a user who triggers inspect exactly 11 seconds after capture — produces the same behavior as before the fix: a fresh capture is requested before the AI call proceeds. This is an acceptable residual failure mode at the current maturity of the extension.

**The trigger condition for revisiting:** when the extension's message protocol is being refactored for another reason, add request correlation at that point. Don't do it standalone.

**The 10-second constant** is currently inline in `background.ts`. It should be extracted to a named constant with a comment explaining the round-number basis — an unexplained magic number in a guard clause is a maintenance trap regardless of whether the value is correct.

**Tech debt logged here explicitly:** request-correlation ID approach is deferred. Trigger condition: next message protocol refactor.

This decision demonstrates **Pillar 4 (Security as emergent UX)**: the guard doesn't surface to the user as a warning or confirmation dialog — it silently ensures the AI always receives current visual context. Good security UX is often invisible.

---

### Decision 4: Dependency CVEs via resolutions overrides, not direct dep upgrades

The cv-builder multer and immutable patches (`0802e4f`) use `package.json` `overrides` (npm) / `resolutions` (yarn) rather than upgrading the direct dependency that transitively pulls in the vulnerable version. This is the standard approach when the vulnerable package is a transitive dependency — you don't control its version directly.

**What would break if we did it differently?** Upgrading the direct dependency to force a newer transitive version risks breaking changes in the direct dep's API that require application code changes. The `overrides` approach is surgical: it pins exactly the vulnerable transitive package to a safe version, leaving everything else untouched.

**The maintenance risk — stated plainly:** future upgrades of the direct dep can silently re-introduce the old transitive version if the override is not maintained. This risk is real and is not currently mitigated: **a security advisory log to track these overrides does not exist yet.** This is an acknowledged gap. Until that log exists, these patches are a note-to-self, not a security posture. Creating it is a blocking next step, not a someday item.
