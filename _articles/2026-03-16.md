The two hero-demo PRs are independent in their file surfaces but tightly coupled in their demo dependency chain.

**[shell] #31** replaces direct local imports of `ShellHeader` and `HomeScreen` in `shell-app` with `HeaderConnected` and `HomeScreenConnected` wrappers that import from `@ojfbot/shell`. **[shell] #32** implements the cross-domain fan-out path in `MetaOrchestratorAgent`, routing a single natural-language query — *"I'm applying for Berlin jobs, how does this affect my trip plan?"* — to both cv-builder's job-analysis agent and TripPlanner's domain tools, then synthesizing isolated responses into a unified reply.

Why does merge order matter? If [shell] #32 lands before [shell] #31, the fan-out executes correctly but renders into unverified local components. You can't trust the visual result. The demo story requires the orchestration layer and the presentation layer to be proven simultaneously — and the concrete failure mode if you skip the order is that the fan-out query succeeds but the render path is still backed by the old local import, not the package boundary. Silent success against the wrong import path is harder to catch than a thrown error.

### Why props-driven with zero Redux inside the package?

The shared package must be consumable by any sub-app, including remotes that run in separate Module Federation scopes with their own Redux stores. If `Header.tsx` imports from the shell's Redux store directly, it becomes un-importable from cv-builder or BlogEngine without bundling the shell's entire state graph. Props-in, callbacks-out is the only safe contract at a federation boundary. Every future extraction follows this rule — this is the constraint the `/adr shell` action item is meant to capture in a decision record that travels with the codebase, not just in this article.

### The isolated synthesis pattern in [shell] #32: design intent and the enforcement gap

The isolation design is deliberate: each sub-agent's context window is kept separate, then the orchestrator merges at the output layer. This maps directly to the **assistant-centric architecture** pillar — ShellAgent orchestrates without collapsing domain boundaries.

Why does isolation matter? cv-builder's resume context includes personal work history. TripPlanner's context includes location data and itinerary state. If both are concatenated into a single prompt before synthesis, the model has no reliable signal for which domain owns which constraint. Cross-domain contamination is a real failure mode, not a theoretical one.

**The enforcement gap:** the article cannot confirm whether the isolation is enforced mechanically (separate API calls, a wrapper that strips state before sub-agent dispatch, a prompt-level fence) or whether it is currently a convention that a future contributor could accidentally break. The `/adr shell` action item covers the props-driven contract; an equivalent ADR for the orchestration pattern is the harder and more consequential decision that still needs a written record. Someone debugging a contamination failure at 11pm needs to know which file to open and what invariant to check.

**On LLM call cost:** one user query to `MetaOrchestratorAgent` potentially triggers one call to the cv-builder agent, one to TripPlanner, and one synthesis pass — potentially four round-trips. The per-query cost and latency budget at scale are not documented anywhere in the current stack. This is a gap worth naming: the isolated synthesis design is an economic decision before it is an architectural elegance decision.
