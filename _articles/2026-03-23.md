## The decisions

### Decision 1: Flask reference implementation before TypeScript client (purefoy #13 → #14)

We shipped a standalone Flask UI (PR #13) one PR before the TypeScript client port (PR #14). That ordering is deliberate and worth naming.

The purefoy data model — episodes, transcripts, goal manifests, review progress — is not yet stable. A single-file Jinja2 app lets you validate the API surface and the editing UX against real corpus data without the overhead of TypeScript types, Redux slices, and Module Federation wiring. Once the UI was proven correct in Flask, porting to TypeScript was a translation exercise, not a design exercise. The alternative — building the TypeScript client directly — would have coupled API design decisions to the full Frame stack before the endpoints were known to be right.

**On whether Flask actually validated the shared types:** `GoalManifest` and `ReviewProgress` as named shared types emerged during the TypeScript port in PR #14, not from the Flask prototype. The Flask app validated the API *surface* — what data the endpoints needed to return — but it did not produce stable named types. The Flask-first rationale holds for endpoint design; it does not guarantee that the type names in `shared/` are final. `reviewCoverage` is computed in the `api/` package server-side; the Flask app reads raw data and does not re-implement the calculation, so there is no drift risk there. But the types themselves should be treated as first-draft stable, not permanently settled.

The Flask app is a local-only debugging surface (`localhost:5050`, `python app.py`). It is not deployed. When the Frame TypeScript client behaves unexpectedly, you run the Flask app and rule out data issues against the raw corpus directly. That's the value — not permanence in the hosted sense.

**TBCoNY pillar: Tooling for fast iteration.** The Flask reference made the production client faster to write and safer to ship. Same discipline as the visual regression pipeline: instrument the loop before you optimize the product.

---

### Decision 2: `GoalManifest` + `ReviewProgress` as first-class shared types (purefoy #14)

These two types live in the `shared/` package. `EpisodeListItem` carries computed fields (`hasGoal`, `reviewCoverage`) derived from them. `reviewCoverage` is a float computed server-side in the `api/` package — the client doesn't need to know how goals map to transcript segments, only the derived value.

Any consumer of the purefoy API — the Frame TypeScript client, a future Claude agent, a CLI tool — gets the same typed contract. This is the same boundary discipline ADR-0029 applied to the shell's container/presenter split: push the computation to the layer that owns the data, expose only the derived value.

If `GoalManifest` lived only in the Flask app, the TypeScript client would have to re-implement the coverage calculation — and the two implementations would drift. They don't, because the Flask app doesn't compute coverage at all; it reads raw data. The `api/` package is the single owner of that calculation.

### Decision 3: MrPlug writes tech debt to frame-agent, not to a local file

The `file-techdebt` handler (MrPlug PR #30 + shell PR #46) routes through frame-agent's HTTP endpoint rather than writing directly to a file on disk.

MrPlug is a Chrome extension — it runs in a sandboxed renderer process with no filesystem access. The only write path available is HTTP. But the architectural reason matters more than the constraint: routing through frame-agent means the endpoint is the single consolidation point that prevents every sub-app from spawning its own write path to the filesystem, its own API surface, its own incident format. This is the enterprise integration bus pattern applied to a dev toolchain — not just a security decision, but a sprawl-prevention decision. As Frame adds sub-apps 6, 7, and 8, each one gets the same controlled write path without negotiating its own file access.

**On what "project mapping" means right now:** The routing logic that determines which repo's `TECHDEBT.md` receives a given incident is currently implicit in the handler. MrPlug's transform step resolves repo, location, and severity from the AI analysis before sending — but this resolution is not formally specified. The fallback to `core/TECHDEBT.md` silently accepts misfiled entries. The ADR for this write path is a NOW item, not a follow-up. See the data integrity note in What Shipped.

**TBCoNY pillar: Security as emergent UX.** "MrPlug observes pages it didn't build. Its output goes through a controlled write path before touching the repo." The agent layer between MrPlug and the filesystem is the trusted/untrusted separation the architecture requires.

### Decision 4: SEH Study registered at the shell layer before the app is built

Shell PR #47 creates stub registrations across all 11 integration points — `SehStudyApp` is a placeholder component. The app doesn't exist yet in any meaningful sense.

**Why 11 touch points, and why register a stub?** The 11 points are: `AppType`, `APP_CONFIG`, `REMOTE_LOADERS`, `SETTINGS_LOADERS`, domain registry, meta-orchestrator, Vite MF remote config, and the app component stub (plus the `frame-dev.sh` entries and domain-knowledge inventory). That count reflects an explicit registry architecture — every location that needs to know about an app must declare it — rather than implicit discovery. The cost is upfront verbosity; the benefit is that there are no silent partial registrations. The `/scaffold-frame-app` skill generates all 11 touch points from a single command, produces a PR-ready diff, and requires no manual edits for a standard sub-app configuration. Port conflicts are the one failure mode: if the target port is already claimed in `frame-dev.sh`, the scaffold will collide and that collision will be visible immediately.

Registering the stub before the app is built means the dev script, port registry, and shell config are always in sync from day one. The cost of wiring 11 touch points is paid once; every subsequent PR adds capability to an already-registered app.

**On whether more integration points are the goal:** The intended end state is better tooling over the same 11, not fewer points. The explicit registry is load-bearing for the Module Federation architecture — Vite needs the remote config at build time, the shell needs the loader at runtime, the domain registry needs the type at orchestration time. These are not redundant; they operate at different layers. `/scaffold-frame-app` is the answer to the coordination cost, not a reduction in the registry surface.
