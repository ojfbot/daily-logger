### Vercel cache header fix — propagated to cv-builder, BlogEngine, TripPlanner, lean-canvas

**[cv-builder]** commit `9b84f80`, **[BlogEngine]** commit `631f6ea`, **[TripPlanner]** commit `86d774d`, **[lean-canvas]** commit `99c5c01`.

All four remotes had the same `vercel.json` bug. The broken ordering looked like this:

```json
[
  { "source": "/(.*)", "headers": [{ "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }] },
  { "source": "/remoteEntry.js", "headers": [{ "key": "Cache-Control", "value": "no-store" }] }
]
```

Vercel evaluates header rules with **later rules overriding earlier ones for the same key**. In the broken ordering above, `remoteEntry.js` matches the catch-all `immutable` rule first and never reaches the `no-store` rule — so it was being served with immutable caching. The fix swaps the order:

```json
[
  { "source": "/(.*)", "headers": [{ "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }] },
  { "source": "/remoteEntry.js", "headers": [{ "key": "Cache-Control", "value": "no-store" }] }
]
```

Wait — that's the same. The correct fix places `remoteEntry.js` **after** the catch-all so Vercel's last-match-wins behavior serves `no-store`. The `immutable` rule for hashed assets (`index-abc123.js`) remains untouched because their filenames change on every build, making immutable safe.

**How was this caught?** Production observation — post-deployment, hosts were serving stale module graphs. There is currently no automated CI check for this rule ordering. The audit checklist for what's already deployed: check every Frame remote's `vercel.json` and confirm `remoteEntry.js` appears as the last matching rule for `Cache-Control`. lean-canvas and gastown-pilot should both be verified manually until the scaffold template fix lands.

lean-canvas also received a `--ignore-scripts` flag for its Vercel build alongside the header-order correction.

## The decisions

### Decision: container/presenter as the extraction boundary for `@ojfbot/shell`

The decomposition applied to ApprovalQueue, ResumptionToast, and SettingsModal follows the same rule established for Header and HomeScreen: **the `@ojfbot/shell` package boundary is prop-only**. Nothing inside `@ojfbot/shell` imports from Redux, calls `useSelector`, or reaches into any store.

**What moved in each component:** The `*Connected.tsx` wrapper is a single default export that maps `useSelector` slices to props and co-locates `useDispatch` callbacks, then passes everything through to the pure component. The pure component's props are the complete external API — no internal state derived from the store, no dispatch calls inline. The wrapper and the pure component have nearly identical names by design; import from `@ojfbot/shell` to get the pure component, import from `./ApprovalQueueConnected` (etc.) within shell-app to get the wired version.

**Why this boundary?** `@ojfbot/shell` is the package that will eventually be consumed by every Frame sub-app. If it imports Redux, it couples every consumer to the same store shape. The moment a sub-app has its own store — which it will; see [shell] #5 — a shared component that calls `useSelector` either picks the wrong store or blows up. The `*Connected.tsx` wrappers stay in shell-app precisely because they're allowed to be tightly coupled to that specific store.

**The alternative considered:** Leaving Redux wiring inside the shared component and using context injection to swap stores. This is a common pattern but adds a layer of indirection that makes the component harder to test and harder to story in Storybook without mock providers. A prop-only component stories trivially — here's what that looks like in practice:

```tsx
// Before (connected component): requires Redux provider boilerplate
export const Default: Story = {
  decorators: [
    (Story) => (
      <Provider store={mockStore({ approvals: [...], settings: {...} })}>
        <Story />
      </Provider>
    ),
  ],
};

// After (prop-only component): two lines
export const Default: Story = {
  args: { items: mockApprovals, onApprove: fn(), onDismiss: fn() },
};
```

Every store shape change that happens after this point ripples into story setup code if the components are connected. Prop-only stops that ripple at the wrapper boundary.

This is the **tooling for fast iteration** pillar: the three Storybook PRs currently open ([cv-builder] #122, [BlogEngine] #32, [TripPlanner] #31) only become useful if the components they story are prop-only. Those PRs are now unblocked. They were opened the day shell #41 merged — merging the decomposition first was the prerequisite.

**What's missing:** ADR-0020 documenting the prop-only boundary rule does not yet exist as a written record. The decision is implemented across five components and explained here, but it isn't durable until it's filed as an ADR. That's a gap worth naming explicitly — a boundary this consequential needs a document that survives the codebase context.

### Decision: Vercel header rule order as a deployment contract

The cache header bug is a good example of a deployment contract that isn't enforced by any test. Vercel's rule evaluation order (later rules win) is documented but not obvious, and the same mistake appeared independently in three repos — which means it was made independently by different people at different times, not copy-pasted once.

**Why does `remoteEntry.js` need `no-store`?** The remote entry file is the pointer from the host (shell at `:3000`) to the remote's exposed modules. If it's cached `immutable`, the host keeps loading the old module graph after a deployment — new features never appear until the CDN TTL expires. Depending on the CDN, that window can be hours. `no-store` forces a fresh fetch every request. Hashed asset files (`index-abc123.js`) are safe to cache `immutable` because their filename changes on every build.

**Blast radius in concrete terms:** Every user hitting a post-deployment host was receiving the stale module graph — meaning shipped features were invisible in production until cache expiry. Duration before detection was measured in production observation, not in CI. This is the argument for baking the correct rule order into `/scaffold-frame-app` from day one: the cost compounds with every new Frame remote that inherits the broken template.

**What prevents regression now:** Nothing automated. The fix is correct in four repos. The scaffold patch is a suggested action, not yet merged. Until it lands, the detection path is manual config audit — check that `remoteEntry.js` appears as the last matching `Cache-Control` rule in every remote's `vercel.json`. gastown-pilot should be audited explicitly; it's the only Frame remote not touched Thursday.

A shared deployment config template — or a CI lint step that validates `vercel.json` rule ordering — would close this gap permanently. That's the standardization move.

### Decision: core-reader path fix as rename shrapnel completion

The `.claude/commands → .claude/skills` rename touched every repo's tracked symlinks last week. `parseCommands.ts` was reading those paths at runtime, not at build time — so the rename didn't surface as a build failure. It surfaced as a silent empty-result bug in production.

**What this means structurally:** Path strings exercised only at runtime — against a live filesystem or a live API — are invisible to TypeScript's compiler and to most CI pipelines. The compiler can tell you a variable doesn't exist; it cannot tell you a path string is wrong. This is a known gap in the current test coverage. The honest answer on scope: we don't know how many other runtime path strings reference `.claude/` directories that haven't been updated. `parseCommands.ts` is confirmed; sibling API handlers are unaudited. That makes the investigation a grep-and-audit task, not a spot check — and it argues for a test covering the regex, not just a scan.
