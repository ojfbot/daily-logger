## The decisions

### session-init.sh symlink tracked in version control across all repos — *tooling-for-iteration*

session-init.sh is a git hook that fires at session start to capture session telemetry — context about which repo, branch, and tooling environment a session is running in, fed back into the ojfbot assistant layer. Before this rollout, the hook existed only if install-agents.sh had been manually re-run after cloning. install-agents.sh is a one-time bootstrap script: it doesn't run in CI, it fails silently if skipped, and there's no warning when the hook is absent. A fresh clone would have no hook and silently drop session telemetry. Tracking the symlink in git means the hook survives branch switches and fresh clones automatically — no manual step required. To verify a healthy install after cloning: check that .git/hooks/session-init exists as a symlink and points to the tracked script. An absent or broken symlink means telemetry is silently not firing. The rollout touched 8 repos (lean-canvas, TripPlanner, BlogEngine, MrPlug, core-reader, seh-study, purefoy, and daily-logger); core is an issue-tracking and shared-tooling repo, not a runtime app repo, so it was excluded intentionally.

### Fleet-wide daily-cleaner PRs as a continuous hygiene signal — *model-behavior-as-design*

The daily-cleaner bot opens one PR per repo per day. Claude Opus validates each edit at high or medium confidence before it's included — high confidence means the change is unambiguously safe (removing a dead code comment, fixing a broken link); medium confidence means the change is likely safe but has a small chance of altering meaning. Low-confidence edits are dropped entirely. The human merge step exists for borderline medium-confidence cases and to catch systematic errors. The failure mode of the pattern is exactly what's visible today: three daily-logger PRs (#155, #157, #159) queued with no merges. The merge cadence slipping behind the open rate isn't a tooling friction problem — the PRs are each under two minutes to review. It's a priority displacement problem: substantive work has absorbed the short review windows. That distinction matters because the same backlog will exist next week if priority isn't explicitly reassigned.

### ProgressReporter writes structured JSON, not log lines — *tooling-for-iteration*

purefoy is the ojfbot podcast processing pipeline — it ingests audio, runs GPU-accelerated transcription on AWS, and produces structured transcript output that will eventually feed a Frame sub-app. Before PR #39, pipeline state was only visible in unstructured log output: knowing whether a job was stuck at segment 47 of 200 required reading logs. ProgressReporter now writes progress.json at each of 8 named pipeline stages (ingest, validate, segment, preprocess, transcribe, align, post-process, export) with GPU utilization metrics and per-segment heartbeats. To check whether a GPU transcription job is stuck right now: read progress.json from the run directory — the last completed stage and timestamp are there. Structured JSON rather than log lines means a future dashboard or Frame sub-app can consume the file directly without regex parsing. The 8 named stages also make debugging concrete: if progress.json shows align completed and no post-process entry exists, you know exactly where the job stopped.
