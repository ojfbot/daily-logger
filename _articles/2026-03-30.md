## The decisions

### Standardise the ecosystem table across all 13 repos rather than per-repo cross-links — *tooling-for-iteration*

Every repo's README now contains an identical 13-row table with absolute GitHub URLs — not relative ../ paths or hand-curated subsets. The alternative (each repo linking only to immediate neighbours) breaks down the moment a visitor lands on any single repo from an external link: they have no map. A full table with absolute URLs makes every entry point self-orienting for a hiring manager who has never seen the project before.

One honest gap to flag: the decision block states this 'costs nothing to maintain (it's templated),' but the canonical template source is not currently documented. The suggested /document action addresses this. If the current state is manual copy-paste across 13 files, the maintenance cost is 13 file edits per port change or repo addition — that is real debt. The right answer is a snippet in CLAUDE.md or a stamp script. Until that exists, treat the table as maintained by convention, not automation.

### First responsive UIs in landing and daily-logger — *portfolio-readiness as a forcing function*

Landing page responsive hardening ([#4](https://github.com/ojfbot/landing/pull/4)) and daily-logger article page mobile hardening (a2dc7a7) represent the **first responsive UIs shipped across the ojfbot project**. This is worth naming explicitly as a decision, not just a fix. The choice was to ship responsive layout as a hard prerequisite for any external portfolio exposure — not as a polish pass after applications go out. The reasoning: a broken mobile layout in the first ten seconds of a hiring manager's review is not recovered from. The technical debt was real but small — both fixes landed in single commits — and the leverage is disproportionately high relative to the effort. Any new UI surface going forward should treat mobile-first layout as a definition-of-done criterion, not an afterthought.

### Portal-based ProjectPopover with viewport clamping over a simple tooltip — *model-behavior-as-design*

The ProjectPopover component renders into a React portal at document body rather than as a positioned child of the card element. This prevents clipping by overflow: hidden ancestors — a common failure mode in masonry/grid layouts — and allows the viewport-clamping logic to work against the full window dimensions rather than a constrained parent box. On mobile it collapses to a bottom-sheet instead of a hover, matching the platform's interaction model.

The screenshot content inside the popover: today's project context does not confirm whether these are static assets, generated screenshots, or live renders. The /investigate action below tracks this. If they are static captures, a refresh pipeline will be needed as the hosted apps evolve — otherwise the previews will silently go stale, which undermines the portfolio readiness goal they were built to serve.

### Scrape both open and closed PRs from the 24h window in the daily-logger pipeline — *tooling-for-iteration*

536da72 fixed the PR scraper to collect all PRs (open + closed) active in the 24h window, not just open ones. This matters because closed PRs represent the actual merge events that daily articles document — an article about yesterday's work that omits merged PRs is structurally incomplete. The fix also corrects global issue sort so the most recent activity surfaces first in the context block.

For re-entry context: the daily-logger pipeline assembles a structured context block from GitHub API responses — commits, PRs, issues — and feeds it to Claude to generate the article draft. The scraper previously filtered to open state only, which meant any PR merged before the pipeline ran was invisible to the article. The fix passes no state filter, then deduplicates by PR number before assembly.
