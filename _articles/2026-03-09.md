## The decisions

### Decision 1: Page URL checked before remoteEntry origins in `resolveProjectMapping`

`resolveProjectMapping` works as a priority-ordered array of resolver tiers. The function iterates the array in sequence and returns on the **first match** — it does not score or merge results. In the old order: [remoteEntry origins, page URL]. In the new order (`7e5a0e1`): [page URL, remoteEntry origins].

**Why this matters mechanically:** A shell page at `frame.jim.software` loads remoteEntry files from *all* currently-active sub-apps. If cv-builder is visible, the page has both the shell's remoteEntry and cv-builder's remoteEntry in scope. Under the old order, the function checked remoteEntry origins first — and because the shell's remoteEntry matched first in iteration order, every element on the page was attributed to the shell repo. Page URL doesn't have this ambiguity: `frame.jim.software/cv-builder` routes to cv-builder regardless of which other remotes are loaded.

**The `data-mf-remote` attribute is the tiebreaker for same-origin deployments.** When the page URL alone doesn't distinguish apps (all sub-apps share the `frame.jim.software` origin), the content script reads `data-mf-remote` from the `AppFrame` mount div (added by `1eb22ed` in shell). `resolveProjectMapping` checks this attribute as part of the URL tier, not the remoteEntry tier — it's a DOM signal, not a network signal.

**What `resolveProjectMapping` does when there's no `data-mf-remote` and the URL is ambiguous:** It falls through to the remoteEntry tier. If only one remoteEntry origin matches a configured project, that project wins. If multiple match, the first in iteration order wins and a misrouted issue is possible. The confirmation UX in `c21c986` is the last-line defense here — the user sees the resolved project before anything fires — but it doesn't prevent the misroute, it only surfaces it. This is a known gap; the hardening action below targets it.

**What would break if done differently:** Every GitHub issue created from a cv-builder page would land on the shell repo. Claude Code context injections would target the wrong project.

**Pillar: Assistant-centric architecture.** Getting repo attribution right is a trust boundary, not a routing convenience. The confirmation UX (`c21c986`) is the second half of the same principle: high-impact agent actions always show a summary before firing.

### Decision 2: Singleton enforcement at the store level, not the UI level

`e910836` adds a schema-level constraint: when the store receives a launch request for purefoy or core-reader and an instance already exists, it dispatches an error action instead of creating a second instance. The sidebar surfaces this as a toast and brings the existing instance into focus. Agents receive the error action directly.

**Why the store, not the UI?** Module Federation remotes can be launched programmatically — by the shell chat bar, by an agent, or by another app. If the constraint lived in a button's `onClick`, a natural-language command like "open purefoy" would bypass it and create a second instance. The store is the only surface that all launch paths flow through.

**What singleton enforcement protects:** purefoy and core-reader are fundamentally single-context apps — one knowledge base, one metadata dashboard. Two instances would share no state, create two separate agent sessions, and produce an "which one is active?" ambiguity the shell has no mechanism to resolve.

**Tradeoff:** The store now encodes product intent — which apps are singletons — as a schema-level constraint. If that intent changes (e.g. purefoy gains multi-corpus support), the store shape must be updated, not just the UI. That's the right tradeoff: better to make the constraint explicit and findable in one place than to defend it across every launch path.

**ADR status:** The singleton enforcement pattern does not yet have a filed ADR. Given the instruction "before the pattern spreads to other app types" in the original review notes, this is a NOW item, not a later one. An ADR number should be reserved before any additional app types are added to the singleton list. See the action item below.

**Pillar: Assistant-centric architecture.** ShellAgent needs the store to be the canonical authority on what's running. A bypassable store means the agent's world model is wrong.

### Decision 3: `data-mf-remote` attribute on `AppFrame` mount div

`1eb22ed` adds `data-mf-remote="[app-name]"` to the div that hosts each Module Federation remote in shell.

**Why a DOM attribute?** MrPlug's content script runs in the page context. It can read the DOM. It cannot read the shell's Redux store. Without a DOM-level signal, the content script has no reliable way to know which sub-app is currently active in the viewport when multiple remotes are mounted but only one is visible.

**Why not just use the URL?** On the live deployment at `frame.jim.software` (reachable now), all sub-apps share the same origin. `frame.jim.software` doesn't change when you switch from cv-builder to TripPlanner — only the visible MF remote changes. The DOM attribute is the out-of-band signal that crosses the browser extension / web app boundary without requiring new extension permissions or a new API endpoint.

**Why this is the cheapest correct fix:** The alternative — a postMessage channel between the shell and the content script — would require a new permissions surface, a new message schema, and a new failure mode (message not received). The DOM attribute requires none of that. It's readable synchronously, requires no handshake, and is visible in DevTools.

**Pillar: Tooling for fast iteration.** Small, surgical, unblocks routing without expanding the permissions model.
