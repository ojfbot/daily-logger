## The decisions

### Decision 1: `pnpm audit` runs non-blocking in CI

Every security scan workflow runs `pnpm audit --audit-level=high || true` — the `|| true` means a failed audit does not block merge.

**Why?** Transitive dependency vulnerabilities in the Node ecosystem are overwhelmingly in devDependencies that never reach a production bundle. Blocking CI on every transitive hit creates alert fatigue: developers learn to ignore the red, and the signal that matters — a direct dep with a known CVE — gets lost in noise. Making it non-blocking means the output is still visible in the CI log. It is a dashboard, not a gate.

**What breaks if you do it differently?** Block on `--audit-level=moderate` in a monorepo with Carbon, Vite, and Playwright as devDeps and you will get 40+ advisories on day one, mostly in test tooling that never ships. The team merges around the red light within a week.

**The tradeoff:** Non-blocking means a genuine high-severity vuln in a production dep could slip through unaddressed. The named mitigations are Dependabot (for known CVEs in direct deps) and TruffleHog (for secret leakage). **One open question**: Dependabot is named as a mitigation here, but it has not been confirmed as configured on all eleven repos. If it is not, the rationale for non-blocking audit has a gap. Enabling Dependabot fleet-wide is a follow-up action.

**ADR status:** This decision is load-bearing and is already in production across the fleet. It is documented here in prose. It has not been committed as a formal ADR to the `core` repo. It should be.

This is **security as emergent UX** — the Samir/TBCoNY pillar. Security tooling that blocks on false positives trains developers to disable it. UX applies to developer tools too: the scan has to be trustworthy to be trusted.

### Decision 2: CI clones `frame-ui-components` via `git clone` rather than resolving a `file:` workspace dep

Every consuming repo has a `file:../frame-ui-components` path in its `package.json`. This works locally. It does not work on Vercel or GitHub Actions — the sibling directory doesn't exist in the CI checkout.

**The fix:** add a `git clone https://github.com/ojfbot/frame-ui-components` step before `pnpm install`. The repo is cloned into the parent directory so the relative path `../frame-ui-components` resolves correctly against the consuming repo's checkout. If the clone target ever diverges from that relative path, `pnpm install` fails silently in a new way — the exact clone destination should be treated as a convention, not an assumption.

**What breaks if you skip it?** `pnpm install` fails with `ERR_PNPM_NO_MATCHING_VERSION` or a path-not-found error. The build never reaches the TypeScript step. This burned four repos before the fix propagated.

**What does `HEAD` pinning mean in practice?** Every CI run clones `frame-ui-components` at `HEAD`. A breaking change in the shared package breaks CI in every consuming repo simultaneously. This is useful signal — it means a changed contract can't hide — but it also means eleven repos can go red at once. The blast radius is manageable at current scale; it is not solved. This is a managed state, not a closed one.

**Failure mode when the clone succeeds but the package breaks:** The failure surfaces at the TypeScript compile step, not at `pnpm install`. If `frame-ui-components` introduces a type error or a removed export, every consuming repo will fail with a TS error pointing at the shared import. That is where to look first when CI goes red fleet-wide.

**The `file:` dep and the publish transition:** The `file:../frame-ui-components` entry in each repo's `package.json` stays as-is until the first versioned publish. When that publish happens, it requires a find-and-replace across all eleven consuming `package.json` files — replacing the `file:` path with a versioned registry reference. That is a concrete dependency on the publish action item and needs to be in the ADR scope.

**Why not publish to npm now?** Publishing adds a release step, a version bump ritual, and a publish token to manage — none of which is worth the overhead while the library is still changing shape daily. The `git clone` pattern is a deliberate short-term choice: fast iteration at the cost of a CI step. When the library stabilises, the right move is a versioned npm publish with a lockfile pin. "When the library stabilises" is not a criterion yet — a specific trip wire (e.g., zero breaking API changes across a sprint, or green CI fleet-wide for N consecutive days) should be named in the publish ADR.

**ADR status:** Not yet committed to `core`. Should be.

This is **tooling for fast iteration** — keeping the feedback loop tight while the shared library is in motion.

### Decision 3: Connected wrappers live in the consuming app, not in `frame-ui-components`

`ThreadSidebarConnected`, `ChatShellConnected`, and similar components wire the presentational components from `frame-ui-components` to each app's Redux store. They live in the consuming app (BlogEngine, TripPlanner, etc.), not in the shared package.

**Why?** `frame-ui-components` does not know about Redux. It ships pure presentational components with typed props. If Connected wrappers lived in the shared package, it would need to depend on each app's Redux store shape — coupling it to every app's state structure and breaking the encapsulation that makes the package useful.

**What breaks if you do it differently?** Circular dependency risk, or a lowest-common-denominator store interface in `frame-ui-components` that satisfies no app well.

**The drift risk:** `ThreadSidebarConnected` now exists independently in multiple repos, each wrapping the same presentational component. The contract that prevents them from drifting is the typed props interface exported by `frame-ui-components`. A prop change in `ThreadSidebar` surfaces as a TypeScript compile error in any consuming wrapper that no longer satisfies the interface — provided each Connected wrapper is typed against the exported prop types. If a wrapper uses `any` or casts the props, this guarantee disappears silently. The convention is that Connected wrappers must be typed against `frame-ui-components`' exported interfaces; this should be documented in the package README.

**ADR status:** Not yet committed to `core`. The three decisions made today — non-blocking audit, `git clone` CI pattern, Connected wrappers in consuming apps — are all in production across eleven repos and documented only in this article. All three should be committed as formal ADRs. ADR-0027 and ADR-0028, by contrast, are committed for work that hasn't started yet. That inversion is worth correcting.
