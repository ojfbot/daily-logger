## The decisions

**Architecture in one sentence (repeated for out-of-context readers):** Nine repos, one shared component package consumed via CI clone, one shell host federating six Module Federation sub-apps at runtime. Every sub-app runs its own React tree and Redux store. The decisions below only make sense with that topology loaded.

---

### Decision 1: `git clone` instead of `actions/checkout` for `frame-ui-components` in CI

The frame-ui-components package is not published to npm — it lives in a private GitHub repo and is consumed by cloning it during CI. Monday's fix wave (`720d62b`, `c70243e`, `6377431`, `d5d146b`) standardised every repo on `git clone` rather than a composite `actions/checkout` step.

**Authentication:** The clone uses the built-in `GITHUB_TOKEN` available in every GitHub Actions runner — no additional secret is required as long as all repos live in the same organisation. This is not zero-dependency, but it is the minimal-additional-secret option.

**How the import resolves after the clone:** Each consuming repo's `package.json` references `@ojfbot/frame-ui-components` via a `file:` path pointing to the cloned directory (e.g., `"@ojfbot/frame-ui-components": "file:../frame-ui-components"`). After the clone runs, pnpm resolves the import from that local path. The `module-not-found` failure happens when the clone step is absent — pnpm has no npm registry entry to fall back on.

**Why not publish to npm?** Publishing to a private npm registry adds an `NPM_TOKEN` to every repo's CI environment and requires versioning and publishing on every component change. At this stage, with rapid API churn, that overhead is higher than the cost of a clone. The tradeoff flips when the component surface stabilises — that point is the Phase 1 `@ojfbot/shell` npm publish gate.

**The Carbon CSS path problem (why three fix commits):** Carbon's package structure under pnpm is not hoisted the same way as under npm or yarn. When `@carbon/styles` is declared as a dependency of `frame-ui-components` but consumed in a cloned `file:` path, pnpm may not surface it at the top-level `node_modules` of the consuming app. The fix is an explicit `@carbon/styles` dependency in the consuming app's own `package.json`, with the import path pointing directly into the Carbon package's distributed CSS entry (e.g., `@carbon/styles/css/styles.css`) rather than a barrel re-export. Three commits were required because the correct path differed between local dev (where pnpm hoisting behaves differently) and CI (where the clone creates a flat directory). This is the exact failure that will recur for every new consumer repo — document it in the onboarding README.

**SHA-pinning — the current known liability:** Every repo's CI now clones `frame-ui-components` at `HEAD` with no pinned SHA. A single breaking commit to the shared package will fail nine CI pipelines simultaneously with no rollback point. This is not a future risk — it is today's state. No breaking change has happened yet, but the blast radius is real. An ADR governing SHA-pinning vs. tag-pinning is the next required decision before the next change to the shared component API.

### Decision 2: `ThreadSidebarConnected` as a Redux wrapper inside each consuming repo

BlogEngine and TripPlanner both ship a `ThreadSidebarConnected` component — a thin Redux-aware wrapper around the shared `ThreadSidebar` from `frame-ui-components`. The shared package exports the presentational component; each app owns its own connected wrapper.

**Why not ship `ThreadSidebarConnected` in the shared package?** The shared package doesn't know which Redux store it will be mounted in. Module Federation means sub-apps run inside their own React tree with their own Redux store — a connected component baked into the shared package would need to import from a store it can't reference at build time. The container/presenter split (described in ADR-0029, currently in draft — see gap note below) solves this: the shared package owns the presentational layer, each app owns the connection layer. This is the prop-only boundary that makes Storybook stories trivial to write.

**What breaks if you put `ThreadSidebarConnected` in the shared package?** Either the component has no access to the correct store (silent bug: no threads render), or you add a store-injection mechanism that couples the shared package to a specific state shape (worse: you've just recreated the monorepo problem you were trying to escape).

**ADR-0029 status gap:** Decision 2 cites ADR-0029 as the governing authority for the container/presenter split. ADR-0029 is currently in draft — it has not been formally accepted. The pattern it governs is in production across BlogEngine and TripPlanner. ADR-0029 should be promoted to accepted before the next consumer repo ships a `Connected` wrapper.

This pattern directly demonstrates the **assistant-centric architecture** pillar: the shared shell components are the neutral substrate; each app's Redux store is the domain-specific layer. ShellAgent will eventually orchestrate both layers without either knowing the other exists.

### Decision 3: Non-blocking dependency audit in the security scan

The `pnpm audit` step in every new security workflow is marked `continue-on-error: true`. A blocking audit would immediately fail CI on every repo that has a transitive vulnerability in its lockfile — which, at Monday's baseline, is almost certainly every repo.

**Why non-blocking first?** A blocking audit is only useful once you have a clean baseline. Adding a hard-fail gate against an unknown number of existing findings creates noise that causes engineers to ignore CI failures entirely — the exact opposite of what a security gate is for. The correct sequence is: establish baseline → triage findings → promote to blocking. Making it non-blocking is not permissiveness; it's sequencing.

**What does "promote to blocking" actually mean?** In the security workflow YAML, the audit step currently reads:

```yaml
- name: Dependency audit
  run: pnpm audit
  continue-on-error: true
```

Promoting to blocking means removing `continue-on-error: true`. That's a one-line change — but it should only happen after running `pnpm audit` locally across all nine repos, triaging every finding, and either patching or explicitly `pnpm audit --audit-level` scoping the findings you're accepting risk on.

**Security as emergent UX:** This sequencing philosophy — make the workflow visible before making it enforced — applies beyond CI. It's the same pattern Frame's shell uses with sub-app registration: the integration point is wired before it's load-tested. Engineers understand what they're committing to *because they've seen it run* before it blocks them. That understanding is what makes the hard gate credible when it arrives.

### Decision 4: ADR-0027 and ADR-0028 drafted together

ADR-0027 (gastown as direct `core-workflows` consumer) and ADR-0028 (react-query for gastown server-state) were written as a pair in [core] #28. The two decisions are coupled: if gastown is a direct `core-workflows` consumer, server-state management becomes a first-class concern that needs an explicit choice immediately.

**Why react-query instead of RTK Query for gastown?** BlogEngine and TripPlanner already use RTK Query. Adding react-query to the fleet has a real cost: two server-state mental models, two upgrade paths, two sets of devtools. The argument for react-query is not "less ceremony" in the abstract — it's that gastown is a *new* consumer with *no existing RTK Query surface*. For a greenfield consumer, react-query's cache invalidation, background refetch, and query key model require less adapter code when there is no existing Redux slice to integrate with. If gastown already had RTK slices, RTK Query would be the obvious choice. It doesn't. The ADR exists so the next engineer doesn't re-litigate this from scratch when gastown's first data-fetching PR lands.
