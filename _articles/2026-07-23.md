## The decisions

### OJF-OPL: process models as committed artifacts, not prose ADRs — *assistant-centric*

ADR-0102 formalises that OPM models live as committed `opm/system.opl` files per repo, with a Mermaid twin for human readability and a `/opm lint` skill for mechanical validation. OPL (Object-Process Language) is a formal modeling standard — the ojfbot implementation uses a DSL-subset that captures objects, processes, and agent/instrument boundaries as structured text. The explicit decision against prose-only ADRs: ADRs describe *decisions*; OPM models describe *what the system is*. Prose ADRs are searched by humans; OPL files are lintable artifacts an agent can traverse to answer structural questions without reading the whole codebase. The alternatives considered were YAML schemas and plain Mermaid — YAML has no process semantics; Mermaid has no formal object/agent distinction. OPL was chosen because it encodes both, and `/opm lint` can validate structural invariants mechanically. The first lint pass on daily-logger's `opm/system.opl` immediately found a real error: the SD1.1 zoom had dropped the Council phase. That's the falsifiable result that justifies the tooling investment.

### /wayfinder: file-canonical decision maps upstream of the roadmap spine — *tooling-for-iteration*

ADR-0101 establishes that the canonical location for a wayfinder map is a file in `decisions/wayfinder/`, with GitHub issues as a tracker projection only — not the source of truth. The F1 learning studio map was the first dogfood: the file was committed before the issues were opened, establishing that the file drives the issues, not the reverse. This prevents wayfinder maps from fragmenting across issue threads where context decays. The ordering matters: if issues are opened first and the file is written to summarise them, the file is always playing catch-up with a moving thread. If the file is committed first, every issue is a downstream projection of a stable artifact that can be read, linked, and linted without reconstructing a thread.

Tags: `wayfinder`, `core`, `assistant-centric`

### /fleet-onboard: drift reconcile as a first-class slash command — *tooling-for-iteration*

Rather than manually auditing REPO_META and standup lists for missing repos, `/fleet-onboard` runs a one-pass registration that diffs the active repo set against every tracked surface and emits reconcile PRs. The first run surfaced fourteen missing repos in morning-cockpit and seven in daily-logger — evidence that manual roster maintenance decays faster than the fleet grows. The missing repos were cases of repos created or activated after the last manual sweep, not renames or deletions; the root cause is a process gap (no automated hook from repo creation to manifest registration), not a schema ambiguity. The command makes drift visible and patchable in a single session. For a fleet of 33 active repos, stale manifest state means stale cost attribution, stale access reviews, and stale dependency maps — it's a controls problem, not just a tidiness problem.

### OPM pilot in daily-logger: cross-repo evidence base for ADR-0102 — *model-behavior-as-design*

ADR-0039 was accepted to make daily-logger the first OPM pilot repo. The logic: the daily-logger pipeline is small, well-documented, and already has a Mermaid diagram tradition — making it the lowest-friction place to validate that the `opm/system.opl` + lint workflow is usable before imposing it fleet-wide. Hypothesis: the lint check catches structural drift that prose review misses. First run result: confirmed. The SD1.1 zoom had dropped the Council phase — a real structural error that passed prose review and failed lint. The second pilot target is the F1 content pipeline (core #259, unblocked). Piloting in two structurally different repos before fleet rollout is the explicit strategy: daily-logger is a linear pipeline; the F1 content pipeline has branching publish paths. Both need to pass before the pattern is declared stable.
