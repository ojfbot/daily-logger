### Seven low-activity days: what the pattern reveals about queue mechanics — *tooling-for-iteration*

When human commit volume drops to near-zero for a week, the automated pipelines (daily-cleaner, blog-bot) continue to generate PRs — but without merge activity those PRs accumulate into a backlog that itself becomes coordination overhead. The right response is not to ship code for its own sake, but to explicitly triage the queue and close the items that are mechanically complete. This audit surfaces at least seven named open actions across the fleet. Some are eleven days old. The action system was designed to resolve backlog — but the evidence in this article is that actions are accumulating the same way PRs are. That's worth naming directly: the action queue may itself need a triage pass, not just the PR queue.

### Six open stale-doc PRs: the daily-cleaner queue is not self-clearing — *tooling-for-iteration*

Daily-cleaner opens one PR per repo per run, validated by Claude Opus, but it does not auto-merge. Six PRs — asset-foundry #30, daily-logger #195, daily-logger #191, landing #23, beaverGame #42, core-reader #29 — have been open between one and five days. Each is low-risk (stale comment removal, high/medium confidence edits only), but they accumulate noise in the open PR list and consume reviewer attention.

What 'Opus-validated' means here: Claude Opus reviews the diff for semantic correctness and assigns a confidence tier (high, medium, or low). Daily-cleaner only queues PRs at high or medium confidence. It does not run tests — it validates that the stale content being removed is genuinely stale and that no live logic is touched. This is sufficient as a safety gate for comment/doc-only changes, but it does not cover changes that touch code paths. Any auto-merge policy should be explicitly scoped to doc-only, no-conflict, high/medium confidence diffs.

The systemic fix is an auto-merge policy for daily-cleaner PRs that pass Opus validation without conflicts. The safety conditions are already articulated — no conflicts, high/medium confidence, single-repo scope, doc-only diff. The open question before implementing is not whether those conditions are sufficient (they are, for this change class) but where in the daily-cleaner workflow the merge step should be inserted and whether it should be gated behind a feature flag for the first run. A draft PR against the daily-cleaner workflow that wires the merge step to a feature flag — disabled by default, enabled explicitly — would be a concrete deliverable. 'Investigate and document' is not.

### ADR-0068 enforcement remains suspended: telemetry audit still unactioned — *model-behavior-as-design*

ADR-0068 requires agents to follow skill suggestions, but enforcement was explicitly suspended pending verification that the telemetry model correctly distinguishes real-time suggestions from PR-comment-marker credits. The 0% follow-rate figure cannot currently be attributed to agent non-compliance versus instrumentation inflation — enforcing the ADR against potentially synthetic data would produce false negatives.

What the instrumentation measures: when an agent receives a skill suggestion, a telemetry event is expected to fire at the moment of the suggestion. Separately, PR-comment markers (annotations left in PR bodies by the review pipeline) may be backfilling credits for suggestions that were never delivered in real time. If those two event classes are not distinguished in the log store, the 0% follow-rate may reflect a counting artifact rather than agent behavior.

What the audit requires concretely: a query against the telemetry log store to classify events by `suggested_at` timestamp versus PR-comment-marker `created_at` — determining whether real-time suggestion events exist at all in the sample window. This is a query against existing data, not a build task, provided the log store is accessible. Three separate actions (act-2026-05-05-investigate-f18cb5, act-2026-05-06-investigate-159af4, act-2026-05-10-investigate-d5cfb2) have been filed across five days without the query running. That is a decision problem, not a backlog problem. Someone needs to open the log store and run the query. Until that happens, ADR-0068 enforcement stays suspended and the 0% figure remains uninterpretable.
