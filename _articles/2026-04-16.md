### session-init.sh symlink rollout: why symlinks need to be tracked in git — *tooling-for-iteration*

The wave of chore(hooks): track session-init.sh symlink PRs merged this week across lean-canvas, TripPlanner, BlogEngine, MrPlug, core-reader, seh-study, and purefoy addresses a specific failure mode. Git tracks symlink targets as blob content — the string of the target path is what is committed. If the symlink lives only in .git/hooks/ it is outside the repo's tracked tree entirely and does not appear in git status at all. The failure this prevents: install-agents.sh creates the symlink at clone time, but if the symlink target path changes (e.g. core is moved or renamed), the hook silently breaks on the next clone or branch switch because git has no record of the old target to flag as drifted. Committing the symlink into a tracked location such as .claude/hooks/ means git status reveals when the on-disk symlink target diverges from the committed target string, and git checkout restores the committed target deterministically. To verify this locally: git show HEAD:.claude/hooks/session-init.sh will print the target path as a string. If that string no longer resolves on disk, git status will show the file as modified.

### purefoy ProgressReporter: observability without blocking the hot path — *model-behavior-as-design*

purefoy PR #39 landed a ProgressReporter that writes structured progress.json at each of 8 GPU transcription pipeline stages. The critical constraint: this must not block the transcription hot path. GPU time is expensive and latency on the reporter write must not gate the next pipeline stage. The implementation uses a write-and-forget pattern — the reporter emits to a well-known path and returns immediately; any consumer polls on its own schedule. The current consumer is the ojfbot session layer, which polls progress.json at a 2-second interval during active transcription jobs. If the GPU job crashes mid-pipeline and progress.json is partially written, the consumer detects staleness via a lastUpdated timestamp field — if the timestamp has not advanced in more than 30 seconds, the job is considered failed and the session layer surfaces an error. This is the same pattern as bead emission: write-and-forget to a well-known path, let a consumer poll. Consistent observability architecture across Python (purefoy) and TypeScript (shell, core-reader) repos matters when core-reader eventually aggregates cross-repo state.

### core #77: npm package distribution for skills vs. symlinks — a foundational fork — *tooling-for-iteration*

The proposal to migrate skill distribution from install-agents.sh symlinks to a versioned @ojfbot/core-skills npm package (core issue #77, opened 2026-04-14) is the most architecturally consequential open issue this week. It is not on the Gas Town critical path — nothing shipping in the next two weeks is blocked by it — but it is the kind of decision that gets harder to reverse as more repos adopt the symlink pattern. Symlinks work fine for a solo developer. They break the moment a second contributor clones without running the install script, or when CI needs to reproduce a specific skill version. An npm package gives semver, changelog, rollback, and lockfile pinning. The tradeoff is publish friction: every skill edit requires a version bump and publish step. At 5 or 10 contributors, a private GitHub Packages registry with a post-merge publish action makes that friction invisible — the registry cost is covered by the existing GitHub org plan and the operational overhead is one workflow file. At current solo-developer scale, the friction is real and manual. The decision note that needs to exist before an ADR is opened should answer: what is the earliest point at which a second contributor will need to clone and run skills without running install-agents.sh? That date determines the urgency. Ownership of the decision note sits with whoever opens the ADR — which is Jim, and the note should be opened as a file in the core repo this week, not deferred further. The tradeoffs are already written in this article. That is the decision note.
