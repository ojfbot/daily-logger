### shell — `POST /api/resumption` thread resumption synthesis endpoint

PR [shell] #33 → commit `c22922e`. Adds a stateless endpoint to `frame-agent` that synthesises a contextual 1–2 sentence opener from an existing conversation history. Mechanically: the endpoint receives the message array, injects it into a Claude prompt, and returns a single synthesised sentence — no thread state is mutated, no session context is required. The design intent is "Last time you were working on X — want to continue?" The endpoint is the technical foundation for [shell] #24 (FeedbackModal conversation context restoration) and the hero demo fan-out path.

### shell — tighten resumption schema

Commit `9c5a455`. Follow-up to the resumption endpoint: `min(2)` on message array, `max(4000)` on content strings. `min(2)` prevents a degenerate single-message synthesis request (a one-message history has no arc to synthesise from). `max(4000)` per content string is a payload guard before the endpoint is publicly accessible — unbounded content could silently exhaust Claude's context window budget without triggering any upstream error.

**Open question on failure mode:** The `max(4000)` guard prevents payload abuse, but a well-formed request could still approach Claude's context window after synthesis begins. The current endpoint behavior in that case — degraded response, 500, or silent omission of the opener — is not yet documented. This needs to be named before the hero demo goes live.

## The decisions

### Decision 1: Four rename-fix PRs merged before new feature work resumed

**Choice:** Before pushing further Phase 1 extractions, all four rename-regression fixes were merged as discrete PRs rather than bundled into the next feature branch.

**Why:** A Module Federation app that builds locally but produces `undefined` remote URLs in production is worse than no rename at all — it deploys broken silently. The localStorage stale-registry issue compounds this: a user who visited the deployed shell before Wednesday would be stuck on a broken state even after the env fix landed. Sequencing the fixes first meant the production build was valid before the wiring PR (`#31`) created new surface area.

**What would break otherwise:** If `HeaderConnected` and `HomeScreenConnected` had been wired before the MF key alignment was resolved, every PR touching `shell-app` would be landing on a codebase that fails in production but passes CI — a debugging trap that's expensive to untangle after the fact. The logic chain: broken prod build → CI passes but deploy fails → every subsequent PR carries the cognitive overhead of "is this new or is it the rename bug?" → fix first, then wire.

**Pillar:** *Tooling for fast iteration* — keeping the production build green is a prerequisite for fast iteration. A broken deploy means every subsequent change has to carry debt that belongs to a previous decision.

---

### Decision 2: `POST /api/resumption` as a stateless, schema-validated endpoint

**Choice:** Thread resumption synthesis lives as a standalone `frame-agent` endpoint rather than being embedded in the existing chat handler. Input is schema-validated with `min(2)` messages and `max(4000)` content length enforced at the boundary.

**Why stateless:** The resumption synthesis is a read operation on history — it doesn't mutate thread state, doesn't require session context, and should be callable from any surface (FeedbackModal, HomeScreen, future shell chat bar). Embedding it in the chat handler would couple the synthesis logic to the threading model of one specific UI flow.

**Why the schema constraints:** `min(2)` prevents a degenerate single-message synthesis (a one-message history has no arc). `max(4000)` per content string is a payload guard before the endpoint is publicly accessible — unbounded content could silently exhaust Claude's context window budget without triggering any upstream error.

**What this enables for the hero demo:** The hero demo query — "I'm applying for Berlin jobs, how does this affect my trip plan and my CV?" — requires the shell to surface prior context from each sub-app before fanning out to their agents. `POST /api/resumption` is the mechanism: shell calls it per-app, gets a one-sentence context opener, prepends it to each agent's prompt. Without this, the cross-domain fan-out ([shell] #32) has no way to ground the query in the user's actual history.

**Open question:** Whether real Berlin jobs query payloads fit within `max(4000)` per content string has not been validated against actual conversation data. The `/investigate` action is flagged but not yet complete. This needs to be confirmed before the hero demo goes live — not assumed.

**Pillar:** *Assistant-centric architecture* — the resumption endpoint is ShellAgent infrastructure. Its purpose is to make the shell a better orchestrator of app-level context, the same bet Dia makes with its AI address bar: the assistant needs memory of the surface it's orchestrating to be useful.

---

### Decision 3: Props-driven extraction as the invariant for `@ojfbot/shell` components

**Choice:** Both `Header` and `HomeScreen` were extracted with zero Redux or AppType imports. All state resolution happens in connected wrappers in `shell-app` before props are passed down.

**Why this invariant:** `@ojfbot/shell` will eventually be consumed by BlogEngine and TripPlanner, which have their own Redux store topologies. A shared component that reaches into a specific store's slice is not actually shared — it's a copy of the shell's assumptions distributed across multiple packages. The props boundary forces each consumer to own the resolution layer.

**What breaks if you violate it:** The first time BlogEngine imports `Header` and hits a missing `cv-builder` slice, you get a runtime error that looks like a component bug but is a package boundary violation. These are the hardest class of Module Federation bugs to diagnose because they only appear when the consuming app's store doesn't match the producing app's expectations.

**Why not a context provider?** A shared context provider would work but requires the consumer to wrap their app in a provider from `@ojfbot/shell` — runtime coupling, not just import coupling. Props are explicit; context is implicit. For a package consumed by three apps with different state models, explicitness is worth the boilerplate.

**This invariant is not yet recorded as an ADR.** The props-driven extraction rule, the MF key naming convention (underscore-separated), and the stateless design of `POST /api/resumption` are all currently tribal knowledge. AppSwitcher extraction — the next Phase 1 unit — should not start until at minimum the props-driven invariant is written down. The pattern lives in `shell-app/src/components/HeaderConnected.tsx`; the rationale needs to live in an ADR.
