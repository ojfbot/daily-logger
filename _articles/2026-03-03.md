**Shell-specific fix — `loadEnv` for MF remote URLs (9d3c9a6):** The federation plugin reads remote URLs from `.env.production` at build time. `import.meta.env` is not available during Vite config evaluation — it only exists at app runtime. Without the fix, the compiled shell bundle contained empty strings for every remote URL. At runtime, that meant the MF runtime attempted fetches to paths like `https://frame.jim.software/undefined/remoteEntry.js`, which 404s silently and leaves the federated component unmounted with no console error. The fix switches to Vite's `loadEnv()` utility, which reads `.env.production` synchronously inside `vite.config.ts`, so the correct production URLs are baked into the bundle at compile time.

**TripPlanner — centralized API client (a8cbed3 + [TripPlanner] #21):** Module Federation exposes TripPlanner as a remote, which means its `axios` calls run in the shell's origin context. Bare relative paths like `/api/trips` resolve against `frame.jim.software`, not `trips.jim.software`. At runtime, this means every API call from the mounted TripPlanner remote would silently 404 against the wrong origin — no error thrown, just empty responses. The fix centralizes all API calls through `src/api/client.ts` with `VITE_API_URL` as the `baseURL` — a one-line env var swap between local and production. This is now the reference implementation for BlogEngine and every future remote.

**Cross-repo CI hardening (all repos):** Four CI fixes applied uniformly across every workflow:
- `contents:read` permission — required for `actions/checkout` in private repos. Without it, `checkout` returns an empty workspace and the build "succeeds" by compiling nothing — a silent green on a deploy that ships an empty bundle.
- `pull-requests:write` permission — required for the PR comment step that posts visual regression thumbnails.
- `VITE_IBM_TELEMETRY_DISABLED=true` — suppresses IBM Carbon's network calls in CI and local dev.
- Remove pnpm version pin — delegated to `packageManager` field in `package.json`, eliminating version drift between local and CI.

**ADR-0013 + ADR-0014 formalized in core ([core] #11, #12):** ADR-0013 documents the safe demo deployment mode — `VITE_FRAME_AGENT_URL` env var gating, `ShellHeader` `agentAvailable` prop, graceful degradation when the Express backend isn't reachable from a static host. ## The decisions

## Why Vercel over continuing with GitHub Pages + K8s?

The previous setup had shell deployed to GitHub Pages at `frame.jim.software` via a CNAME. Sub-apps had no production URLs — they ran as MF remotes only in local dev. K8s manifests existed in the shell repo but weren't actively serving traffic; they've been superseded by ADR-0014 because the operational overhead of self-managed ingress isn't justified until the Express backend needs a stable production home (see below).

GitHub Pages works for static files with no build-time environment injection. Module Federation breaks that constraint: each remote's URL must be baked into the host's production bundle at build time. GitHub Pages has no concept of per-PR preview deploys, no CORS configuration, and no build pipeline integration. Vercel solves all three.

> **Why does MF require CORS headers on `/assets/*`?**
> When `frame.jim.software` loads `remoteEntry.js` from `cv.jim.software`, the browser enforces same-origin policy on the fetch. Without `Access-Control-Allow-Origin: *` on the `/assets/` path, the browser blocks the remote bundle load and the federated component never mounts. `/assets/*` is the right scope because Vite outputs both `remoteEntry.js` and all hashed chunks under that path by default. If a future Vite config changes the output directory, the CORS scope would need a corresponding update — this is not automatic. Every `vercel.json` in today's PRs includes this config; it's non-optional for MF to work cross-domain.

> **What is ADR-0013's "safe demo mode" solving?**
> Frame's shell at `frame.jim.software` is a static Vercel deploy. The Express backend (Claude API, agent routing) runs separately and isn't always reachable from a public URL. Safe demo mode means the shell degrades gracefully: `ShellHeader` checks `agentAvailable` and hides the chat command bar when the backend is unreachable — the UI element disappears cleanly, no broken state visible. It does not show an error message or a disabled state; the app simply presents without AI orchestration. The alternative — hardcoding a backend URL and showing unhandled promise rejections — fails visibly in the worst possible context: a recruiter demo. Naming this as "safe demo mode" in the ADR frames it as a designed state, not just a suppressed error.

## Why apply CI fixes across all repos atomically?

The `contents:read` and `pull-requests:write` permission fixes, the IBM telemetry disable, and the pnpm version delegation were applied to every repo in the same day rather than incrementally. The reason is failure mode consistency: if cv-builder's CI passes and TripPlanner's silently fails because it's missing `contents:read`, you get misleading green/red signals across repos. Uniform CI configuration is a prerequisite for trusting CI signals at all.

> **What does "silent failure" look like here?**
> Without `contents:read`, `actions/checkout` in a private repo returns an empty workspace — no error, no warning, exit code 0. The subsequent build step runs against an empty directory, compiles nothing, and Vercel deploys a near-empty bundle. The deploy status shows green. The only way to detect this without having read this article: check the deployed bundle size (it will be anomalously small) or add a post-deploy smoke test that fetches a known asset path. Neither guard exists yet — that's an open gap.

This is **Pillar 2 — Tooling for fast iteration** at the infrastructure layer. The `setup-vercel-ci.sh` script and the uniform workflow template mean the next new Frame repo gets CI/CD in minutes, not hours of manual dashboard clicking.

## Why centralize TripPlanner's API client now, not at MF integration time?

TripPlanner's Module Federation integration landed yesterday. The relative-path API bug was latent — it wouldn't surface until the remote ran in the shell's origin context in production. The `VITE_API_URL` fix in PR #21 is exactly the kind of issue that only becomes visible when you actually deploy. The centralized `src/api/client.ts` pattern is now the reference implementation for BlogEngine and future remotes.

> **On the "missing env var → silent empty-string" problem:** This is an unsolved architectural constraint. A missing `VITE_API_URL` at build time produces a bundle where `baseURL` is `""`, which causes API calls to resolve against the current origin — no build error, no runtime throw, just wrong behavior. There is no build-time assertion or Vite plugin guard for this today. Until one exists, the mitigation is the workflow audit suggested in What's Next.

## Honest gap: ADR-0014 Layer 2 is TBD

ADR-0014 documents a two-layer topology where Layer 2 — the Express backend running Claude API and agent routing — is marked as cloud services TBD. That backend is real and running; it doesn't have a permanent production URL. The K8s path was abandoned because operational overhead exceeds current need. The three options under consideration are: (1) Vercel serverless functions, constrained by runtime limits for long-running agent calls; (2) Railway or Render for persistent Node processes; (3) a minimal VPS. This decision is not made, and ADR-0015 (or a follow-on ADR) needs to make it before the deployment topology can be called fully resolved. The article's claim that the topology is "newly complete" applies to the static layer only.
