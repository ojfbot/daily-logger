## The decisions

### Standup funnel as a first-class skill consumer (ADR-0054) — *tooling-for-iteration*

The standup funnel exists for one developer reviewing their own session output — not a PM digest, not a team report. It reads the `.handoff/` bead ledger (a JSONL file written by skill invocations during a session, living at `.handoff/` in the repo root), walks the audit trail, and emits structured `standup:suggested` events. Those events are what the daily-logger consumes to produce the next day's digest. Keeping the standup artefact in the same event-driven, bead-readable flow as every other skill invocation means the daily-logger needs no separate data pipeline — it reads the same ledger everything else does.

To invoke: run `/standup --funnel=standup` inside a Claude Code session after skills are installed via core #99. The emitted event lands in `~/.claude/suggestion-log.jsonl` as a `standup:suggested` entry. A passing run produces at least one such entry; a failing run produces none or an error in the funnel output.

This decision was sequenced correctly: the standup funnel was built after the bead schema was stable (ADR-0054 companion in `068bc2e`) and before the installation step (core #99) so the consumer exists before the installer.

### Skill adoption measurement as a catalog responsibility (ADR-0050) — *model-behavior-as-design*

Tracking whether skills are used is now a first-class catalog concern, not an external observability bolt-on. ADR-0050 defines invocation counts, suggestion-to-follow conversion, and per-session coverage — all written to `~/.claude/suggestion-log.jsonl`, the same file the suggestion pipeline already writes to.

The tradeoff is scope: the catalog now owns both skill definitions and the adoption signal. The rationale for keeping this local rather than in a shared store is single-developer scope — this system currently runs on one workstation, and a lightweight shared store would introduce a coordination dependency before there is a second developer to coordinate with. The migration path when a second developer joins is an open architectural question that ADR-0050 explicitly defers; the right time to answer it is when the aggregation requirement exists, not before.

One honest gap: the validation story for the telemetry pipeline itself is not documented. Before the adoption signal is treated as authoritative, a test showing that a known invocation produces a known ledger entry should exist. That test does not exist yet.

### Pocock suite: full skill bodies before integration (ADR-0047, 0048, 0049) — *tooling-for-iteration*

We completed `/tdd`, `/deepen`, `/triage`, and the `/plan-feature` Pocock extensions as full skill bodies — rubric, agent defaults, and measure script — before wiring them into the suggestion pipeline. The mechanical reason: the suggestion engine derives invocation signals from the skill body's rubric. A stub with no parseable rubric cannot emit a signal the adoption system can count; counting stub invocations would corrupt the ADR-0050 baseline from day one with meaningless entries indistinguishable from real invocations.

"Full" means: rubric complete, measure script present, agent defaults set. Thursday's commits meet that definition. "Reachable in a session" requires core #99 to merge — that is a separate condition that Thursday's work deliberately left as the next sequenced step, not an oversight. The PR was staged as a clean installation unit so the symlink targets could be reviewed independently of the skill body commits.

### Bead prefix reservations prevent namespace collisions (ADR-0054 companion) — *tooling-for-iteration*

ADR-0054's companion reserves bead prefix namespaces per skill domain: standup beads get `standup:`, foundry beads get `foundry:`, and so on. Without prefix reservations, two skills writing to the same `.handoff/` ledger could produce beads with identical event types, and the audit walker would silently merge them — the adoption signal would be wrong and there would be no error to catch it.

Honest caveat: the reservations are currently enforced by convention documented in the ADR, not by a validator or lint rule. A documented convention prevents the bug only as long as every skill author reads the ADR. A bead-schema validator that rejects writes with unregistered prefixes is the missing mechanical enforcement. This is a known gap; it belongs on the backlog before a second skill author joins the project.
