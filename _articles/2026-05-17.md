## The decisions

### Hammerspoon + app-bundle belong in workstation-yuri, not core — *tooling-for-iteration*

`core` is the portable skill catalog and slash-command engine for the ojfbot fleet — a collection of repos (including `workstation-yuri`, `daily-logger`, `selfco`, `selfco-box`, and others) that share a common Claude Code skill runtime. For `core` to be usable on a CI runner or a second developer machine, it cannot carry macOS-only, display-layout-specific, or single-machine dependencies.

Hammerspoon fails all three disqualifiers: it is macOS-only, window-layout-specific, and tied to the physical workstation named 'yuri'. The test for any future capability is the same three questions: Is it OS-specific? Is it tied to a display profile or hardware layout? Is it named after or configured for a single machine? Three 'yes' answers means `workstation-yuri`. Any 'no' that implies it should work on a CI runner means `core`.

This rule should be the single sentence in `workstation-yuri/CLAUDE.md`. Commit `1029c2b` established the boundary by extraction — that's not sufficient to make it stick. Codifying the rule before the next ambiguous capability lands is the follow-through this commit requires.

Core ADR-0064 documents the broader orchestration pattern: workstation modules are consumers of `core`'s skill engine that extend it with local capabilities. The short version: `core` provides the runtime; `workstation-yuri` adds the machine-specific layer on top.

### Workstation modules are first-class citizens in the ojfbot fleet — *assistant-centric*

`workstation-yuri` is not a scratch directory — it is the machine-level analogue of a sub-app in the ojfbot fleet. It consumes `core`'s skill engine, extends it with local capabilities (Hammerspoon, display profiles, app bundles), and owns its own ADR history.

Treating it as a proper repo boundary means future fleet nodes — a CI runner, a second developer machine, a headless selfco-box — follow the same extraction pattern rather than accumulating platform-specific concerns inside `core`. The fleet currently has one named workstation (yuri); the extraction pattern this commit establishes is the template for any node that follows.

One honest gap: the full fleet topology isn't documented in a single place. The repos active in the ojfbot fleet include at minimum `core`, `workstation-yuri`, `daily-logger`, `asset-foundry`, `landing`, `beaverGame`, `core-reader`, `core-notion-docs`, `selfco`, and `selfco-box`. A repo map — even a one-paragraph description of how these relate — would make every subsequent architectural decision legible to a new reader or a handoff recipient. That document doesn't exist yet.
