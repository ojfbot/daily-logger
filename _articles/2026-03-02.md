## The decisions

**`shellMode` as the single-prop contract (Pillar 1 — Assistant-centric architecture):** One boolean, four remotes. The alternative — shell reaching into remote DOM to suppress chrome — would couple to component trees and break under MF hot-reload. `shellMode` is intentionally coarse: suppress all duplicate chrome in one flag, trust each remote to implement it correctly. What it defers is session ownership: when you're inside TripPlanner mounted in Frame, how do you navigate back? The prop solves chrome duplication; session model waits for shell #6. Building the prop before writing that spec was the right order — you can't write the interface contract for a pattern you haven't proven across multiple remotes.

> **What does "proven across four remotes" actually validate?** cv-builder, BlogEngine, TripPlanner, and the CoreReader slot each implement the pattern. But CoreReader's is a static registration — there's no remote component rendering yet. The contract is proven on three live remotes. The fourth entry proves the registration surface, not the prop. When CoreReader UI ships, it will be the first `shellMode` implementation written against a documented spec rather than discovered through iteration.

**`vite-plugin-css-injected-by-js` scoped to the exposed chunk (Pillar 2 — Tooling for fast iteration):** The naive config injects all CSS. We scope to the Federation-exposed chunk in each app's Vite config explicitly. The plugin doesn't enforce this — convention and code review do. The rationale lives across three PRs' commit history, not in an ADR. A new remote author has to read history to understand a non-obvious constraint. Closing this gap before CoreReader UI onboarding is required, not optional.

**SSE race caught in review (Pillar 3 — Model behavior as design discipline):** The multi-phase streaming race in TripPlanner #11 was identified during integration testing, not discovered in a deployed environment. Treating streaming behavior as a design problem — specifying the ready-signal contract before writing the implementation — is what made it catchable in review. This is Pillar 3 in practice: model behavior specified and tested as a first-class design artifact, not left to runtime debugging. The same discipline that drives cv-builder's visual regression CI applies here at the agent streaming layer.

**Register before implement — the CoreReader pattern:** CoreReader hit a milestone today that isn't a user-visible feature but is architecturally significant: it is now a named remote in the shell's MF loader. This means the next CoreReader PR has a proven integration surface to mount against, and integration bugs surface against a static registration rather than a feature-complete remote. The lesson from TripPlanner's long gap between registration and #11 is that deferred integration is expensive. CoreReader's path is known; the infrastructure is in place; what remains is the work.
