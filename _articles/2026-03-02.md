## The decisions

**`shellMode` as the single-prop contract (Pillar 1 — Assistant-centric architecture):** One boolean, four remotes. The alternative — shell reaching into remote DOM to suppress chrome — would couple to component trees and break under MF hot-reload. `shellMode` is intentionally coarse: suppress all duplicate chrome in one flag, trust each remote to implement it correctly. What it doesn't answer is session ownership: when you're inside TripPlanner mounted in Frame, how do you navigate back? The prop solves chrome duplication; it defers session model to shell #6. Building the prop before writing that spec was the right order — you can't write the interface contract for a pattern you haven't proven across multiple remotes.

**`vite-plugin-css-injected-by-js` scoped to the exposed chunk (Pillar 2 — Tooling for fast iteration):** The naive config injects all CSS. We scope to the Federation-exposed chunk in each app's Vite config explicitly. The plugin doesn't enforce this — convention and code review do. The rationale lives across three PRs' commit history. It does not live in an ADR. A new remote author has to read history to understand a non-obvious constraint. Closing this gap before CoreReader UI onboarding is required, not optional.

**SSE race caught in review (Pillar 3 — Model behavior as design discipline):** The multi-phase streaming race in TripPlanner #11 was identified during integration testing, not discovered in a deployed environment. Treating streaming behavior as a design problem — specifying the ready-signal contract before writing the implementation — is what made it catchable in review. This is Pillar 3 in practice: model behavior specified and tested as a first-class design artifact, not left to runtime debugging.
