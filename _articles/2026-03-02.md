## The decisions

**`vite-plugin-css-injected-by-js` scoped to the exposed chunk only.** The naive approach injects all CSS into the JS bundle. We scope it to `__federation_expose_Dashboard` so unrelated styles don't bloat the remote. This is a constraint the plugin doesn't enforce by default — we enforce it explicitly in each app's Vite config. ADR covering this constraint is a flagged gap: the rationale exists in commit history and in this log, but there is no written spec that travels with the code. That needs to be closed before a new remote author can onboard without reading history.

**`shellMode` as the single prop crossing the host/remote boundary.** Each remote gets one boolean that suppresses its own header, sidenav, and settings chrome when mounted inside Frame. The alternative — the shell reaching into remote DOM to hide elements — would be fragile and untestable. One prop, one contract. Four remotes now implement it identically.

What `shellMode` doesn't yet answer is the user's mental model: when someone is inside TripPlanner mounted in Frame, do they know which app owns the current context, and how do they navigate back to something else? The prop solves duplicate chrome. The session model — who owns context, what "being in Frame" means versus "being in a sub-app" — is an open product question this sprint didn't address. Shell #6 (settings modal migration) is the next place this gets forced.

An ADR for `shellMode` as a formal interface spec is a flagged next action. The contract is stable and running across four remotes; the written spec that would let CoreReader and future remotes implement it without reading prior PRs doesn't exist yet.

**CoreReader registered as a first-class remote before feature-complete UI.** The registration pattern (MF config + app-switcher entry + `frame-dev.sh` entry) is cheap to add early and expensive to retrofit later. Getting it into the shell now means it participates in the same routing and launch infrastructure as the production apps from day one.

The legitimate concern here: TripPlanner #9 has been open since December 2025 and is now superseded, and MrPlug #22 has been open since December 2025 as acknowledged Phase 5 work. That's a pattern of things added without cleanup. The discipline that prevents CoreReader registration from becoming the same debt is explicit: CoreReader is registered with ADR-0010 and a UX research brief already committed (`65f3e4a`, `cfc649f`), and its next milestone (UI work) is the first item on Phase 3. It has a scope. TripPlanner #9 will be closed this week with a reference to #11.

**CoreReader as Frame OS dashboard — the thesis.** The UX research brief committed at `cfc649f` argues that a reading and synthesis surface is the right default view because Frame's primary value is the assistant orchestrating across apps. A task-switcher or home screen puts navigation first; a reader-as-dashboard puts synthesis first — the assistant's output, not the app list, is what the user sees when they arrive. That's the bet. It's consistent with the assistant-centric framing of Frame OS as a whole.

**`transition: all` is always wrong for layout-adjacent panels.** The sidebar bug in TripPlanner (#15) and the cv-builder fix (#105) are the same root cause: `transition: all` fires on every animatable property including width, height, and padding changes triggered by unrelated layout recalcs. An explicit `right 0.2s cubic-bezier(0.4,0,0.2,1)` is deterministic. We applied it to every remote in the same sweep rather than waiting for regressions to surface per-app.

**Claude Code review workflow switched to `workflow_dispatch` across all four repos.** Automatic PR triggers on Claude Code review were generating noise on every push. Manual-only via `workflow_dispatch` + `pr_number` input matches the pattern already in place in core.
