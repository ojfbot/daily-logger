## The decisions

**`vite-plugin-css-injected-by-js` scoped to the exposed chunk only (Pillar 2 — Tooling for fast iteration):**

The naive approach injects all CSS into the JS bundle. We scope it to `__federation_expose_Dashboard` so unrelated styles don't bloat the remote. This constraint isn't enforced by the plugin by default — we enforce it explicitly in each app's Vite config. An ADR covering this constraint is a flagged gap: the rationale exists in commit history and in this log, but there is no written spec that travels with the code. That needs to be closed before a new remote author can onboard without reading history.

**`shellMode` as the single prop crossing the host/remote boundary:**

Each remote gets one boolean that suppresses its own header, sidenav, and settings chrome when mounted inside Frame. The alternative — the shell reaching into remote DOM to hide elements — would be fragile and untestable. One prop, one contract. Four remotes now implement it identically. What `shellMode` doesn't yet answer is the user's mental model: when someone is inside TripPlanner mounted in Frame, how do they navigate back? The prop solves duplicate chrome. The session model — who owns context, what "being in Frame" means — is an open product question that shell #6 (settings modal migration) is the next place it gets forced.

**CoreReader registered as a first-class remote before feature-complete UI:**

The registration pattern (MF config + app-switcher entry + `frame-dev.sh` entry) is cheap to add early and expensive to retrofit. Getting it into the shell now means it participates in the same routing and launch infrastructure as the production apps from day one. The legitimate concern: this pattern has produced debt before (TripPlanner #9, MrPlug #22 — both open since December). CoreReader's registration is different in one concrete way: ADR-0010 and a UX research brief are already committed (`65f3e4a`, `cfc649f`), and its next milestone (UI work) is the first item on Phase 3. It has a scope and a written rationale.

**`transition: all` is always wrong for layout-adjacent panels:**

The sidebar bug in TripPlanner (#15) and the cv-builder fix (#105) are the same root cause: `transition: all` fires on every animatable property including width, height, and padding changes triggered by unrelated layout recalcs. An explicit `right 0.2s cubic-bezier(0.4,0,0.2,1)` is deterministic. Applied to every remote in the same sweep rather than waiting for regressions to surface per-app.
