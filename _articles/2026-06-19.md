## The decisions

### Dual-recorder design for skill:acted — CLI + Stop-hook — *model-behavior-as-design*

The capture runtime emits the `skill:acted` signal through two independent recorders. The CLI recorder fires on explicit invocation (the agent or developer calls it directly). The Stop-hook recorder fires automatically when a Claude session ends — it captures everything that happened in the session without requiring the agent to self-report. Running both closes the gap where an agent invokes a skill but forgets to emit the signal explicitly.

The deduplication question is real: if a developer manually invokes the CLI recorder *and* the session ends naturally, can the same action produce two `skill:acted` events? The current design does not deduplicate across recorders — the two paths are intended to capture structurally distinct events (deliberate explicit invocation vs. session-end summary), so double-counting is not expected in normal operation. If that assumption proves wrong in the first validation pass, deduplication logic is the next ADR. This is an acknowledged open question, not a resolved one.

What 'the light is on' looks like: after a session, inspect the telemetry log file in the repo. A successful emission from either path appends a timestamped `skill:acted` entry. If the log has no new entries after a session in which `/validate` was invoked, the capture runtime is not working.

### Fix fnm default alias under launchd before emitting telemetry — *tooling-for-iteration*

The Stop-hook fires under launchd, which does not source a user's shell profile — so `node` resolves via the system PATH, not `fnm`'s shim directory. The fix pins `fnm default` to the required Node version inside the hook script itself (not the launchd plist), so the hook's node invocation reaches the right binary regardless of shell context. Shipping the runtime without this fix would have produced silent failures: the hook fires, finds no `node`, exits 127 with no visible error, and telemetry goes dark.

If a future `fnm install` of a new Node version breaks this again, the place to look is the hook script's path resolution block — the fix is a one-line `fnm default <version>` call that needs to be updated to match the new pinned version. The launchd plist itself does not need to change.

### ADR-0096: skill-architecture taxonomy as a standing audit instrument — *assistant-centric*

ADR-0096 classifies every skill in the catalog along four axes: **scope** (session-local vs. cross-session), **trigger** (explicit invocation vs. automatic), **side-effects** (read-only vs. mutating), and **composability** (standalone vs. depends-on-other-skills). In practice, each skill gets a structured annotation block — currently a YAML frontmatter section in its definition file — that records its values on each axis. A skill that cannot be classified on any axis is a candidate for decomposition or retirement.

This matters because the catalog has grown organically. Without a taxonomy, dead skills and duplicate responsibilities are invisible until a session wastes time invoking the wrong one. The taxonomy makes the catalog self-auditing.

What the first audit pass would likely surface: with 35 catalog skills and only `/validate` invoked in Thursday's sessions, the majority of skills have no recent invocation record. ADR-0096 gives a framework for distinguishing legitimately dormant skills (correct scope, rarely triggered by design) from genuinely unreachable ones (no plausible trigger in current session contexts). The first audit hasn't run yet — that's a named next step, not an oversight.
