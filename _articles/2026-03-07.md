### MF-manifest-as-routing-authority: why this design, and what breaks

[MrPlug] #25 routes GitHub issue creation by reading the Module Federation remote manifest at inspection time to determine which remote "owns" the inspected element. This is **assistant-centric architecture (Samir Pillar 1)** expressed at the tooling layer: MrPlug isn't a passive screenshot tool — it's an agent that understands Frame's topology.

The manifest is the right routing oracle because it's the only runtime-authoritative source for the remote→repo mapping. The alternatives — hard-coded URL patterns, a separate routing table owned by frame-agent, asking the user — all fail in the same direction: they go stale as soon as a new remote registers. The manifest doesn't.

**Mechanically:** The manifest is expected to be a JSON structure fetchable at a known Frame endpoint (the precise URL and whether it's `window.__webpack_remotes__`, a `/remotes.json` endpoint, or a frame-agent API is not confirmed in the current project context — this is a gap the ADR needs to close). A manifest parse failure today falls back to prompting the user to select a repo manually. That fallback exists in Phase 5A's production path, which doesn't route through frame-agent at all.

**The coupling risk is live, not theoretical.** Phase 1 shell extraction is still open. If that work changes the manifest schema, MrPlug's routing breaks silently — no exception, no visible error, just wrong repo assignment. The mitigation is the user-selection fallback, but that fallback is currently *specified*, not *designed*. "Ask the user which repo" can be a graceful informative prompt or a broken empty state. Which one exists in the Phase 5A codebase is not confirmed here. **This is the gap the ADR must close before #25 exits draft** — not after.

> **ADR scope:** Document (1) why the MF manifest is the routing authority, (2) what the manifest fetch mechanism is and where that code lives, (3) what the fallback UX looks like and whether it's been designed or only specified, (4) the trigger condition for switching from frame-agent path to production path.

### Claude Code relay: the contract that needs to be written

The second new surface in #25 is a relay that forwards MrPlug-generated context — screenshot, element selector, AI analysis — to Claude Code via the background service worker. A developer inspecting a layout bug goes from "I see this bug" to "there's a GitHub issue with a screenshot, acceptance criteria, and a Claude Code context payload" in one click.

The relay doesn't invoke Claude Code directly. It structures the payload so that when a developer opens Claude Code in that repo, the context is already waiting. **Where it waits is not confirmed in the current project context.** The candidate mechanisms — a file written to the repo (e.g., `.claude/context.json`), a local socket, a clipboard artifact — have meaningfully different security and reliability profiles. This is an unresolved interface contract between two systems, and it isn't documented in an ADR anywhere visible in the project context.

That contract needs to exist before #25 exits draft. If MrPlug generates one payload shape and Claude Code expects another, the relay fails silently — same failure mode as the manifest coupling risk. **The security boundary is correct:** MrPlug should never have write access to a developer's active Claude Code session, and the separation (MrPlug generates context, Claude Code consumes it) is the right design. But "correct boundary" and "documented contract" are two different things. Right now only the first exists.

### Security boundary: structural, not advisory

Phase 5A established the rule: AI calls live in the background service worker, not the content script. Phase 5B extends that rule to GitHub API calls. The content script captures the element and screenshot; the background worker owns all authenticated network calls.

This is **security as emergent UX (Samir Pillar 4):** the user never sees a permission prompt for GitHub auth in a content script context because that call never happens there. The trust boundary is structural, not advisory.

**What's not yet confirmed:** There's no lint rule, CSP header, or integration test in the current project context that would catch a GitHub API call surfacing in the content script. The `/hardening` action below is a verification step, not a rubber stamp — if that enforcement mechanism doesn't exist, name it as a gap, don't claim the boundary is settled.
