## The decisions

### Article-quality verification layer: embed the ADR registry and PR state buckets into every generation prompt — *model-behavior-as-design*

Monday's article cited ADR-0067 with incorrect framing and described shell PR #76 as open when it had already merged. Both errors had the same root cause: the ground-truth data was not in the generation context, so the drafter had no mechanism to verify its own claims.

Before `5ab335f`, the pipeline had no structured pre-generation context injection — ground-truth inclusion was a blind spot from the start, not a previously attempted mechanism that was abandoned. `5ab335f` adds a structured pre-generation step that reads the swept ADR registry and the three PR state buckets (merged, open, closed-without-merge) and injects them as authoritative input before the article is drafted. If a claim cannot be verified from those tables, the drafter must not make it.

Mechanically: the step is a script in the daily-logger pipeline that reads the local ADR index and the PR state sweep output, serialises both as structured tables, and prepends them to the generation prompt. The failure output when a claim can't be verified is a flagged placeholder rather than a confident assertion. The layer is merged but unvalidated against a live failure case — see the `/validate` action.

'Validated by Claude Opus' in the four cleanup PRs is a named gate in the cleanup workflow: Opus reviews the diff for unintended deletions or scope creep before merge. It is not a general quality certification — it is specifically scoped to confirming the cleanup pass touched only stale comments and did not alter functional code.

### Cross-link ADRs as a fleet-visibility mechanism: ADR-0069 in core points to asset-foundry ADR-0011 — *tooling-for-iteration*

The dual Blender transport decision lives in asset-foundry's ADR-0011 — the owning repo holds the decision. Core is the fleet-level knowledge index. ADR-0069 in core registers the existence of ADR-0011 and names the contract it establishes, so any agent or developer reading core's decision log can follow the cross-link rather than discovering the transport architecture by accident.

The governance rule for when a cross-link ADR is warranted versus a README reference: a cross-link ADR is written when the decision in the owning repo establishes a contract that other repos in the fleet must be aware of to make compatible decisions. A README reference is sufficient for informational pointers. ADR-0011 establishes a transport contract that beaverGame and the asset pipeline depend on — that clears the bar. A two-line index entry would not be discoverable from a new session's ADR scan of core.

The owning repo holds the decision; the fleet index holds the pointer. This is the same mechanism used for any cross-repo architectural dependency.

### Stale-action triage as a hygiene gate: closing 8 actions older than 14 days prevents list rot — *tooling-for-iteration*

The open actions list had grown to 27 entries, several pre-dating current work by weeks. The `/frame-standup` triage in `0e41da1` applied a 14-day rule: any action where the corresponding work is demonstrably complete or the context has changed enough to make the original prescription obsolete gets closed.

To address the audit concern directly: the 8 closures were not batch-closed against the rule alone. Each was reviewed individually with a closure reason — either 'corresponding work merged' (named commit or PR) or 'context superseded' (named subsequent decision or ADR). The 14-day rule was the trigger for review, not the closure justification itself. Recourse if something was closed incorrectly: the action IDs are preserved in the triage commit (`0e41da1`) and can be re-opened with a correcting commit citing the original action ID.

Letting the list grow unbounded degrades signal — the actions that genuinely need attention get buried under stale entries that will never be acted on.
