## The decisions

### ADR-0098 accepted: two-track skill telemetry — use-funnel and evolution stream never blend — *model-behavior-as-design*

The skill-telemetry system now maintains two permanently separate accounting streams: the use-funnel (was a skill invoked? was a suggestion followed?) and the evolution stream (is a skill improving? is the authoring discriminator getting better?). Blending them would corrupt both — a heavily-used but poorly-written skill would look healthier than it is; a rarely-used but well-authored skill would look sicker. ADR-0098's acceptance is encoded in commit `2a373a5` as part of S16. The ADR document lives in `core`'s ADR directory; if you need the canonical one-line decision statement and the formally rejected alternative (the blended stream), look there — the decisions section here is a summary, not a substitute. What the ADR formally prohibits: any code path that aggregates use-funnel and evolution-stream counts into a single composite score.

### Population-tagged scoring: installed vs. uninstalled vs. legacy are never collapsed — *model-behavior-as-design*

S3 (`ee1aae7`) and morning-cockpit PR #31 establish that suggestion populations must be scored and rendered with their population tag intact. Collapsing installed and legacy suggestions into a single funnel denominator produces a meaningless follow-rate: the denominator grows with every legacy (un-followable) suggestion, driving the rate toward zero regardless of actual agent behavior. The fix is mechanical — tag every suggestion at capture time, separate the funnels, render zeros explicitly — but the discipline must be enforced at the data model level, not patched at display time. Patching at display time only hides the corruption; the corrupted data remains available to downstream callers.

Tags: `skill-telemetry`, `assistant-centric`

### Single reconciler writer declared: the duplicate-entry RCA — *assistant-centric*

S4 (`f501b25`) resolves an ambiguity that was producing duplicate northstar entries: more than one code path was writing to the reconciler, so the same movement could appear twice in the ledger. The fix is not a deduplication step at read time — it is a declaration that core's own reconciler is the single sanctioned writer. This is the same architectural reflex as ADR-0005 in morning-cockpit (Approve & emit is the single sanctioned write path for the cockpit): when you have two writers, you don't have eventual consistency, you have a race. Name one writer and enforce it. Note: S4's commit message documents the RCA, but no standalone ADR yet formalises the single-reconciler-writer pattern. The `/adr` action item below addresses this gap.

Tags: `skill-telemetry`, `adr`, `assistant-centric`

### The ruler fails loudly: S17 measurement-plumbing reliability — *tooling-for-iteration*

S17 (`7c65b7d`) flips the failure mode of the measurement infrastructure from silent degradation to loud failure. When the plumbing breaks — a scorer returns undefined, a join misses a population, a file path drifts — the system now halts and surfaces the error rather than producing a 0% follow rate that looks like agent non-compliance. This is the same principle as a circuit breaker: a metric that silently returns wrong data is more dangerous than a metric that refuses to return data at all, because the wrong data can be acted on. A 0% follow rate that is actually a plumbing failure will trigger the wrong intervention — debugging agent behavior when the problem is in the scoring harness.

### Loop pane as independently-degradable read-model: three sources, none blocking — *tooling-for-iteration*

The morning-cockpit Loop pane (PR #30, `6f5b8e0`) reads from three independently-degradable read-only sources: `~/selfco/tracking/skill-telemetry.jsonl`, the odometer file, and the audit freshness record. If any source is missing or stale, the pane renders what it has and explicitly marks the degraded section — it does not block render or return an error. To verify degradation behavior locally: remove or rename one of the three source files and re-run the cockpit; the affected section should render with an explicit staleness or missing-data marker rather than a blank or an exception. This is the same degradation contract as the Delivery pane's three sources. The pattern is now a first-class architectural norm: a read-model that degrades gracefully is more honest than one that hides its data gaps behind a loading spinner.
