> **Why register everything at once?** A sub-app that's partially registered creates silent runtime failures. The checklist in `/scaffold-frame-app` exists precisely because we've hit each of these failure modes in prior app additions.

## The decisions

### Decision 1: `remoteEntry.js` must be `no-store`, everything else `immutable`

The Vercel config in lean-canvas #3 applies two different caching strategies to two different asset classes. This is a correctness requirement for Module Federation, not a micro-optimisation.

**Why?** `remoteEntry.js` is the manifest the MF host fetches at runtime to discover what a remote exposes. If it's cached with `immutable` (or even a long `max-age`), the host loads stale module definitions after a deploy. `no-store` means every page load fetches the current manifest. The cost is one small network round-trip per session; the benefit is that deploys take effect immediately.

All other static assets — JS chunks, CSS, fonts — are content-hashed by Vite. Their filenames change on every build, so `immutable` caching is safe and desirable: once fetched, they never need to be re-fetched.

**What breaks if you get this wrong?** A shared library version mismatch between host and remote, caught at runtime as a silent module resolution failure or a duplicate React instance that breaks hooks. This looks like a random rendering error and takes hours to trace back to a stale manifest.

**Cost acknowledgment:** `no-store` on `remoteEntry.js` means one uncached manifest fetch per user session. At 1,000 users doing 10 sessions a day, that's 10,000 manifest fetches per day against Vercel's edge network. Whether that's within the free tier or starts accruing cost depends on the Vercel plan and `remoteEntry.js` file size — a number not available in this article. The correctness argument for `no-store` is sound; the operational cost at scale has not been calculated. Track it when the user cohort grows.

To verify the header is set in the live deployment: `curl -I https://<lean-canvas-vercel-url>/remoteEntry.js` — look for `cache-control: no-store` in the response.

This demonstrates the **tooling for fast iteration** pillar: deploy infrastructure that makes every push safe to ship requires getting caching semantics right by default, not as a later fix.

### Decision 2: /scaffold-frame-app as a machine-readable checklist, not prose documentation

The 28-item checklist in [core] #23 is structured as a skill that Claude Code executes against the actual file tree, not a README section a developer has to remember to read. Each item is a binary assertion — *does `APP_CONFIG` contain this key? Does `.env.production` contain this variable? Is this port in `frame-dev.sh`?* — evaluated by Claude reading files, not by a human ticking boxes.

**Why a skill rather than a wiki page?** Wiki pages drift. The `/scaffold-frame-app` skill is invoked at scaffolding time, so it validates the output of the process that just ran. When a new integration point is added to the shell (as gastown-pilot's additions demonstrated this week), the checklist gets a new item — and every future scaffolding run is evaluated against the updated spec.

**What would break if we skipped this?** The gastown-pilot integration required touching seven distinct files across three repos. Omitting any one of them produces a different failure mode. Without a checklist, the only way to discover a gap is to run the app and interpret a cryptic error. The skill makes the gap visible before the first `npm run dev`.

### Decision 3: `AppType` as the discriminated union source of truth

`AppType` in shell is a TypeScript discriminated union. Every integration point — `APP_CONFIG`, `REMOTE_LOADERS`, `frame-agent` routing — is typed against it. Adding a new string literal to `AppType` makes the compiler flag every location that exhaustively switches on the union.

**Important caveat on enforcement coverage:** The compiler catches missing variants in exhaustive `switch` statements — primarily the `frame-agent` routing handlers. `REMOTE_LOADERS` and `SETTINGS_LOADERS` are likely keyed on `AppType` strings rather than exhaustively switched, which means TypeScript will not catch a missing entry there at compile time — it will be `undefined` at runtime. The `/scaffold-frame-app` checklist is the enforcement mechanism for those non-switch integration points. The compiler and the skill together cover the full seven integration points; neither covers them alone.

**What breaks if `AppType` is incomplete?** The TypeScript compiler narrows the union at every exhaustive switch in `frame-agent` routing. A missing variant means the default branch handles the new app — typically a no-op or error log, not a routed agent call. The app loads but natural-language commands addressed to it are silently dropped. No exception is thrown.

**The extensibility question:** A discriminated union requires a compile-time change in the shell for every new app. This is intentional — the shell is the authoritative registry of what Frame knows about. It also means Frame currently has no model for external app developers who cannot merge into the shell repo. That is a product decision about Frame's extensibility boundary. As of this writing, there is no external developer model, but the decision deserves to be written down as an ADR if that boundary is expected to hold.

**The tradeoff:** A string map with no type constraints allows runtime-only registration but removes the compiler's ability to catch missing integration points. For a system where each app has seven integration points across three repos, the compile-time guarantee is worth the friction at current team size.

This demonstrates the **assistant-centric architecture** pillar: `frame-agent` routing is the mechanism by which the shell's natural-language interface knows which agent handles which app. Keeping that routing exhaustively typed is what makes the assistant reliable.
