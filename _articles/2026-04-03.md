### The structural gap the hardening sprint exposed: tsc --noEmit is not a required CI step — *tooling-for-iteration*

TripPlanner #42 and BlogEngine #43 both fixed type errors discovered post-merge. The root cause: tsc --noEmit was not wired as a blocking CI job. Here's the Express 5 failure mechanically — Express 5 changed the type of route handler params from `string` to `string | ParsedQs`, which is a narrower match than most handler signatures expected. pnpm lock file rebuilds that bumped @types/express pulled in this type change silently; TypeScript never ran in CI, so the incompatibility passed review and broke only at compile time in a downstream step. The fix is not more careful review — it's making the compiler a required reviewer. Concretely: add a tsc --noEmit step to ci.yml (or the equivalent workflow file in each repo) after the lint job and before the build job, then add it to branch protection rules as a required status check. Two repos are confirmed affected — TripPlanner and BlogEngine. The remaining Express-backed repos in the fleet should be audited before the next lock file rebuild cycle, not parked for Phase 2, because the same silent regression path is live in any repo where @types/express is a dependency and tsc is not required in CI.
