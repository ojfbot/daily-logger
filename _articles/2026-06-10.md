## Why the chronicle slept

**The mechanism.** `GH_PAT` was set in daily-logger's repo secrets on 2026-02-25 and expired inside the outage window. Every `gh api` call in the daily sweep failed against the dead token — and every failure was swallowed. `ghApi()` (`src/collect-context.ts:38`) pipes stderr to `/dev/null` and a bare catch returns `null` with a "skipped" warning. So `collectContext` returned an empty context, and `shouldSkipRun` (`src/should-skip-run.ts` — the skip-bot-only-days logic merged 05-11, a sensible feature on its own) looked at each empty day and retired it as "no activity." The pipeline did exactly what it was told with the data it had. The data was a lie it had built itself.

One timeline item remains open: a token created 2026-02-25 with a 90-day lifetime expires 2026-05-26, but the silence starts 05-19 — seven days earlier. Either the secrets-set date isn't the token-creation date, or the first week has a second cause. The exact expiry is visible on the token settings page and corroborated by the cleaner's first failure timestamp in the Actions tab; pinning it down is part of closing this incident. Until then, "roughly 90 days" is flagged as an approximation, not a diagnosis.

**The diagnosis was already in the Actions tab.** The Daily Cleaner (`.github/workflows/daily-cleaner.yml`) failed loudly, twice a day, for the entire window — same expired token. The difference: the cleaner has an auth smoke-test step and `daily-blog.yml` did not. The evidence sat there for three weeks, attributed to the wrong workflow.

**April predicted this.** The 04-27 digest closed with: *"Write the workflow-failure alert. This is the new tech-debt item. The credit-depletion failure was invisible for five days because nothing was watching."* It was never built. This outage is that exact debt collecting interest — the same alerting gap, expressed through a different credential. In April it was an Anthropic billing boundary; in June a GitHub token lifetime. The only reason either failure surfaced is that a human eventually looked.

**A second blind spot, independent of the token:** four of the window's most active repos weren't registered in the sweep at all. `f1-pit-wall`, `f1-substrate`, `lofi-beaver`, and `golf-platform-scripts` were all created or first-pushed during the silence — none are in the `REPOS` array (`src/collect-context.ts:14`). Even with a healthy token, the pipeline would have chronicled a fleet that no longer matched reality. `morning-cockpit` got registered on the final day (PR #206); the others still need it.

**The fix, precisely.** [daily-logger] PR #208 adds two checks:

- `collectContext` now runs a `gh api rate_limit` preflight and throws on token rejection instead of swallowing it (`src/collect-context.ts`, top of `collectContext()`).
- `daily-blog.yml` gained a "Smoke test sweep auth" step running `gh auth status` before the sweep — the check the cleaner always had and the blog workflow lacked.

> **Why `rate_limit`, not `/user`?** Fine-grained PATs and the Actions installation token can 403 on `/user` even when perfectly healthy. A `/user` preflight would have introduced a new false-failure mode; `rate_limit` authenticates both. This is the load-bearing decision of PR #208 — if you grep for `gh api user`, you'll find nothing, on purpose.

The `GH_PAT` rotation itself is still pending — an account-level action no PR can perform.

## The decisions

**Slug-as-identity for ADRs (core ADR-0087)** — the window's keystone governance decision. *Mechanism:* the slug is the permanent identifier; serials are assigned at accept and never reserved; cross-references are written `adr:<slug>`; revisions happen in place. *Where you'll feel it:* when writing a new ADR, you no longer pick a number — and a `traces:` value that doesn't resolve on disk fails lint. Backfilled across all 78 records rather than grandfathered.

**Shadow-before-enforce, everywhere** — the Control-Gated Slices pattern (ADR-0086 + `/gated-slice` skill) was codified and then visibly practiced: the loading-discipline gate shipped a shadow stage before binding (ADR-0081 Slice 2), the vault lint gate graduated advisory → CI-blocking only after its shadow period (ADR-0089), the defuddle extraction trial is collecting a ~15–20-sample data gate before promotion (ADR-0090), and the sprite-modality ADR in asset-foundry is deliberately held at Proposed pending a second consumer. The fleet's first closed-loop metric — the vault lint rig feeding per-skill eval — was declared 06-07.

**Subagents: default-deny, define none (core ADR-0082)** — ratified. *Mechanism:* no repo defines subagents; any task that seems to want one must instead be decomposed into sequential skill invocations or explicitly justify an exception in an ADR. *Where you'll feel it:* large parallel-feeling work (fleet-wide audits, multi-repo rollouts) gets sliced and sequenced rather than fanned out.

**Auth failures are now load-bearing (PR #208)** — preflight `gh api rate_limit`, throw on rejection, `gh auth status` smoke test in the workflow. The implicit doctrine: a sweep that can't authenticate must crash, not chronicle an empty world. One gap: this doctrine currently lives in a PR description. The fleet just rebuilt its decision-record system; this decision should be recorded with it — an ADR for the auth-failure doctrine is part of closing the incident, not optional polish.
