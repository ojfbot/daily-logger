## The decisions

### ADR-0034: Isolated Redux stores per remote, coordinated via FrameBus — *assistant-centric*

Each Module Federation remote owns its own Redux store; cross-app state flows through a FrameBus pub/sub bus rather than a shared store. A single shared store creates hard compile-time coupling between the shell and every remote — breaking independent deployability, the core promise of federation. Mechanically: when a remote dispatches a state change that other remotes need to observe, it publishes a typed FrameBus event; the shell does not re-broadcast; subscriber remotes listen directly. The event catalog is owned by the shell repo as the authoritative schema. FrameBus is not yet wired in a running remote — ADR-0034 is a closed decision, but the integration is a deferred implementation. The `/validate` action exists to produce a proof-of-concept in cv-builder before the pattern spreads. The architecture is designed for a fleet of 5–15 remotes; at that scale the pub/sub traversal overhead is negligible and the isolation benefit is substantial. Load-testing the bus path has not been done.

Tags: `module-federation`, `shell`, `arch`

### ADR-0033: Three-tier confidence threshold for daily-cleaner bot — *model-behavior-as-design*

The daily-cleaner pipeline classifies every proposed edit as `high_confidence_stale`, `medium_confidence_stale`, or `uncertain`. The first two tiers merge automatically; `uncertain` edits are logged, surfaced in the daily-logger UI as pending review items, and do not merge without explicit human approval. Without this policy, the bot would either apply risky rewrites silently or never merge anything. The three-tier model is the minimum viable safety gate for an automated documentation pruner.

Tags: `adr`, `daily-logger`, `tooling-for-iteration`

### ADR-0036: Structured decision output drives rich UI — *tooling-for-iteration*

The `decisions` array in the article schema is now a typed, structured object rather than a freeform markdown block, enabling the blog UI to render collapsible accordions, filter by pillar, and link decisions to code references. Keeping it as markdown would mean parsing intent out of prose every time the UI wants to do anything programmatic with it.

Tags: `adr`, `daily-logger`, `phase-1`

### Filing cv-builder #99: name the minification workaround before it compounds — *tooling-for-iteration*

The current `minify: false` flag in `packages/browser-app/vite.config.ts` disables minification for the entire cv-builder bundle — not just the federation chunk — shipping ~1.6 MB of readable source to production. The flag was set at bundle scope because `@originjs/vite-plugin-federation` 1.4.1 doesn't expose per-chunk minification control. Filing #99 formally prevents this from becoming a silent assumption in future perf work and surfaces the version-gate investigation path. This is background debt rather than an active incident — no demo or phase gate is currently blocked — but it is a live production regression on a portfolio URL.

Tags: `module-federation`, `cv-builder`, `phase-1`

### MrPlug relay security hardening before Phase 5 promotion — *security-as-emergent-ux*

PR #32 addressed missing origin validation and unhandled error branches in the Chrome extension's message-passing path. MrPlug is a browser extension relay: it sits between untrusted page content (arbitrary web pages that can send postMessage events) and a privileged Claude Code session. The specific vulnerability class was unvalidated message origin — a malicious page could craft a relay payload that passed through to the Claude Code session without schema enforcement — combined with error branches that could leak session state in their responses. PR #32 closes these vectors. The hardening is a prerequisite for Phase 5's Frame dev-companion rebuild, where MrPlug moves further up the privilege stack.
