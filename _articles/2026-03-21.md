> **Why a build gate and not just a visual test?**
> Visual regression tests (pixelmatch, Playwright) are expensive and require a stable baseline. A build gate is cheap and catches the most common failure mode: a story that imports a component with a broken prop contract or a missing provider. The correct sequence is: (1) get components into Storybook, (2) enforce that Storybook builds, (3) add visual baselines once the build is stable. Skipping step 2 is the specific failure mode worth naming: if a story breaks silently — returns a stale build artifact or exits with a non-fatal error — the pixelmatch step runs against stale output and passes vacuously. You lose the visual regression signal without any indication it's gone. The build gate closes that hole.

## The decisions

### ADR-0029: why prop-only is a hard rule, not a guideline

**[core] #25** doesn't just document a pattern — it closes an architectural escape hatch. The precise tradeoff:

`@ojfbot/shell` is a shared package consumed by cv-builder, BlogEngine, and TripPlanner as a Module Federation remote. If a component in `packages/ui/` imports `useSelector` from `@reduxjs/toolkit`, it creates an implicit dependency on *a* Redux store being present at render time. But each sub-app runs its own store. The shell package has no store of its own at the package boundary level.

> **What breaks if you violate this?**
> A component calling `useSelector` inside `@ojfbot/shell` either (a) throws at runtime because there's no `<Provider>` in scope, or (b) silently reads from the wrong store if the sub-app's Provider happens to be an ancestor. Both failure modes are invisible at build time and hard to trace in production. The prop-only rule makes the failure impossible by construction.

**The canonical example to read first:** `packages/ui/SettingsModal.tsx` (pure presenter) + its `*Connected.tsx` sibling. The connected wrapper does the `useSelector`/`useDispatch` calls and passes results as props. The presenter knows nothing about the store. Storybook stories for `SettingsModal` require no Provider and no mock store — you pass in data, assert on render.

**The alternative considered:** co-locating a minimal Redux store inside `@ojfbot/shell` and sharing it via Module Federation's `shared` config. Rejected because: (1) components that know about state require a mock store in Storybook, adding Provider boilerplate to every story; (2) they can't be reused in a different state-management context without a store shim; (3) the federation `shared` config for a store introduces version-pinning constraints across all three sub-apps simultaneously. All three costs are paid at the exact moment you're trying to move fast on new sub-app extraction.

**Enforcement is incomplete.** The Storybook build gate enforces the rule at render time if no mock Provider is present. It does not enforce it statically. A lint rule flagging redux imports inside `packages/ui/` is the missing layer — it is not yet written. That gap is named here so it doesn't disappear.

This is **Pillar 3 — model behavior as design discipline** applied to component architecture: the prop contract is the spec. Storybook stories are the evals. A component that can't render without a Provider fails the eval before it ships.

### Why the Storybook build gate comes before visual regression

The sequencing is deliberate. Visual regression with pixelmatch is already running in cv-builder's CI. Adding it to BlogEngine and TripPlanner before those repos had Storybook would have gated on an unstable surface.

The build → enforce → baseline order has a specific failure mode at each skipped step. If you skip step 2: a story that breaks silently (stale artifact, non-fatal TypeScript error, missing provider at story level) causes the pixelmatch step to run against stale output. It passes. You believe your visual regression is working. It isn't. The build gate is a forcing function: a broken story is a merge blocker instead of a "we'll fix it later" item.

**ADR-0030 is a prerequisite, not a suggestion.** The article's earlier draft listed it as a suggested action. That was wrong. "Document the sequencing rationale as ADR-0030 before adding pixelmatch to BlogEngine or TripPlanner" is a gate on the next CI step. It belongs in the Phase 1 status list as NOT STARTED — blocks visual regression rollout. It is listed there now.

### Docker CI as warning, not failure

The Docker startup fix in cv-builder (`2c6374c`, `182c5e2`) is a pragmatic call: the containerised Playwright pipeline is the most operationally fragile part of cv-builder's CI.

> **What breaks with the alternative?**
> If Docker startup failures remain blocking, a transient runner resource contention event on GitHub Actions — not a code regression — holds up a legitimate refactor PR. The signal value of a startup failure is low: it tells you the runner couldn't start the container, not that the application is broken. Demoting it to a warning preserves the actual signal (Playwright test results) while removing the false-negative merge blocker.

The remaining question — whether the warning surfaces visibly or disappears in a collapsed CI step — is not confirmed. If it disappears, it defeats the purpose of the change. That confirmation is a tracked item, not a techdebt punt. The Docker flake rate that drove this decision is also not documented in the available context; it should be, because "infrastructure flakes" without a number is an explanation that doesn't survive scrutiny.
