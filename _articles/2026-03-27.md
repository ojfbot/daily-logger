## The decisions

### Decision 1: `LLMClient` interface in GroupThink — provider abstraction before you need it

GroupThink started with a single hardcoded Anthropic call. Adding Ollama could have been a simple `if (useLocal)` branch. We didn't do that.

Instead, PR #5 introduced a `LLMClient` interface that both Anthropic and Ollama implement. The interface is minimal — approximately:

```typescript
interface LLMClient {
  complete(prompt: string): Promise<string>;
}
```

The call site never references a concrete provider. Anthropic and Ollama are the two current implementations; adding OpenAI or any other provider means implementing `complete()` and wiring the constructor — nothing else changes.

**Why does this matter?**
The moment you have two providers, the question "which one am I calling?" starts leaking into every layer that touches the LLM call — configuration, error handling, retry logic, prompt formatting. An interface boundary answers that question once, at the construction site, and keeps it out of the pipeline entirely. The alternative — an `if/else` branch in the grouping pipeline — means every future provider multiplies the conditional surface. You end up with logic that is simultaneously hard to test and hard to extend.

**On test coverage:** the testability argument is currently forward-looking, not realized. There is no test suite that exercises both Anthropic and Ollama against the same contract. The interface makes that test straightforward to write — a mock `LLMClient` can be injected anywhere the concrete providers are used — but the tests don't exist yet. This should be honest: the abstraction is correct, the test coverage is the next obligation.

**On cost:** `qwen2.5:7b` on Apple Silicon is free at the margin; Anthropic API calls are not. The `LLMClient` interface is also a cost dial — switch providers at the construction site and the entire pipeline runs locally. At current GroupThink usage volume the API cost isn't meaningful, but the architecture makes it trivially switchable if that changes. This framing — provider abstraction as economic flexibility — is worth carrying forward as the fleet grows.

The tradeoff: a `LLMClient` interface is mild over-engineering if you're certain there will only ever be two providers. We're not certain. The pattern costs almost nothing to introduce now and would cost a full refactor to retrofit later.

This is **Samir Pillar 3 — model behaviour as design discipline**. The prompt optimisation pass for local models (`3d41e31`) is not a performance hack — it is a deliberate behaviour specification. `qwen2.5:7b` on Apple Silicon has different latency and context-length constraints than Claude. Writing to those constraints is the same discipline as writing a persona prompt for an orchestrator agent: the model's behaviour is a design surface, not an implementation detail. The validation step (running both providers on the same tab set and comparing cluster output) will tell us whether the trim held semantic fidelity — that result will be published when it exists.

### Decision 2: Two fields, not one, in fleet registration

When `landing` was added to the fleet, two commits were required — not one:

1. Add to the `REPOS` array in `collect-context.ts` (sweep inclusion)
2. Add to the system prompt in `generate-article.ts` (LLM context)

These are different concerns. The first determines which commits get collected. The second determines whether the LLM that writes this article understands what those commits mean.

**What would break if you only did one?**
If you add a repo to the sweep but not the system prompt, commits appear in the 24h window but the article generator has no schema for them — it can't correctly categorise the work, may misattribute it to an existing repo, or hedges with vague language. If you add to the system prompt but not the sweep, the LLM has context that never gets exercised — a documentation/reality drift that compounds silently. Neither failure is loud. Both are the kind of thing you debug two weeks later wondering why the article got a repo wrong.

Core PR [#32](https://github.com/ojfbot/core/pull/32) codified this as a two-step checklist item in `/scaffold-app`. Thursday's `landing` addition was the first time that checklist was exercised against a real new repo. It worked. That's a meaningful signal — but the checklist is currently a document someone reads, not a programmatic gate. The institutional knowledge is written down; it isn't yet encoded. Converting Step 7.2 into an automated validation (e.g., a pre-commit check that compares the `REPOS` array against the system prompt's repo list) would make it structural. That's the right next move for **Samir Pillar 2 — tooling for fast iteration**.

### Decision 3: Resumé PDF as a repo asset in `landing`

The resumé PDF is committed directly to the `landing` repo rather than hosted externally (e.g., on S3 or a CDN). Vercel deploys it as a static asset, the URL is stable, and there are no external dependencies to rotate or expire. The cost is repo size. For a document that changes at most a few times a year, that's an acceptable tradeoff — and keeping it in the repo means a git history of every version, which is the right audit trail for a job-search artifact.
