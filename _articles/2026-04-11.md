Friday was a fleet-wide sprint in the most literal sense: 75 commits across 12 repos, 59 PRs merged in the surrounding week, and a single architectural decision at the centre of it all. The AgentBead bridge — ADR-0043 — landed in `core` and propagated outward, giving every Frame sub-app a uniform `GET /api/beads` endpoint and a session coordination layer that tracks skill invocations, commits, and PRs as typed beads. When a sprint of this shape lands, the right question isn't "what moved?" but "what is now structurally different?" The answer: Frame has a nervous system — though its sensory output is not yet visible. The endpoints exist and return the correct shape; the end-to-end aggregation path has not been validated yet, and no UI is rendering bead signals. That's the next gate, named plainly here rather than buried in a suggested action.

### shell

The `/api/beads` aggregation endpoint upgraded to Dolt-first with filesystem fallback — shell is the Mayor host that collects beads from all sub-apps. The Dolt-first strategy was chosen because bead history needs diff and branch semantics: Dolt is a version-controlled SQL store that lets you query bead history across time without losing the git-like auditability the rest of the system depends on. SQLite has no branch semantics; Postgres has no diff primitive. Filesystem fallback means every sub-app can start emitting beads to disk immediately while Dolt adoption rolls out — the aggregation layer is never blocked by storage infra not yet in place. CI fixes also landed: PR comment permissions and fleet pattern alignment for the browser-automation workflow, plus a cv-builder clone step for tsconfig dependency resolution. Important caveat: the filesystem fallback path has not yet been validated end-to-end. The `/validate` action below is a prerequisite, not a polish step.

## The decisions

### ADR-0043: AgentBead bridge — mapping Claude Code lifecycle to Gas Town bead emissions — *assistant-centric*

Every Claude Code agent lifecycle event (session create, idle, task sling) now emits a typed bead via `bead-emit.mjs` commands. ADR-0043 is published at `docs/adr/0043-agent-bead-bridge.md` in the `core` repo. The bridge lives in Claude Code post-tool-use hooks — shell scripts that fire on agent lifecycle transitions — not in agent code itself. This means the agent has no knowledge of Gas Town: it runs its tasks, and the hooks observe and emit. The decision record covers the rationale against alternatives: a sidecar was rejected because it introduces a network dependency on every agent start; a message bus was rejected because the bead volume doesn't justify the operational overhead; hooks were chosen because they're zero-dependency, inspectable in `.claude/settings.json`, and co-located with the skills they observe.

Tags: `gas-town`, `assistant-centric`, `core`

### Dolt-first with filesystem fallback for shell's /api/beads aggregation — *tooling-for-iteration*

Shell's Mayor aggregation endpoint tries Dolt storage first and falls back to filesystem bead files if Dolt isn't available. Dolt was chosen over SQLite or Postgres because bead history needs diff and branch semantics — you need to query what beads existed at a prior commit, not just at the current state. The filesystem fallback path allows every sub-app to start emitting beads to disk immediately; the aggregation layer is never blocked by storage infrastructure not yet deployed. Bead files follow the convention `{repo}/.beads/{session-id}/{event-type}.json`. When Dolt comes online, filesystem beads will be migrated via an import script — they are not orphaned, but migration is not automatic and has not been written yet. That is a known gap.

Tags: `gas-town`, `shell`, `tooling-for-iteration`

### GET /api/beads as a fleet-wide ADR-0016 contract, not a per-app feature — *model-behavior-as-design*

Every sub-app now exposes `GET /api/beads` using the typed `FrameBeadLike` shape — `{ id, type, app, prefix, payload, createdAt }` — defined in `core/packages/types/src/bead.ts`, with an app-specific prefix (`pure-`, `blog-`, `trip-`, `seh-`, `lean-`). ADR-0016 governs the projection pattern (how domain objects map to bead types and prefixes). The endpoints intentionally return empty arrays until persistent storage is wired — the shape contract is established before the data exists. This prevents the Mayor aggregator in shell from having to handle per-app response shapes. To be direct about the current state: BlogEngine is the only sub-app returning live bead data today. lean-canvas returns empty by design. The other apps return beads only when agent sessions are active. The fleet-wide contract is real; a fleet-wide data flow is not yet real. A dedicated ADR documenting the fleet extension of ADR-0016 — covering the prefix convention, the empty-return pattern, and the Mayor aggregation strategy — should be written now that 8 repos implement it.

Tags: `gas-town`, `assistant-centric`, `shell`

### Silent mock fallback removal in TripPlanner's modify-prompt flow — *security-as-emergent-ux*

The `useImportAgent.handleGenerateModifications` function had a fallback path that silently returned mock data when `/modify-prompt` doesn't exist, making the feature appear to work in development. PR #53 replaces this with explicit stub state — a disabled button with explanatory copy — that makes the unimplemented path visible to anyone running the app. The tradeoff is intentional: the UI now shows an unimplemented state instead of mock content. False-positive UX in dev leads to shipping broken features to production; a clearly unimplemented state is the correct signal.
