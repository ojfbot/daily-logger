## The decisions

### Use `Request<{ param: string }>` generics rather than casting `req.params` at the use site — *tooling-for-iteration*

When `@types/express` tightened its params typing — moving `req.params` fields from implicit `any` to `string | undefined` in the updated type definitions — the safe fix is to thread the param shape into the route handler's generic slot: `Request<{ threadId: string }>`. The alternative is casting at the use site: `req.params.threadId as string`. The cast silences the compiler but enforces nothing — if the route is renamed or the param removed, the cast still compiles. The generic approach pushes the contract to the boundary where Express resolves it, so TypeScript catches mismatches structurally.

The recognition signal: after a lock file rebuild touching `@types/express`, you will see errors of the form `Type 'string | undefined' is not assignable to type 'string'` on `req.params.*` accesses in route handlers. The before-state is an untyped handler signature (`(req: Request, res: Response)`); the after-state adds the params shape as the first generic argument (`(req: Request<{ threadId: string }>, res: Response)`). Two lines changed per handler.

This pattern will recur on any repo where Express route handlers were written against the older, looser typings.

### Lock file rebuilds that touch `@types/*` or framework packages require `tsc --noEmit` as a required CI step before the PR is considered done — *tooling-for-iteration*

TripPlanner PR #41 and BlogEngine PR #42 rebuilt lock files for transitive vulnerability remediation. Both passed CI and merged. Both introduced downstream type breakage that required follow-up PRs (#42 and #43 respectively) to close.

How the build succeeded while types were broken: `tsc --noEmit` was not in the CI pipeline. The build step compiled successfully because TypeScript in build mode can tolerate paths that a strict `--noEmit` pass would reject — particularly implicit `any` coercions that were already present in the codebase and were silently widened further by the new type definitions. The build did not fail because nothing in the build path forced a strict type resolution of `req.params`.

The protocol to encode: any lock file rebuild that touches `@types/*` or framework packages (Express, LangChain, React) must run `pnpm exec tsc --noEmit` as a required CI job — separate from the build step, blocking merge — before the PR is marked complete. A build pass is not a type-check pass.

This protocol is not yet documented as a standing ADR. The `/adr` action in the suggested actions below is the step to encode it. Until that ADR exists, treat this decision entry as the authoritative statement of the protocol.
