## The decisions

### FrameBus PoC scoped as ADR-0013 + Playwright e2e in a single PR — *assistant-centric*

Shell PR #58 bundles three previously deferred backlog items — ADR-0013, the FrameBus proof-of-concept, and an end-to-end Playwright test — into one reviewable unit. FrameBus is a typed pub/sub event bus running in the shell that child apps (cv-builder, BlogEngine, TripPlanner) subscribe to via Module Federation's shared runtime. It replaces ad-hoc postMessage calls with a typed event catalog — a TypeScript discriminated union of all cross-domain event shapes — so that the shell can fanout a single user action (e.g., opening the command bar) to multiple micro-frontends simultaneously without each app implementing its own listener. ADR-0013 decides in favor of broadcast fanout over point-to-point routing, and async delivery over synchronous. The alternative — a central Zustand slice shared via Module Federation — was rejected because it couples app state to shell lifecycle. Keeping the ADR and the PoC in the same PR ensures the architectural rationale is never separated from the code that implements it; the most common failure mode is ADRs written speculatively before working code, then never reconciled when the implementation diverges. Shell #58 was opened Saturday, 2026-04-05.

### CI review workflow switched to manual trigger in MrPlug — *security-as-emergent-ux*

MrPlug is the project's GitHub Actions workflow library — it hosts the Claude Code Review bot, the stale-docs cleaner, and the security audit runner used across the fleet. It uses happy-dom (a headless DOM implementation) to run workflow validation tests without a browser. PR #37 replaces the automatic pull_request trigger on the Claude Code Review workflow with workflow_dispatch, requiring a PR number as an explicit input. The proximate cause: the daily bot pair was generating several PRs per day (stale-doc cleanups, daily-logger commits), each triggering an Opus-backed code review that added latency, API cost, and reviewer noise with no signal return on bot-authored diffs. This is a concrete instance of security-as-emergent-UX — a project design principle that high-cost or high-authority agent actions should require explicit human invocation rather than firing on ambient events. The principle isn't post-hoc labeling: it's the same reasoning behind requiring a PR number as input (rather than inferring it from context) and behind Claude Opus's confidence-gated edit acceptance in the stale-doc cleaner. MrPlug #37 has no dependency on #36; the two PRs touch different workflow files and can be reviewed and merged in either order.
