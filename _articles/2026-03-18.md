None of these are UI polish; all three are load-bearing architecture. ## What shipped

### shell — NL instance spawning, thread resumption, G3 Approval Queue (`3621e5c`)

**[shell] #37** merged — the largest shell PR since cross-domain fan-out. Three platform capabilities land together:

- **Natural-language instance spawning** — a user typing "open a new TripPlanner for Paris" from the chat bar now triggers ShellAgent to instantiate a fresh app window without any sidebar click. The Module Federation remote is loaded, a new Redux thread slice is initialised, and the app is focused — all from a single NL utterance. If the Federation remote is unavailable at load time, ShellAgent surfaces an error and the Redux thread slice is not initialized; the approval queue does not render a partial state.
- **Thread resumption** — paused or backgrounded sessions can be resumed by name or by recency. ShellAgent reconstructs the conversation context from the thread store and hands control back to the sub-app's agent. **Important:** this covers the Redux state side only. The MrPlug relay context re-injection is tracked separately in [shell] #24 (still open). Thread resumption as shipped is functional for apps that do not depend on relay context; for apps that do, it is a partial capability until #24 closes.
- **G3 Approval Queue** — before any high-impact cross-app action (fan-out writes, instance destruction, privileged tool calls), ShellAgent now routes through an approval-queue UI. The user sees a structured diff of what will change before the action fires, and can edit the plan — not just approve or cancel. ShellAgent produces a declarative plan object before execution; the queue renders that plan. The user is approving a description of intent, not a post-hoc reconstruction. **The suggested action to verify the structured diff for a cross-domain fan-out write is still outstanding** — Phase 4 is merged, not fully verified. That distinction is real and matters for production readiness.

## The decisions

### Decision 1: NL spawning via ShellAgent, not a sidebar affordance

**[shell] #37** deliberately routes NL instance spawning through ShellAgent rather than adding a "new window" button to the sidebar. The alternative — a dedicated UI control — would be faster to implement and easier to test, but it contradicts the central thesis of Frame: **the assistant is the interface**.

If spawning goes through a button, the user learns "click here to open apps." If it goes through ShellAgent, the user learns "tell Frame what you want." The second pattern scales to arbitrary app definitions; the first requires a new button for every new capability. This is the same bet Dia makes with its AI address bar — one input surface, unlimited intent surface. "A button has one outcome; an NL utterance has a distribution."

The known failure mode of this bet: ShellAgent can guess wrong. If a user says "open TripPlanner for Paris" and two TripPlanner instances are already running, ShellAgent has to resolve the ambiguity. The Approval Queue handles "do you understand the blast radius" — it does not handle "ShellAgent misidentified which instance you meant." Ambiguity resolution under multi-instance spawning is not yet specified; it is an open design gap that [shell] #5 must address. The queue is not a substitute for getting the intent right.

The tradeoff is mitigated, not eliminated, by the G3 Approval Queue. **Samir pillar: Assistant-centric architecture.**

---

### Decision 2: G3 Approval Queue as the security primitive, not permission dialogs

The Approval Queue is not a modal saying "Are you sure?" It is a structured diff of the intended state change — which apps will open, which threads will be modified, which tool calls will fire — rendered before execution. The user can edit the plan, not just approve or cancel it.

The critical distinction: **a confirm dialog answers "did the user consent?" The Approval Queue answers "does the user understand what they're consenting to?"** For cross-domain fan-out writes — where ShellAgent might touch three sub-apps in one utterance — a binary confirm is not sufficient UX. The user needs to see the blast radius before it fires.

Mechanically: ShellAgent produces a declarative plan object before any action executes. The queue renders that plan. Execution is a separate step gated on user confirmation. The queue is not reconstructing intent from an action log; it is displaying the plan before the plan runs. This is why the verification action item in What's next is a prerequisite for treating cross-domain fan-out as production-ready — the correctness of the queue under complex fan-out has not yet been confirmed with a real test case.

This is **Security as emergent UX** — the fourth Samir pillar. The queue is the mechanism by which NL spawning is safe to ship at all, not a feature bolted on afterward.

---

### Decision 3: Gas Town A2–A8 in `@core/workflows` before shell adoption

The Gas Town execution primitives live in `@core/workflows`, not in the shell repo. The shell and cv-builder will consume them as package dependencies once [shell] #21 and [cv-builder] #111 proceed. Shipping the primitives first and the adopters second defines the contract once and references it many times.

`@core/workflows` is the source of truth for atom registration, dependency resolution, and budget governance. If Gas Town had been implemented directly inside the shell, cv-builder would need its own copy — and two copies of a dependency graph protocol will eventually drift. The package boundary is the durability guarantee.

**When does this boundary become a problem?** If `@core/workflows` requires a breaking change while shell and cv-builder are mid-adoption at different versions, both consumers must coordinate upgrades together. That coordination cost is real. The mitigation is semantic versioning and explicit deprecation windows — neither of which is currently documented. That is a known gap to address before both consumers are deep in adoption. **Samir pillar: Tooling for fast iteration.**

---

### Decision 4: CoreReader write-capability in Phase 5, not Phase 1

The OKR edit flow was deferred until the read tabs were stable. Writing to `frame-os-context.md` through a UI is a non-trivial operation — a bad write corrupts the metadata that every daily-logger prompt depends on. Phasing read-before-write validated the GET paths against real data before any PUT surface was exposed.

**What does failure look like now that the write surface is live?** A malformed OKR write would corrupt `frame-os-context.md`. The first observable symptom would be daily-logger producing incoherent or context-free prompts on the next run. The recovery path is a manual git revert of `frame-os-context.md`. The optimistic Redux rollback handles network failures but does not undo a persisted write. A formal write-validation layer — schema validation before the PUT fires — has not been scoped yet. The phase gate protected development; it does not protect production.

The sequencing was right. The production gap it leaves is real and should be the next scoped item against [core-reader] #7.

### On [shell] #5 — this is a hard stop, not a suggestion

[shell] #5 (Redux store isolation under multi-instance spawning) is not an "important open issue with an action item attached." It is a hard stop on second-instance integration testing.

**What breaks today if a user spawns two TripPlanner instances?** The current implementation has not been tested for this case. Based on the store architecture, the likely outcome is silent state collision — the second instance's thread slice overwrites keys in the first instance's slice, producing subtly wrong state in both rather than a crash. This is the worst failure mode: no error, wrong behavior, hard to diagnose. It is possible the current spawn implementation is artificially scoped to prevent a second instance — but if so, that constraint is not documented and would not survive a determined user.

**The gate:** no second-instance integration test gets written until ADR-0020 (per-instance Redux slice namespacing) is drafted and reviewed. ADR-0020 is the next action on this codebase, not an optional follow-up.
