### The silent degradation path ('emit the prompt on failure') must be replaced with a hard error and a defined fallback protocol — *tooling-for-iteration*

The generation pipeline currently has no validated fallback: when the LLM response fails schema validation, times out, or returns malformed output, the system emits the raw input context rather than raising. That is a trust problem for a load-bearing daily habit. The fix is two-part: (1) the pipeline must raise a hard, observable error on invalid output rather than degrading silently; (2) there must be a documented fallback protocol — at minimum: fail loudly, log the failure with a short manual substitute, reference a tracking issue. Neither exists today. Both need to.

## What's next

Two things need to happen before the next generation run. First: the pipeline needs a hard-error path — if the LLM response fails schema validation or is otherwise invalid, raise loudly and do not emit the prompt. Second: a fallback protocol needs to be written down, even if it is one paragraph — what happens when generation fails, who writes the substitute, and where the failure gets tracked. Without both of those, this incident repeats with the same result. Additionally, if the raw LLM output from Monday's failed run is recoverable, reviewing it would clarify whether the root cause was context-window overflow, schema mismatch, timeout, or model refusal — each of those has a different fix.
