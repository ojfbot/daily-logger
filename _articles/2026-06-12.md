## The decisions

### ADR-0004's hybrid NL router: deterministic first, LLM slot-fill fallback — *model-behavior-as-design*

The NL router in `@f1/shared/src/router.ts` runs a deterministic rule pass — keyword and pattern matching against a typed intent corpus — before handing off to the LLM for slot-filling. The seam is explicit: if `routeQuery` resolves an intent with confidence above threshold from the rule set, it returns a typed discriminated union (e.g., `{ type: 'LAP_COMPARE', driver1, driver2, lap }`) directly. If the rule pass misses or underflows confidence, the same union shape is returned but populated via LLM slot-fill against the raw query string. The eval harness (committed in `1a59ff0`) labels queries with expected intent type and expected slot values — a pass means the correct union variant is returned with all required slots populated. The harness has not yet been run; the current corpus size and passing threshold are not yet established. This is the first validation gate before the chat sidebar is demo-safe. For a concrete example: the query 'how did Leclerc's S2 compare to Hamilton's on lap 34?' would be expected to resolve deterministically to `{ type: 'SECTOR_COMPARE', driver1: 'LEC', driver2: 'HAM', sector: 2, lap: 34 }` — the presence of explicit driver codes, a sector number, and a lap number puts it firmly in the deterministic path. A freeform query like 'who was quickest in the middle sector on the soft tyre?' would fall through to LLM slot-fill. This keeps latency low for structured queries while preserving model flexibility for freeform phrasing.

### Per-panel LIVE/pinned replay sync rather than global playhead — *assistant-centric*

Each panel in the pit-wall UI independently decides whether it is tracking the live session or pinned to a replay timestamp — there is no single global playhead. This lets an engineer freeze the delta-overlay panel at a specific lap while the gap-timing chart continues updating live. A global playhead would require coordinating panel state across the Redux store via a single shared slice, which would break the single-panel inspection workflow — you couldn't hold one panel in the past while another runs forward. The user behavior this enables: a race engineer can fix the delta chart at lap 28 (where a strategy call happened), drag the gap chart forward to lap 35, and compare the two states simultaneously without either panel resetting. The failure mode of the chosen approach: if a panel is in LIVE mode and the substrate query returns stale data (e.g., DuckDB is running a slow aggregate), the panel currently shows stale data without an explicit error state. A staleness indicator is not yet implemented — this is a known gap in the current design.

### Delta attribution pushed to f1-substrate, not computed in the UI layer — *tooling-for-iteration*

Per-corner time swings and exact sector deltas are computed by `f1-substrate`'s DuckDB query layer (`064d32a`) and returned as aggregates, rather than being derived in the React layer from raw lap data. The router output (a typed discriminated union) is consumed by `POST /api/route`, which calls the appropriate substrate query and returns the aggregate directly to the UI. The UI layer is a pure display consumer — it receives structured data and renders it. This makes the delta logic testable in isolation at the DuckDB layer, ensures the sector-bests aggregate (`94a02a4`) shares the same query path, and means the React layer never holds intermediate lap arrays in memory. The alternative — deriving deltas client-side from raw lap data — would have required shipping larger payloads to the browser and made the delta logic invisible to substrate-level tests.

### Pipeline-alert issue-ops: failure surfaces as an assigned GitHub issue with push notification — *security-as-emergent-ux*

Context first: in April, the daily-logger pipeline silently failed for 22 days. The workflow returned green (or simply didn't alert), the operator didn't notice, and 22 days of logs were lost. Silent green is worse than loud red. The new alert layer (1d21c5d) opens a GitHub issue assigned to the operator on any generation failure, delivers a push via the GitHub App integration, and exposes a `/disable` command so alert fatigue doesn't cause the layer to be bypassed — which would recreate the same 22-day failure. The suppression state is stored as a GitHub Actions repo-level variable and requires manual re-enable. The hardening test (a suggested action below) needs to verify the suppression path: that `/disable` prevents a second issue on the next failure, and that the variable can be inspected and reset. No test currently covers this path.

### CULTIVATE_MODEL tier: separating ingest model from synthesis model — *tooling-for-iteration*

Introducing `CULTIVATE_MODEL` as a separate env var lets the cultivate (serendipity) pass run on a different — typically faster and cheaper — model than the ingest agent. The cultivate pass runs daily, unattended, and tolerates lower reasoning precision. The ingest pass handles write-path decisions where model quality matters. Conflating them into one model choice was an implicit cost that grew as cultivate volume increased. The exact monthly delta depends on which model pair is configured at deploy time — the commit establishes the mechanism but does not pin the model selection. Tracking the before/after cost estimate is an acknowledged gap; the configuration flexibility is the deliverable here.

### Scaffold skill registers new repos as vault entities at spawn time — *assistant-centric*

Teaching `/scaffold` to emit a vault entity registration at creation time means every new repo enters the fleet already indexed in the selfco knowledge graph — there is no manual step to add it later. Without this, vault coverage lags the actual fleet size by however long it takes someone to remember to register it. The registration is emitted as part of the scaffold skill's completion hook, writing a minimal entity record to the vault at the same moment the repo is created.
