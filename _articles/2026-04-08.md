## The decisions

### Fix CI in commit-by-commit layers, not a single omnibus patch — *tooling-for-iteration*

Each MrPlug CI failure was addressed in its own commit rather than batched into one. This keeps the blame graph readable: if any fix introduces a regression, you can bisect to exactly the right commit without untangling combined intent. The four failures — pnpm version mismatch, missing `--run` flag, pre-existing ESLint violations, pre-existing TypeScript errors — were causally independent, so fixing them separately is also technically correct, not just good hygiene.

Tags: `ci-cd`, `hardening`

### Read pnpm version from packageManager field, not the action config — *tooling-for-iteration*

Removing the explicit `version: 9` from `pnpm/action-setup` and letting it derive the version from the `packageManager` field in `package.json` (pnpm@8.15.0) is the correct approach: the version declaration is co-located with the project, not hidden inside CI YAML. When you pin the version in two places and they diverge — as they did here — the CI runner installs the wrong toolchain and every subsequent step is suspect. Note that the pnpm mismatch and the `--run` flag issue were independent failures: fixing pnpm alone would not have unstuck the test runner.

Tags: `ci-cd`, `hardening`

### Downgrade pre-existing lint violations to warnings rather than silencing them — *tooling-for-iteration*

The six pre-existing ESLint errors (`ban-types` and `ban-ts-comment` violations) were downgraded from `error` to `warn` rather than suppressed with `// eslint-disable` comments. This keeps the violations visible in CI output — they appear in the log and can be promoted back to errors in a targeted cleanup PR — without blocking the pipeline on debt that predates the CI setup. This is a time-bounded tradeoff, not a permanent policy: the violations need a tracked issue and a cleanup PR, or they will calcify into ignored noise. That issue is not yet open as of 2026-04-08 — that's the gap, and it needs to be closed this week.

Tags: `ci-cd`, `hardening`

### Exclude e2e and MCP relay tests from the standard CI run rather than mocking their environment — *security-as-emergent-ux*

MrPlug's e2e suite and MCP relay tests require a running Chrome extension context and a live relay server — neither is available in a GitHub Actions runner. Mechanically, this is implemented with `--exclude '**/e2e/**' --exclude '**/relay/**'` in the Vitest invocation (check `.github/workflows/ci.yml` for the current exact flags). Rather than mocking the environment — which would give false confidence about extension-context behavior — we exclude those files explicitly. The exclusion is visible in config, not hidden, so a future contributor knows exactly what coverage gap exists. The concrete trigger for moving these tests into CI is a maintained test environment that can provision a headless Chrome extension context — that work is not scheduled yet, and that's the honest state.

Tags: `ci-cd`, `hardening`, `phase-5`

### Regenerate shell lockfile as a standalone PR after devDependency additions — *tooling-for-iteration*

Shell PR #62 exists solely to regenerate `pnpm-lock.yaml` after ESLint and typescript-eslint were added in PR #61. Keeping the lockfile update in its own PR makes the `--frozen-lockfile` CI contract explicit: the main branch always has a lockfile that matches the manifest, and any manifest change that doesn't include a lockfile update will fail CI immediately. This is the correct sequencing for all future devDependency additions in the shell repo.
