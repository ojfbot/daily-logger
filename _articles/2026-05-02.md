## The decisions

### Gate window.__beaver debug rigs behind import.meta.env.DEV at the assignment site — *security-as-emergent-ux*

The test-rig methods that Playwright's snap script drives — testFellNearestTree, testBuildDam, setOverviewCamera — were removed from production bundles by checking import.meta.env.DEV at the assignment site rather than at each call site. The distinction matters: per-call-site checks still ship the function bodies to the bundle; a single assignment-site check elides the entire object from the production graph. This keeps the snap script functional in dev without shipping console-accessible engine hooks in the production build. The pattern is auditable: grep for window.__beaver to find every assignment; every one should be inside a DEV guard.

### load-glb validation tripwire as the machine-verifiable cross-repo contract boundary — *tooling-for-iteration*

Every .glb consumed by beaverGame must arrive with a sibling .validation.json containing status: "validated" — enforced at load time by load-glb.ts, now covered by four Vitest rejection-path tests. The four rejection paths are: missing .validation.json entirely; .validation.json present but unparseable JSON; status field absent; status field present but not equal to "validated". Each throws a distinct error at load time rather than silently producing a broken render. The .validation.json is produced by asset-foundry's validator pipeline; a file that never entered that pipeline will have no sibling and will fail the first rejection path. This is the only gate preventing a malformed or unvalidated asset from silently entering the render pipeline — "machine-verifiable rather than documentation-only" is the operative standard.
