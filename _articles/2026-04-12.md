## The decisions

### Suggestion follow-through as a first-class telemetry event — *tooling-for-iteration*

Rather than inferring whether a suggestion was followed by correlating log timestamps, we added explicit follow-through tracking directly in `log-skill`. The matching key is skill ID (not name), with no time-window constraint in the current implementation — any invocation within the same session that matches a prior suggestion ID is counted as a follow-through. This is a known simplification: a three-hour gap between suggestion and invocation would still count. The 0% conversion in Saturday's session is not a matching artifact — the suggestion log entries were verified to reference skills that exist in the catalog. The suggestions were made and not acted on. That is the signal.

Tags: `tooling-for-iteration`, `skill-telemetry`, `ci-cd`

### Post skill usage directly on the PR, not just in a log file — *model-behavior-as-design*

The new `pr-skill-audit.sh` hook posts a structured comment on the GitHub PR associated with the current session. PR association is resolved by reading the current git branch name and calling the GitHub API to find the matching open PR. If no open PR exists for the branch, the hook no-ops silently — a gap worth noting, since it means pre-PR work produces no audit trail. The core argument for PR-posting over log-only: a reviewer can see at a glance whether `/validate` or `/hardening` ran before the PR was opened. Log-only buries the signal where no one looks.

Tags: `model-behavior-as-design`, `assistant-centric`, `ci-cd`

### Telemetry feeds the article generator directly, not a dashboard — *assistant-centric*

Rather than building a separate observability dashboard, the skill adoption report script writes structured data that the daily-logger generator reads at article-generation time. The nightly article becomes the primary observability surface. This is a real architectural tradeoff worth naming: the data is one generation step away from the raw signal, it isn't queryable, it isn't alertable, and prose can misrepresent emphasis. The decision accepts those costs deliberately. In a one-developer project where the developer reads every article, a dashboard is a second surface to maintain with no additional reader. The assistant is the UI — the article is the dashboard. If this project ever adds collaborators or needs alerting, the raw JSONL sources are the stable layer to build on; the article generator is replaceable.

Tags: `assistant-centric`, `daily-logger`, `tooling-for-iteration`

### purefoy ProgressReporter as a non-blocking side-channel — *tooling-for-iteration*

The GPU transcription pipeline is long-running (hours per batch). ProgressReporter writes `progress.json` as a side-channel — it does not sit in the hot path of the audio-to-text loop. This means observability never becomes a performance bottleneck, and the file can be polled by any external monitor without coordinating with the transcription process. The current implementation does not handle write failures or define staleness: if the write fails, transcription continues silently; if `progress.json` stops updating, nothing raises an alert. For a long-running GPU job, knowing what 'broken observability' looks like before you need to debug it is the next hardening item here.
