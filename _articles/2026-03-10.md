### Extracting AppSwitcher first: why the smallest component validates the biggest pattern

[shell] #23 opens Phase 1 with a deliberate minimal scope: extract only `AppSwitcher` from `shell-app` into the new `packages/ui/` directory as `@ojfbot/shell`. This is not the whole extraction — it's a proof of the extraction pattern.

**Why AppSwitcher and not the Header or ChatBar?**
AppSwitcher has the cleanest dependency profile: it reads from Redux for the active app list but doesn't write anything complex back, and it has no Claude API surface. That makes it the lowest-risk candidate to move. If the extraction pattern breaks — import resolution, Module Federation boundaries, Storybook wiring — it breaks cheaply on the simplest component. Fix the pattern once, then apply it to Header and ChatBar.

The PR replaces the Redux-coupled inline component with a version that accepts props directly — decoupling the component from the store so it can be tested in Storybook without a Redux Provider. This is the same tradeoff IBM Carbon Design System makes: components are display primitives; state lives one layer up.

**What would break if we skipped this decoupling?**
Storybook would need a full Redux store mock for every story. That's not impossible, but it means every story file imports the store, which means Storybook becomes a downstream consumer of application state rather than a standalone design system. The value of a design system is environment-agnostic rendering — whether the host is the shell, a test harness, or a Figma plugin.

**What does a "dirty" Module Federation boundary look like?**
A clean MF boundary means the remote (`@ojfbot/shell`) exposes only what's declared in its `exposes` config, and the host (`shell-app`) consumes it only through the shared module resolution map. Dirty means: a component in the remote directly imports from the host's own bundle (circular), or two separately-bundled copies of React end up in the runtime (context split — hooks silently fail across the boundary because each copy has its own `currentDispatcher`). The review criterion for [shell] #23 is specifically: does AppSwitcher import anything from outside `packages/ui/`? If yes, the boundary is dirty and the extraction isn't complete.

**On the visual design layer**: the article documents an architectural extraction, not a visual design decision. AppSwitcher's spacing, typography, and motion language are not defined in [shell] #23 — that layer is explicitly deferred to Phase 2 (Figma MCP integration). This is a conscious scope decision, not an oversight. Phase 1 establishes the structural package boundary and props API; Phase 2 connects it to design tooling. The absence of a Figma file for `@ojfbot/shell` today is by design. It should not be taken as evidence that visual design isn't a category in this project.

This demonstrates the **tooling for fast iteration** pillar: Storybook as the design system source of truth means designers and engineers share a live component reference before any app wires it up — but the design artifacts that Storybook is built *toward* arrive in Phase 2.

### FrameBead as universal work primitive: ADR-0016 and ADR-0017

[core] #17 commits two ADRs in the `adr/` directory of the `[core]` repo.

**What is Gas Town in one sentence?** Gas Town is the shared execution layer — an event-driven activity bus — through which ShellAgent routes work items across independently-authored apps. It is not a task queue in the traditional sense; it's closer to a typed message bus with a manifest-declared routing table and a shell-level aggregation view (the Mayor/Deacon layer).

**ADR-0016** defines FrameBead — the universal work primitive. Every gas-town-aware app maps its domain objects to FrameBeads with a typed prefix (`cv-`, `shell-`, `core-`). Without a shared shape, each app would serialize its work items differently, and the shell — which aggregates activity across all apps as the Mayor/Deacon layer — would need an adapter for every app. FrameBead is the adapter *contract*: if your domain object implements `FrameBeadLike`, the shell can display it, route it, and pass it to the Gas Town bus without knowing what the object represents.

**The `FrameBeadLike` interface** (from ADR-0016) is the grounding claim for this entire section. The interface has not been reproduced inline in this article — that is a gap. At minimum, a reader should know the required fields: the typed `id` (prefixed string), a `status` discriminant, a `sourceApp` identifier, and a `payload` generic. The full TypeScript shape lives in `[core]` ADR-0016. Before [cv-builder] #111 Sprint 1 begins implementing FrameBeads for CVs, that interface should be the first thing the implementing developer reads.

**ADR-0017** formalizes singleton enforcement — the architectural decision that only one instance of certain app types (e.g., shell itself, core-reader) can be active at a time. This complements last week's multi-instance UI work in [shell] #19: multi-instance by default, with explicit singleton overrides declared in the app manifest. When ShellAgent receives a "launch core-reader" command, it checks the singleton flag before spawning. If an instance exists, it focuses it rather than duplicating it. Without this enforcement, the Gas Town Mayor layer would receive duplicate activity streams from two core-reader instances with no principled deduplication path.

**What does singleton enforcement look like in failure?** ADR-0017 names the happy path — focus existing instance — but the failure modes are not documented in this article. What happens if ShellAgent crashes between the check and the spawn? What if the manifest flag is missing or malformed? These are open questions acknowledged here; the ADR should address them explicitly before [shell] #21 (Mayor host) begins implementation.

**What does a FrameBead look like in the UI?** That answer does not exist yet. ADR-0016 defines the data contract; the render contract — card, row, notification, aggregation view — is upstream design work. This is the same deferred-visual-layer caveat as the AppSwitcher extraction. The architecture is documented. The experience will be designed in Phase 2.

This demonstrates **assistant-centric architecture**: ShellAgent's routing decisions are governed by manifest-declared constraints, not ad-hoc conditionals scattered through the UI.

### Thread resumption synthesis: naming the gap before building the bridge

[shell] #24 is a filed issue, not a PR. It names a real architectural gap: the relay from FeedbackModal to Claude Code is a one-way pipe. MrPlug sends context *to* Claude Code, but the modal never learns what Claude Code did. The relay is a launcher, not a companion.

**What the issue sketches as a solution**: a structured relay payload that includes a diff summary and a Claude Code session ID, which FeedbackModal uses to reconstruct the thread. **This architecture is aspirational.** Claude Code does not expose a stable session ID through a documented public API surface — this is an open design question, not a settled mechanism. The issue is genuinely in design phase. The concrete open questions are: (1) what callback mechanism does Claude Code expose, if any — webhook, filesystem write, stdout pipe? (2) what is the minimal diff summary format that FeedbackModal needs to reconstruct useful thread state? (3) does the relay payload travel back through the same MrPlug channel or a separate return channel?

This is **model behavior as design discipline** applied to a UX gap: the relay payload isn't just a transport mechanism, it's a conversational artifact that must carry enough structure for the receiving UI to synthesize a coherent thread state. The gap was known as a limitation at Phase 5B merge time and was consciously deferred — this is disciplined scope management, not a missed requirement discovered post-ship.

**[core-reader] #4** (opened 2026-03-10) adds `GET /api/beads` as a Vercel serverless function. `parseADRsAsBeads()` reads ADR manifest files from the `[core]` repo and maps them to `FrameBeadLike` objects — this is the first live implementation of the FrameBead mapping pattern. **Deployment status**: as of 2026-03-10, [core-reader] #4 is an open PR — it is not yet merged or deployed to a live Vercel URL. **On the Vercel infrastructure choice**: deploying Gas Town API surfaces as Vercel serverless functions is a convenience choice appropriate to the prototype phase, not a finalized runtime decision. At current traffic (internal team, no external consumers), cost is negligible. If Gas Town API surfaces proliferate and attract external consumers, the Vercel-per-function model will need to be re-evaluated against a dedicated runtime. That decision is not urgent today but should be made explicitly before [shell] #21 (Mayor host, which will call these endpoints) enters implementation.

**On `parseADRsAsBeads()` mechanics**: the function reads ADR markdown files, extracts the YAML frontmatter (title, status, date, tags), and maps them to `FrameBeadLike` objects with `id: "core-adr-{number}"`, `sourceApp: "core-reader"`, `status` derived from the ADR status field, and `payload` containing the full frontmatter. This is the mapping pattern every subsequent Gas Town app will replicate with its own prefix and domain fields. If this mapping is wrong, every downstream implementation inherits the wrongness — [core-reader] #4 review should include a concrete assertion against a known ADR's expected output.

### On the shared runtime category

The Frame architecture — a shell hosting independently-authored apps, each mapping domain objects to a shared primitive (FrameBead), with singleton enforcement and a Mayor-layer aggregation view — is an instance of the shared runtime problem: how do independently-built experiences compose into a coherent environment with a common session model? This is the same category of problem that large platform environments have been solving for years at scale. Frame's approach differs in that the composition layer is language-model-mediated: ShellAgent, not a static routing table, decides what to surface and when. That distinction is what makes the FrameBead contract and singleton enforcement architectural decisions rather than implementation details — they define the contract that makes LLM-mediated routing principled.
