**How the pipeline works mechanically:** The daily-cleaner runs on a schedule (not triggered by a push event). It scans the repo for comments and docstrings, sends each candidate to Opus with a prompt that asks for a categorical judgment — `high_confidence_stale`, `medium_confidence_stale`, or `uncertain` — along with a one-sentence rationale. Only the first two categories are included in the PR. `uncertain` items are silently skipped. The PR is then created automatically; a human can inspect the diff before merge. **Why "high and medium confidence only" is a real architectural decision, not a label:** The daily-cleaner isn't diffing against a spec — it's asking a model to judge whether a comment still describes reality. False positives (deleting accurate comments) are worse than false negatives (leaving stale ones). Opus validation exists precisely because GPT-class models can be confidently wrong about code intent. By gating on high and medium confidence, the pipeline accepts some stale residue in exchange for not silently corrupting the documentation corpus.

## The decisions

### The bot-first maintenance model: why it holds up under low-velocity days

On a high-velocity week — GroupThink went from zero to Ollama support in five days — stale documentation accumulates faster than any developer can chase it. The cleaner pipeline is a forcing function: it runs every day, surfaces what's drifted, and uses a confidence-gated model review to decide what's safe to remove. The developer doesn't schedule "docs cleanup." The system schedules it.

This is **Pillar 2: Tooling for fast iteration**. The argument isn't that bots replace developer judgment — it's that automated, confidence-gated maintenance removes the activation energy cost of hygiene work. When the cost of keeping docs accurate drops to zero, accuracy becomes the default state rather than a periodic project.

The tradeoff is real. Confidence gating is a heuristic, not a guarantee. The mitigation is PR review — every cleaner PR is a diff a developer can inspect before merge. But as flagged above: whether that review happened for PR #6 on a Friday is unconfirmed. The pipeline automates the work. It doesn't remove human accountability for the output — but that accountability only functions if someone actually opens the diff. That's the gap to close before this pattern extends to the rest of the fleet.

One more gap to name honestly: "high confidence" and "medium confidence" are operationally defined by the prompt language Opus receives, not by a formal ADR. The suggested action to `/adr` this threshold isn't a future nicety — it means the most consequential gate in the pipeline currently runs on an informal prompt heuristic. Someone debugging a bad deletion today has no documented threshold to trace back to.
