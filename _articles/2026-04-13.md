## The decisions

### suggested_after and layer_affinity as first-class fields in the skill catalog — *model-behavior-as-design*

Two new fields were added to the skill catalog schema. suggested_after lists skill IDs that contextually precede the current skill (e.g. /validate after /scaffold). layer_affinity encodes which orchestration layer a skill belongs to — planning, execution, or review. At runtime, the suggest-skills engine fires after a skill invocation and checks the just-run skill's ID against other skills' suggested_after lists to find what should come next; layer_affinity is used to filter or scope those suggestions to the appropriate orchestration context. When a skill has no suggested_after entry pointing to it, no suggestion fires — this is a silent no-op by design, not a throw. That behavior is the intended contract but is not yet covered by a test; verifying it is part of Monday's /validate pass. The chosen direction — catalog fields rather than runtime inference — was taken because inference would require session history heuristics that don't yet exist and would make suggestion behavior opaque. The ADR documenting these fields, their read semantics, and the no-match behavior is listed as an open action below. The catalog itself lives in core and is currently writable by any contributor with core access; access control and update governance for the catalog are not yet formally specified.

### suggest-skills.mjs as a reusable engine, not inline hook logic — *tooling-for-iteration*

Rather than embedding suggestion logic inside the orchestration hook, Sunday's refactor extracted it into a standalone suggest-skills.mjs module that both the orchestration hook and the suggest-skill hook import. Suggestion logic is cross-cutting: the same catalog-reading and layer-filtering behavior needs to fire in multiple hook contexts without duplication. A single module is the only way to keep that logic consistent across both entry points. Inline logic was the prior state — the refactor commit (c2632b3) is the fork. The module's exports and the exact arguments each hook passes into it are not yet documented in prose; that is the target of the ADR action item. One open question the article cannot answer today: whether the layer_affinity filtering in suggest-skills.mjs is aware of shell's App → Instance → Thread context model, or whether it operates on a parallel taxonomy that shell does not yet know about. That relationship needs to be resolved before suggestions can be considered session-context-aware rather than just catalog-aware.

### Fleet-wide session-init.sh symlink tracked in git to survive clone and branch-switch — *tooling-for-iteration*

The session-init.sh hook is installed by core/scripts/install-agents.sh section 6 as a symlink inside each repo's .claude/hooks/ directory. Without tracking the symlink in git, a fresh clone or branch switch silently loses the hook — session initialisation fires once and is never seen again on any subsequent clone. Tracking the symlink target in git closes that silent failure mode. To confirm a working state on any machine: ls -la .claude/hooks/session-init.sh should show a symlink pointing to the core-managed script. A broken state is the file being absent or a dangling symlink. One open verification gap: the article records that all 10 repos received the symlink commit, but does not confirm whether install-agents.sh was re-run on each developer machine after the merge, or whether existing installed hooks are now in sync with what's in git. Fleet-wide changes of this kind need a verification step beyond merge count.

### JSONL truncation bug fix accepted with ADR-0037 — *tooling-for-iteration*

The hooks layer was writing JSONL log lines that could be truncated mid-write under concurrent hook invocations — the bug was a missing newline causing two records to merge into a single malformed line, undetectable without parsing. ADR-0037 formalises the fix: append-only, line-at-a-time writes with a newline sentinel. Its acceptance is a prerequisite for trusting any telemetry collected by suggest-skill, log-skill, or pr-skill-audit. All telemetry collected before Sunday's fix is suspect. Sunday's session-init invocations are the first data written under the corrected write path — ~/.claude/suggestion-log.jsonl entries from Sunday onward should be treated as the integrity boundary. Prior data is not retroactively repaired. ADR-0037 is accepted and lives in the core repo; a suggested action to write the companion ADR for suggested_after and layer_affinity is listed below, as those catalog-schema decisions do not yet have their own written record.
