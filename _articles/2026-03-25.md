## The decisions

### Why `file:` links require special CI treatment — and why we chose them anyway

When you write `"frame-ui-components": "file:../../frame-ui-components"` in a `package.json`, pnpm resolves it to a sibling directory on the local filesystem. On a developer machine, that directory always exists. On a fresh CI runner, the repo is cloned in isolation — the sibling directory does not exist unless you explicitly clone it first.

The fix is a `actions/checkout` step added before `pnpm install`, using `with: repository: ojfbot/frame-ui-components` and `with: path: ../frame-ui-components` to land the clone at the exact relative path the `file:` link expects. For private repos this requires a PAT or GitHub App token scoped to the `frame-ui-components` repo. The pattern from `c07f576` (shell) is the canonical reference; the `/scaffold-app` template (`e3c5ff8`) now emits this step automatically for new repos — though end-to-end verification of that template output is the first Wednesday action item (see below).

**Why not just publish to npm?** Publishing to a private registry adds a release step between "I changed a component" and "all apps see the change." During active extraction — Phase 1 — that ceremony slows down the feedback loop. `file:` links give monorepo ergonomics without monorepo tooling overhead. The tradeoff is exactly what we paid Tuesday: every CI pipeline needs an explicit clone step. Once the shared library stabilises and Phase 1 closes, the migration is a one-line change per repo (`file:` → `@ojfbot/frame-ui-components@x.y.z`) with no CI restructuring — the clone step becomes a no-op registry fetch.

**Migration trigger criteria** (to be formalised in ADR-0030 addendum): "surface stabilises" is not a decision gate. Three candidate signals: (1) all five shared components have Storybook stories and a passing visual regression baseline; (2) a first external consumer (outside the current seven apps) needs to adopt the library; (3) a breaking API change requires coordinated versioning across more than two repos simultaneously. Any one of these should trigger the `file:` → publish flip. This is a draft — the `/adr` action below is the place to harden it.

**Cost-of-delay note**: every new repo added to the fleet before Phase 1 closes inherits the two-fix `optimizeDeps` pattern and the CI clone step. The `/scaffold-app` automation propagates the pattern correctly, but it does not eliminate the underlying debt. There is a firm line between "right call for Phase 1" and "technical debt if Phase 1 drags." The Storybook gate (see Roadmap Pulse) is the forcing function.

### Why Vite `optimizeDeps` needs two changes, not one

Vite's pre-bundling scans entry points, finds imports, and pre-bundles CJS dependencies to ESM. The `file:` link breaks this in two distinct ways:

**1. Exclude the source package itself.** Vite tries to pre-bundle `frame-ui-components` as if it were a node_modules package. Because it's a source link, pre-bundling produces a stale snapshot that doesn't update on file changes.
```js
optimizeDeps: { exclude: ['frame-ui-components'] }
```

**2. Include CJS transitive deps explicitly.** When Vite skips pre-bundling the entry package, it also skips scanning its imports — so CJS dependencies of `frame-ui-components` (Carbon's CommonJS exports, and any other CJS packages it pulls in) are served as raw CJS inside a module graph Vite expects to be ESM.
```js
optimizeDeps: { include: ['@carbon/react', '@carbon/icons-react', ...] }
```
The full include list varies slightly per repo depending on which Carbon sub-packages each app uses — check the `vite.config.ts` diff in each PR for the exact list. The canonical starting point is the set committed in [BlogEngine] #37.

> **What breaks if you only do one?** `exclude` without `include` — stale-snapshot errors gone, runtime `require is not defined` errors appear in the browser. `include` without `exclude` — no runtime errors, but HMR stops working correctly because Vite's pre-bundled snapshot doesn't update when you edit a source file in `frame-ui-components`. Both fixes are required for every repo consuming the `file:` link.

**Re-entry checklist for the next repo onboarded to `frame-ui-components`:**
1. `optimizeDeps.exclude: ['frame-ui-components']` in `vite.config.ts`
2. `optimizeDeps.include: ['@carbon/react', ...]` in `vite.config.ts` (copy from nearest sibling repo, adjust for local Carbon sub-package usage)
3. `server.fs.allow` entry for the `file:` path in `vite.config.ts` (see [shell] #50)
4. CI clone step for `frame-ui-components` before `pnpm install` (see `c07f576`)
5. Remove any `branches: [main]` restrictions from workflow triggers

This pattern is now the `/scaffold-app` default. If you're wiring a repo manually, use this list.

This demonstrates the **tooling for fast iteration** pillar: the pattern is documented in ADR-0030, templated in `/scaffold-app`, and recorded in each repo's `CLAUDE.md`. The cost of the first adoption is the documentation cost of every future one.

### Branch restriction removal — the silent test-swallowing failure

GitHub Actions triggers with `branches: [main]` only fire on pushes to `main`. A PR targeting `main` from a feature branch does not match — the workflow is never queued. In the GitHub UI, this manifests as a PR with **no status checks at all** — not a pending check, not a failing check, an absent one. If branch protection is configured to require status checks, the merge button stays red with "Required status checks have not run." If branch protection is not configured, the merge button is green and the absence of CI is invisible. The latter is the dangerous case: "CI is passing" because no CI ran.

Tuesday's removal of `branches: [main]` across every repo is a correctness fix. The entire point of PR-gating is that checks run on the PR branch, not after it merges.

**The cv-builder asymmetry is worth naming explicitly**: browser tests *run* on all PRs (`23507b1`), but the visual regression *deploy* is restricted to `main` (`70b86d6`). This is intentional — you want the verification signal on every PR branch, but you don't want parallel preview deploys from forty open PRs overwriting each other's artifact storage. The distinction is between the *verification* step (branch-agnostic, always runs) and the *publication* step (main-only, produces a persistent artifact). The config looks asymmetric by design and should stay that way.
