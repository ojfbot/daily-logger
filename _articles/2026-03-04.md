## The decisions

### Decision 1: frame-agent as the CORS trust boundary for MrPlug

The naive implementation would be to allow `chrome-extension://` as a CORS origin wildcard. [shell] #16 shipped that way; [shell] #17 tightened it to a per-extension-ID allowlist.

**Why does this matter?** A CORS wildcard on `chrome-extension://` accepts requests from *any* installed Chrome extension — not just MrPlug. Since frame-agent proxies to the Anthropic API using server-side credentials, that wildcard lets any extension on the user's machine make authenticated LLM calls through your gateway. The per-extension-ID allowlist closes that surface to exactly one origin.

The tradeoff: Chrome assigns different extension IDs to dev and production builds. The allowlist is therefore environment-aware — both IDs must be enumerated in frame-agent's config. Where does that config live? In frame-agent's environment variables, not hardcoded in the route handler. If the extension is republished or re-signed, the allowlist ID must be updated in that env config before the gateway will accept requests. This is the operational path; missing it at 11pm is the failure mode.

This is **Pillar 4 — Security as emergent UX**. The restriction isn't an afterthought; it's the feature. Verifying that both IDs are correctly enumerated is a hard gate for Phase 5B.

---

### Decision 2: Two-store architecture for BlogEngine settings — shell Redux + localStorage

BlogEngine #25 doesn't put everything in shell Redux. API keys stay in localStorage. Every other preference goes into the shared Redux singleton.

**Why not put everything in Redux?** The mechanism matters here, not just the principle. In Module Federation, sub-apps import from a *shared singleton instance* of the shell's Redux store. That means any remote that imports the singleton can read any key in the store. Redux DevTools, error reporting, and state serialization (for hydration or logging) will also expose Redux state. API keys in serializable shared state are visible to every remote that touches the singleton — that's lateral privilege escalation across app boundaries, exactly what the isolation model is meant to prevent.

**What would break if you put API keys in Redux?** Any sub-app that imports the shared Redux singleton could read `state.blogengine.anthropicApiKey`. That's not a theoretical risk — it's the default behavior of the shared singleton pattern.

**localStorage doesn't have this problem** because it's scoped to the origin and doesn't traverse the MF boundary. The two-store split is a deliberate isolation model, not a compromise.

**Which sub-app are we least confident about?** cv-builder and TripPlanner shipped their settings panels the same day, before the ADR is written. The isolation rule hasn't been audited across those panels. This is the gap the ADR closes. The reference implementation to audit against is [BlogEngine] #25 — specifically the `apps/BlogEngine/src/settings/` directory and the `SettingsPanel` component, which is the cleanest expression of the two-store pattern in the codebase.

This is **Pillar 4 — Security as emergent UX** again: the architecture makes the secure path the default.

### Decision 3: `apiBaseUrl` as a runtime connection override in settingsSlice

Every sub-app's settings include `apiBaseUrl` — a field that overrides the backend URL at runtime from the settings modal, without a redeploy. The live connection status indicator reflects whether the *currently configured* `apiBaseUrl` is reachable, not just whether the default is up.

**Why does this matter for a multi-remote MF architecture?** Remotes are built and deployed independently. During development, cv-builder might be running at `:3001` while you're testing a staging backend at a different port. Without a runtime override, you'd need to rebuild the remote with the new URL baked in. The `apiBaseUrl` field makes the connection target a user-configurable value, not a compile-time constant.

The settings modal is not a preferences screen — it's the control plane. ShellAgent will eventually read and write these values to reconfigure sub-app connections via natural language. The Redux shape is already structured for that: per-app namespaced state means ShellAgent can dispatch `settings/updateCvBuilderSettings({ apiBaseUrl: '...' })` without touching other apps' state. This is architectural intent, not demonstrated behavior yet — no one outside the build has driven it via natural language. The groundwork is in the shape; the interaction is Phase 5B territory.

This is **Pillar 1 — Assistant-centric architecture**: the assistant orchestrates; the settings panel is the surface it controls.

### Decision 4: `frame-os-context.md` as the authoritative symlinked context file

`frame-os-context.md` lives in the `core` repo and is symlinked from every other repo's `decisions/` directory. One source of truth, consumed by every repo's Claude Code context. The daily-logger symlink incident is the failure mode — but the fix is a safety check in CI, not abandoning the pattern.

**The remaining honest gap:** "breaks loudly in CI" is only a complete answer if CI is the only place it can break. A developer who clones one repo without the sibling will hit a dangling symlink with no CI to catch it. The local developer path is: run `find decisions/ -xtype l` before pushing. That command is now in `pr-check` but is not yet documented as a local pre-push step. That gap should be noted in the symlink pattern's eventual ADR.

This is **Pillar 2 — Tooling for fast iteration**: the context file is a versioned artifact, and the tooling (symlinks + CI guard) keeps it authoritative across nine repos.
