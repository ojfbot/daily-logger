## The decisions

### Mint a SUGGESTION_ID to repair the suggestion-follow denominator (ADR-0093) — *model-behavior-as-design*

Every suggestion event now gets a UUID stamped at suggestion-creation time, flowing through the skill-telemetry layer so that follow/ignore classification can be attributed to the correct suggestion instance. Without this token, the denominator of the follow-rate metric was undefined: multiple suggestions in a session could be credited or debited to the wrong event, and the rate calculation would silently return zero — indistinguishable from genuine non-compliance. ADR-0093 is the keystone that makes the OPAV loop's Observe phase produce interpretable data. Until core #158 merges and a real session runs under the repaired instrumentation, the 0% reading remains unvalidated — the fix is shipped but not yet proven. On the acceptance criterion: one followed suggestion out of sixteen is not a pass. The target is interpretability first (non-null, non-zero denominator), then a baseline measurement that can anchor future ADR comparisons. What constitutes a healthy follow rate — 20%? 60%? — is a question the first post-fix measurement cycle is meant to open, not answer.

### Strike the false 0.8% baseline in ADR-0068 — *model-behavior-as-design*

The 0.8% figure in ADR-0068 was derived from denominator-broken telemetry — it was never a meaningful measurement. Rather than silently update the ADR, the decision was to explicitly strike the figure and record why it was wrong, preserving the audit trail. Keeping a false baseline in an ADR is worse than having no baseline: it anchors future comparisons to a fictitious number. A struck entry with a recorded reason is the honest form of revision; a silent overwrite removes the evidence that the error occurred.

### Deliverable-tracking event spine as a first-class observable (OPAV S1 foundation) — *tooling-for-iteration*

A structured event spine for deliverable-tracking was committed separately from the gate-event consumer (`4c7d384`), following the pattern of decoupling the event log from its consumers. This lets gate-event consumers evolve independently — core #158 (OPAV S1, skill-action instrumentation C0–C2, currently in shadow mode) can attach without modifying the spine itself. Co-locating event production with consumption would couple gate logic to the tracking schema and make future schema changes a breaking change for every consumer simultaneously.

### ADR-0088 rev B: canvas-fit gate enforcement for Obsidian canvas nodes — *security-as-emergent-ux*

The selfco vault's Obsidian canvas layer now enforces size-nodes-to-content as a hard gate in CI — the `canvas-fit.py` script runs as a CI step, and a failure blocks the push. Rev A shipped the enforcement script; rev B added the gate call and fixed the edge-label occlusion that rev A introduced. The `security-as-emergent-ux` tag is doing real conceptual work here: enforcing layout rules at the gate boundary means a canvas that violates sizing conventions cannot reach the vault. The security model is not about confidentiality — it's about the vault's structural integrity being an invariant that the agent cannot accidentally break. Shipping the fix as two labelled revisions keeps the ADR history readable; reviewers can see exactly what changed between passes rather than having a single commit silently overwrite prior behaviour. Rev A and rev B in a single day is a flag worth naming: the gate was incomplete when first recorded, and the revision exists because the initial enforcement missed the edge-label interaction. That's acceptable cadence when the revision is labelled and traceable; it would not be acceptable as a silent update.

Tags: `vault`, `adr`, `security-as-emergent-ux`

### Canonicalise selfco-ingest in core with tag-proposal discipline — *assistant-centric*

The selfco-ingest skill previously existed in multiple forms across repos. `selfco-ingest` is the skill the agent uses to process incoming captures from Notion, iOS, and MCP into the personal Obsidian vault via the `selfco-box` daemon — it's a load-bearing ingest path. Establishing the canonical in `core` with explicit tag-proposal discipline means every downstream session (any repo running a session-init that pulls from core's skill baseline) gets the same ingest behaviour. The ADR-0073 reconcile step — which removes the draft gate from selfco ingest — was bundled into the same commit to avoid a window where the canonical skill and the ADR governing it were inconsistent.

Tags: `skill-catalog`, `assistant-centric`

### Make suggest-skill.sh tests CI-portable across BSD and GNU date — *tooling-for-iteration*

The test suite was using GNU `date` syntax that fails silently on macOS (BSD `date`). If CI runs on Linux but local dev runs on macOS, a broken test that passes locally and fails in CI produces a false green signal on the branch that matters least. The fix normalises date handling to a portable subset, making the suggest-skill CI gate trustworthy on both surfaces. This matters most now because validating the post-fix OPAV telemetry requires a trustworthy CI run — a portability failure in the test harness would contaminate the first real measurement cycle.
