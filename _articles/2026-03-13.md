> **Why does the cleaner run on a confidence threshold?**
> The daily-cleaner pipeline doesn't delete every comment — it runs each candidate edit through Claude Opus with a confidence classification (high / medium / low) and drops anything below medium. The cost asymmetry is intentional: a false positive (deleting a comment that was load-bearing documentation) is expensive and irreversible relative to the cost of leaving one stale comment until tomorrow. This is model behavior as design discipline — the prompt isn't "delete all comments," it's "classify and filter." The threshold isn't a safety net; it's the design.
>
> *What the threshold is in practice:* "medium confidence" is a qualitative classification baked into the prompt instruction set, not a numeric score. The model is asked to label each removal candidate as high, medium, or low confidence based on whether the comment's removal would change reader understanding of the surrounding code. Anything labeled low is left untouched. This is replicable: the same prompt + the same classifier instruction = the same threshold behavior across runs.

### Decision 1: Cross-domain fan-out lands in `MetaOrchestratorAgent`, not in individual app agents — [shell] #32

[shell] #32 implements the fan-out path that makes the hero demo query work end-to-end:

> *"I'm applying for Berlin jobs — how does this affect my trip plan and what should I put on my resume?"*

This is the specific query the entire phase has been building toward. It is not yet running against the live system — [shell] #32 is still in review. Once it merges, the fan-out path is code-complete. Whether the demo runs end-to-end after merge is the first validation task.

**Where the logic lives in the file tree:** `MetaOrchestratorAgent` lives in `frame-agent` within the shell package. The entry point is the agent's `handle()` method, which receives the parsed query, dispatches to registered sub-app tool endpoints at `/api/tools`, collects domain responses, and synthesizes a unified reply. The call sequence is: shell receives query → MetaOrchestratorAgent reads capability manifest → calls `POST /api/tools` on each registered sub-app → collects responses → synthesizes. `POST /api/resumption` ([shell] #33, merged Wednesday) is the synthesis endpoint that produces the contextual opener from conversation history; [shell] #32 is the fan-out layer upstream of it.

**The architectural choice:** Option A — each sub-app agent handles its own slice; the shell stitches results. Option B — `MetaOrchestratorAgent` in frame-agent fans out to sub-app tool endpoints, collects domain responses, and synthesizes a unified reply.

We chose Option B. The reason: sub-apps don't have visibility into each other's state. cv-builder doesn't know TripPlanner has a Berlin trip in progress. The synthesis step — "here's how these two contexts interact" — requires a layer that can hold both simultaneously. Putting fan-out in MetaOrchestratorAgent keeps sub-apps dumb and composable.

> **What would break if fan-out lived in individual app agents?**
> Each sub-app would need a reference to every other sub-app's API — N² coupling. Add a fourth app and you'd need to update three existing apps to know about it. The MetaOrchestratorAgent pattern means you add one entry to the capability manifest and the orchestrator discovers it dynamically. "Add a fourth app and you'd need to update three existing apps" is the sentence that makes this decision stick.

**The god-object risk and its actual mitigation:** Naming the risk is not mitigating it. ADR-0007's capability manifest contract is a *structural* constraint — sub-apps declare tool signatures at `/api/tools` and the orchestrator calls those endpoints, so it can't know a sub-app's internals. But a structural constraint doesn't prevent semantic accumulation: nothing in ADR-0007 stops someone from routing "notify the user" or "write to shared cache" through MetaOrchestratorAgent six months from now. The honest mitigation today is the PR review checklist item for [shell] #32: confirm no hardcoded sub-app routes, confirm capability discovery is dynamic, and confirm the orchestrator's concerns are limited to fan-out and synthesis. That review step is the enforcement mechanism until a lint rule or CI check exists. The ADR is the record; the review is the gate.

**Cost note:** The Berlin query fans out to at least two sub-app tool endpoints, each of which may invoke a model call. At this scale (single developer, demo traffic) that's not a budgetary concern. At 10 concurrent users running hero demo queries, each fan-out generating 2–3 LLM calls, the inference bill becomes meaningful. That cost analysis is not in scope for Phase 1, but it needs to be named in the Phase 2 planning doc before MetaOrchestratorAgent is opened to real traffic.

This demonstrates the **assistant-centric architecture** pillar: the ShellAgent (here, MetaOrchestratorAgent) is the organizing primitive. Sub-apps are tools it calls, not peers it coordinates with.

### Decision 2: `HeaderConnected` + `HomeScreenConnected` as the boundary proof — [shell] #31

[shell] #31 replaces local component usage in `shell-app` with connected wrappers that import from `@ojfbot/shell`. `HeaderConnected` resolves Redux state and env vars; `HomeScreenConnected` maps the app registry to tile props. Both re-export pure components from `packages/ui`.

**On naming:** `packages/ui` is the source directory; `@ojfbot/shell` is the published package name. These are the same package. The article uses both names — `packages/ui` when referring to file paths, `@ojfbot/shell` when referring to the import contract. This is a distinction that matters when returning to the codebase cold.

**The connected/pure split:** `Header.tsx` and `HomeScreen.tsx` in `packages/ui` are props-driven with zero Redux imports. The connected wrappers live in `shell-app` and own the state resolution. This means the design system package remains portable — importable into Storybook, a test harness, or a different host without Redux as a peer dependency.

> **What does the connected/pure split mean in practice?**
> If `Header.tsx` imported from Redux directly, every consumer of `@ojfbot/shell` would need to configure a compatible Redux store — a steep adoption cost for a UI primitive. The split means Storybook stories can render `Header` with plain props and no store setup. The connected wrapper is the app-specific adapter, not the component itself.

**This decision needs a durable artifact.** The connected/pure split is documented here as prose explanation, and there's a `/adr` suggested action below. That action is not a nice-to-have. If someone inherits this codebase and doesn't find this decision recorded in an ADR, they'll undo it inside a month. The `/adr` for the connected/pure split should include: the decision, the rejected alternative (Redux imports in the component), the portability consequence, and the Storybook example as the concrete test case.

**The [shell] #5 dependency question:** [shell] #31 adds connected wrappers that resolve Redux state. [shell] #5 (frame-wide Redux store strategy) is still open. The durable fix is a CI lint rule that fails the build if a Redux import is detected in `packages/ui`. That rule should be the first follow-on commit after #31 merges.
