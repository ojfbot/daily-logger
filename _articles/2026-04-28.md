## The decisions

### Five atomic milestone commits as the vertical-slice architecture — *tooling-for-iteration*

beaverGame #2 was structured as five single-commit milestones (M-α through M-ε), each independently snap-verifiable, rather than one large feature branch. This makes bisect trivial, gives the AI assistant a clear checkpoint to resume from via /bead, and enforces that each game mechanic is genuinely self-contained before the next one depends on it. The M-α through M-ε labels map directly to the five commits listed under beaverGame above — each commit is a complete, runnable state of the game, not a partial implementation.

Tags: `assistant-centric`, `ci-cd`, `sprint`

### asset-foundry as a separate repo rather than a beaverGame subdirectory — *tooling-for-iteration*

The Blender asset pipeline lives in its own repo with its own CI, manifest schema, and issue tracker. The contract between the two repos is a file-copy sync: running pnpm gen-asset <name> in asset-foundry produces a .glb and a .validation.json that land in beaverGame/public/assets/ when both repos are checked out as siblings. File-copy was chosen over a package dependency because assets are binary build artifacts, not versioned library code — a package dependency would require a publish step for every geometry change and would make local iteration slow. A monorepo internal import was rejected because it would couple the game client's build graph to the foundry's Blender environment. The tradeoff is that the sync path has never been verified end-to-end against a real beaverGame checkout — AF-013 is filed and unstarted. The /validate action below is the manual precursor to that CI gate.

Tags: `ci-cd`, `module-federation`

### Water-as-highway as a load-bearing game mechanic, not a visual flourish — *model-behavior-as-design*

The M-γ hauling milestone bakes in land-vs-water speed modifiers from day one, before any water shader exists. The current values are a 1.0x base walk speed on land and 1.8x on water surface (carrying a log reduces both by 0.6x). These numbers are tunable but the asymmetry is locked: the water shader spike (BG-007) must produce a surface the player can navigate faster, otherwise the Mode B flooding arc has no mechanical payoff. Committing to the speed asymmetry now prevents the shader from being designed purely for aesthetics — that's the trap this decision explicitly forecloses.

### First-class ecosystem citizenship from commit one: /bead + install-agents + repo registry — *assistant-centric*

Both new repos were registered in core's architecture docs, install-agents.sh mapping, and repo tables before the first feature PR merged. The /bead skill was added to core on the same day. When a new Claude Code session opens in any registered repo, install-agents.sh injects the most recent bead alongside the architecture overview and skill catalog. This means the assistant resumes with full context rather than re-reading the codebase cold.
