## The decisions

### Separate CVE for MrPlug: named vulnerability requires explicit version pin, not just lock file refresh — *security-as-emergent-ux*

All other repos needed a lock file rebuild to pull in already-patched transitive deps — their existing semver ranges included the patched versions, so a rebuild was sufficient. MrPlug was different: `@langchain/core` had a named CVE (CVE-2025-68665, High severity, serialization injection) and the existing semver range in `package.json` did *not* guarantee resolution to 0.3.80. The fix was an explicit `resolutions` entry in `package.json` pinning `@langchain/core` to `0.3.80`, not just a lock file rebuild. A generic rebuild might or might not have landed on the patched version depending on how pnpm resolved within the existing range; the explicit pin guarantees it regardless of resolution strategy.

MrPlug uses `@langchain/core` as the LangChain integration layer for ShellAgent tool-use orchestration — it handles serialized tool call payloads. CVE-2025-68665 is a serialization injection vulnerability, meaning a maliciously crafted payload could have escaped the expected deserialization boundary. In a public repo where the tool-use surface is the product, that's not theoretical exposure.

This pin is intentional and should not be removed without verifying that the upstream package has published a clean release that supersedes 0.3.80.

### Jekyll removed entirely; Vercel serverless is now the only deployment target for daily-logger — *tooling-for-iteration*

The Jekyll/GitHub Pages setup had been coexisting with the Vercel deployment in an ambiguous state — two deployment models, one repo, no clear canonical. This wasn't the original plan; Jekyll was drift from an earlier phase before the serverless revision pipeline existed. The risk was operational: Vercel's routing resolves static build output before function routes, so a static file at the same path as a serverless function causes the function to silently 404 or return HTML instead of JSON. `4ac70e0` addresses this directly by restructuring the catch-all route as a single function with no static file at a conflicting path. The rule going forward: never place a static output file at a path that matches a `api/` function route in the same Vercel project.

Vercel is the canonical deployment target for daily-logger specifically. Whether it becomes the fleet-wide standard for all Frame OS apps is an open architectural question — core [#15] (RFC: Frame × Gas Town adoption roadmap) is the right place for that decision, and it isn't resolved yet. GitHub Pages was free; Vercel's hobby tier covers the current daily-logger traffic with no cost delta at present scale. The operational gain — a single deployment model, Vercel preview URLs for draft articles, serverless revision pipeline — justified the migration over a static host that couldn't run functions.

### Editorial revision CI: `pnpm exec tsx` over bare `tsx` invocation — *tooling-for-iteration*

The editorial revision workflow was calling `tsx` directly, which fails in a CI environment where `tsx` is a project-local devDependency rather than a global binary. Using `pnpm exec tsx` resolves the binary through the project's own `node_modules/.bin`, making the workflow portable across any runner without requiring a global install step.

This pattern is now documented in `CLAUDE.md` for daily-logger. It is not yet enforced by a lint rule — if a future workflow file invokes a local binary directly, it will fail silently in CI rather than at write time. A shellcheck or custom ESLint rule for workflow YAML would close that gap; filed as a known gap, not yet scheduled.

### Fleet-wide lock file hardening as a single coordinated pass — *security-as-emergent-ux*

Rather than patching repos one-by-one as Dependabot surfaces them, Tuesday's pass rebuilt lock files across all 11 affected repos simultaneously. The triggering mechanism was a manual audit prompted by CVE-2025-68665 surfacing in MrPlug — once that CVE was identified, the same transitive packages (`braces`, `cross-spawn`, `nanoid`, `form-data`, `lodash`, `minimatch`, `serialize-javascript`) were checked across the full repo inventory.

Honestly: this was 11 sequential PRs, not an atomic deployment. There was a window — roughly the duration of the merge queue — during which some repos were patched and others were not. For internal tooling, that window is acceptable. If this fleet included customer-facing APIs, the answer would be a monorepo or a coordinated deploy pipeline. The 'repeatable pattern' claim rests on the uniform pnpm workspace structure making the audit and rebuild scriptable; the atomicity is aspirational, not current.

A scheduled `pnpm audit --audit-level=high` across all repos would make future triggering proactive rather than reactive. Not yet wired.

### MetricsBar extracted as a standalone component with hover popovers — *tooling-for-iteration*

MetricsBar was pulled out of its inline context in the daily-logger frontend and given its own file with five stat cards, each with a hover popover. The extraction follows the same component discipline being applied fleet-wide via `@ojfbot/frame-ui-components` — the component boundary makes the stat definitions and popover content independently testable and prevents the bar from accumulating inline logic that belongs to the page layer.

This is internal tooling; there is no Figma spec for MetricsBar and visual design is intentionally deferred at this stage. The component is built from the existing token system. Whether MetricsBar eventually moves into the shared `@ojfbot/frame-ui-components` library is unresolved — it's daily-logger-specific enough that promotion isn't obviously right yet. If a second consumer appears, that's the trigger to promote it. The `ci-cd` tag on this decision in the original draft was incorrect; this is a frontend architecture decision.
