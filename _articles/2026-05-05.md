Monday was one of the highest-volume days in the project's history: 88 commits across 15 repos, 37 PRs merged, and 10 issues closed. Three distinct stories ran in parallel — a new shared CI infrastructure repo went from zero to fleet-wide adoption in a single day, beaverGame completed its Three.js retirement and shipped a full Babylon gameplay loop with dodge mechanics, and asset-foundry formalised dual Blender transports behind a stable internal contract. Each story compounds the others: the CI infrastructure makes the gameplay and asset work auditable; the asset pipeline feeds the gameplay; the gameplay validates the asset contract.

## What shipped

### github-actions ([#https://github.com/ojfbot/github-actions/pull/1](https://github.com/ojfbot/github-actions/pull/https://github.com/ojfbot/github-actions/pull/1), [#https://github.com/ojfbot/github-actions/pull/2](https://github.com/ojfbot/github-actions/pull/https://github.com/ojfbot/github-actions/pull/2), [#https://github.com/ojfbot/github-actions/pull/3](https://github.com/ojfbot/github-actions/pull/https://github.com/ojfbot/github-actions/pull/3))

A brand-new shared CI repo (`ojfbot/github-actions`) shipped from initial commit to adoption across 6 of the fleet's active repos in one day. It exports a composite action at `ojfbot/github-actions/.github/actions/skill-audit/action.yml`, referenced by consumers as `uses: ojfbot/github-actions/.github/actions/skill-audit@v1`. Two immediate bug-fixes landed in the same session: single-commit PRs were producing a zero-width audit window — `local_audit()` used identical first/last timestamps, which meant those PRs returned empty audit results silently rather than erroring — and `/investigate` invocations made via comment markers weren't being credited. Both fixes are now at the source; every consumer repo inherits them automatically on the next run. The count of PRs silently mis-audited before the fix is not currently known — historical audit data for single-commit PRs should be treated as potentially incomplete. ADR-0067, which will codify the rationale for this extraction, is not yet written. The article's Decisions section describes the intent; the ADR document itself is a pending task (see Suggested Actions).

## The decisions

### Extract skill-audit CI into a shared composite action (ADR-0067 — pending) — *tooling-for-iteration*

Rather than maintaining 100 lines of duplicated YAML across every repo, we extracted the skill-audit workflow into a composite action defined at `.github/actions/skill-audit/action.yml` in `ojfbot/github-actions`, referenced by consumers as `uses: ojfbot/github-actions/.github/actions/skill-audit@v1`. Six repos adopted it on day one: core, landing, cv-builder, core-reader, seh-study, and beaverGame. The remainder of the active fleet has not yet migrated — 'fleet-wide' in the lede is shorthand for the six repos that adopted Monday; it is not a claim of 100% coverage. All bug-fixes propagate automatically to consumers. One important caveat: ADR-0067 is cited here and in multiple PRs, but the document does not yet exist. This section describes the decision as made and implemented; the written rationale record is a pending task. To test a change to the composite action without rolling it to all consumers, push to a branch and reference it as `ojfbot/github-actions/.github/actions/skill-audit@your-branch` in one consumer repo before updating @v1.

Tags: `ci-cd`, `tooling-for-iteration`

### Dual Blender transports behind a stable contract (ADR-0011) — *model-behavior-as-design*

asset-foundry now exposes two peer transports — subprocess (isolated, always available) and kernel-MCP (more capable, requires a running Blender instance) — both behind the same `runBlenderScript(opts: RunOptions): Promise<RunResult>` signature. The transport selection is currently determined by whether a Blender kernel is active; there is no explicit health-check API exposed to callers — if you are debugging a case where the wrong transport is active, check whether a Blender kernel process is running at the expected socket. The public API is unchanged regardless of which transport is active. The `module-federation` tag on this ADR refers to asset-foundry's registration in the shell MF host, not the transport design itself.

Tags: `assistant-centric`, `model-behavior-as-design`

### Retire Three.js in favour of Babylon.js for beaverGame — *tooling-for-iteration*

The legacy Three.js client is gone. Babylon.js now owns all gameplay surface because it provides web-native physics, built-in collision, and a managed render loop that Three.js required manual scaffolding for. The cost was a full port sprint; the benefit is that every future gameplay mechanic — including the tree-fall dodge loop that shipped Monday — can use Babylon's documented, typed subsystems rather than hand-rolled equivalents.

Tags: `sprint`, `tooling-for-iteration`

### Codify 'agents must follow skill suggestions' as a durable rule (ADR-0068) — *assistant-centric*

The 0% suggestion-follow rate in telemetry is now an architectural violation, not just a metric. ADR-0068 makes agent compliance with skill suggestions a first-class rule in the `core` catalog, meaning future sessions can be audited against it. The timing here is not ironic — it is causal: the rule exists because the data demanded it. One important upstream question remains open: the 46 suggestions that went unfollowed may represent a behavior gap, a tooling/timing gap (suggestions surfaced at moments in the session flow where acting on them was impractical), or a measurement artifact (the telemetry model may be crediting PR-comment markers as real-time invocations, which would mean the compliance rule is being constructed on top of synthetic data). The /investigate action item below is the prerequisite for knowing which problem ADR-0068 is actually solving.

Tags: `assistant-centric`, `ci-cd`

The credits on PRs came from PR-comment markers written at review time, not from real-time session invocations. A PR-comment marker is an annotation posted to a PR (by a human or agent) that the skill-audit CI step reads and counts as a skill invocation for that PR. This is distinct from the agent invoking the skill live during an edit session. If the credits are predominantly marker-based rather than session-based, the audit is measuring annotation coverage, not behavioral coverage — and ADR-0068 is a governance rule built on top of a measurement that may not reflect what it purports to measure.
