## The decisions

### session-init.sh symlink tracked in version control across all repos — *tooling-for-iteration*

session-init.sh is a shell hook that runs on session start to register the repo with Frame's shell — setting up environment bindings and ensuring skill resolution works correctly in that working context. Without it tracked in version control, the symlink exists on a developer's machine after running install-agents.sh but disappears after a branch switch that predates the hook, or is absent entirely on fresh clones and CI agents. The result is a silent failure: hook invocations that work locally simply don't fire in CI. Tracking the symlink in each repo eliminates that class of failure permanently. For anyone who cloned before this change: run install-agents.sh again from the repo root — it is idempotent and will reconcile the local symlink state with what's now committed. The fleet sweep closed the immediate correctness gap. Core issue #77 (migrate to @ojfbot/core-skills npm package) is the logical next step: symlinks have no semver, no changelog, and no rollback story. The evidence for moving to npm is now concrete — seven repos just required a manual sweep to fix a problem a versioned package would have prevented by design. The symlink approach is sufficient while the skill catalog is still stabilizing; the trigger condition for moving to npm is when the catalog crosses teams or when the first rollback is needed. That's a go/no-go criterion, not an investigation.

### daily-cleaner uses Claude Opus with confidence thresholds — no human review required for doc cleanup PRs — *model-behavior-as-design*

The stale docs/comments PRs merged across purefoy, core-reader, lean-canvas, TripPlanner, seh-study, and BlogEngine this week were generated by the daily-cleaner pipeline and validated by Claude Opus under ADR-0033. The ADR establishes that high and medium confidence edits are merge-safe without a human reviewer; low-confidence edits are rejected by the pipeline and never open a PR. The threshold values and calibration methodology live in ADR-0033 — the short version is that 'low confidence' means Opus flagged ambiguity about whether the removed content was stale or load-bearing. The design decision is to treat the threshold filter as the quality gate, not a human approval step. The accepted failure mode is a false positive: a medium-confidence edit that removes a comment that was actually meaningful. The cost of that failure is a minor documentation error, not a runtime failure — and it's recoverable from git history. What this pipeline does *not* catch: cases where Opus is confidently wrong about something that has no runtime effect but carries real contextual value (e.g., a comment explaining a non-obvious business rule). That's a known gap, not a solved problem. The threshold filter reduces but does not eliminate it.

### purefoy ProgressReporter: structured JSON at pipeline stage boundaries, not log lines — *tooling-for-iteration*

ProgressReporter writes a progress.json file at each of 8 pipeline stages (ingest, segment, transcribe, align, QA, export, index, notify) rather than emitting log lines. A representative mid-run snapshot looks like: { stage: 'transcribe', segment: 14, total_segments: 60, gpu_util_pct: 87, elapsed_s: 412, eta_s: 1340 }. The choice of structured JSON over log output means any observer — a monitoring UI, a Frame sub-app, a polling script — can parse progress without log parsing. GPU metrics and segment heartbeats are co-located in the same file, making pipeline state observable without tailing logs. The preflight() function gates job start by validating IAM permissions, instance type, S3 bucket existence, and GPU availability — this exists primarily for cost control: an unchecked GPU run on AWS that fails at segment 2 of 60 wastes real money. The cost angle is the most important sentence in this section: preflight() is as much a billing guard as an observability feature. The integration path into Frame is not yet designed — polling the JSON file is the simplest option, but an event bus or webhook variant is possible. 'Eventually' is honest here; the ProgressReporter is a prerequisite for that integration, not the integration itself. It is not on Gas Town Sprint 1.
