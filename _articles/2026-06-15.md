### core

The C0–C2 shadow implementation of ADR-0095 shipped: the `skill:acted` event is now emitted on every confirmed skill invocation, but operates in shadow mode — it writes to the shadow telemetry log without blocking, alerting, or gating anything. This is the first time the instrumentation layer has live signal rather than synthetic test data. C0–C2 are the first three confidence tiers in the instrumentation rollout taxonomy (defined in ADR-0095): C0 is raw emission with no validation, C1 adds the two-source cross-check, C2 validates agreement rate against a minimum threshold. C3 — not yet active — is soft enforcement: the signal begins to surface warnings and can gate merges. Advancing from C2 to C3 requires a sustained two-source agreement rate above the threshold defined in ADR-0095 across a statistically meaningful session window; that criterion is not yet quantified in the accepted ADR text, which is an acknowledged gap. The `skill:acted` emission lives in the skill-dispatch layer; the confirming hook is a middleware intercept that sits between the agent self-report and the ledger write, independently reading the invocation event from the runtime before the agent's self-report is recorded. When the two sources disagree, the discrepancy is logged to the shadow log but no enforcement action is taken at C0–C2. To verify the shadow log is receiving events locally: `tail -f .shadow/skill-acted.log` from the core repo root. There is not yet a dashboard; the current inspection surface is that log file and the `/investigate` command described in the suggested actions.

## The decisions

### ADR accept before implementation: why the order matters — *tooling-for-iteration*

ADR-0094 and ADR-0095 were formally accepted in commit `4f5102f` before the C0–C2 implementation landed in `4db42aa`. Shipping code against a Proposed ADR creates an implicit contract that cannot be audited — if the ADR later changes direction, the implementation's ancestry is ambiguous. Accepting first establishes a clean causal record: the code exists *because* of a deliberate decision, not alongside one. In practice, this means `git log --follow` on any implementation file will surface the ADR commit in its ancestry; a reviewer can confirm the ADR was accepted before the first implementation commit by comparing timestamps. Squashing the two would erase that causal boundary. This is the kind of sentence that ends a recurring debate: if you find code whose ADR is still in Proposed state, the code predates the decision and its intent cannot be reconstructed from the record alone.

### Shadow mode as the safe rollout shape for instrumentation — *model-behavior-as-design*

The `skill:acted` signal in `4db42aa` is tagged C0–C2 SHADOW — it emits telemetry events but does not gate sessions, block merges, or surface warnings. This matters because instrumentation that immediately enforces tends to produce false positives that erode trust in the signal before the signal has been validated. Shadow mode lets real usage data accumulate so the two-source agreement rate can be measured against the C2 threshold before enforcement is layered on. ADR-0095's two-source design — the agent self-reports, and a middleware hook independently confirms by reading the invocation event before the self-report is recorded — is what makes eventual enforcement honest rather than just authoritative. If the agent self-reports a `skill:acted` event and the hook does not see a corresponding invocation, that is a discrepancy. If the hook sees an invocation the agent did not self-report, that is also a discrepancy. Both are logged in shadow mode. The C3 advancement criterion (minimum agreement rate over a minimum session window) is the open quantification gap in ADR-0095 — it must be resolved before shadow mode can have an exit condition rather than becoming a permanent state.
