## The decisions

### Publish @ojfbot/frame-ui-components to npm instead of relying on file: paths — *tooling-for-iteration*

The `file:` protocol requires every consuming repo's CI to clone or symlink the source repo before type-checking can begin — that's a hidden dependency graph baked into CI scripts rather than package.json. Publishing to npm at `^1.0.1` makes the dependency explicit, lockfile-pinned, and resolvable without repo-local knowledge. The tradeoff is that local changes to the package now require a publish step before they're visible to consumers, which we mitigate with a Vite alias that resolves from source during development.

Tags: `frame-ui-components`, `ci-cd`, `module-federation`, `npm`

### Vite alias as the bridge between npm resolution and local source development — *tooling-for-iteration*

When `@ojfbot/frame-ui-components` is an npm package, `vite` resolves it from `node_modules` — which is correct for CI but wrong during active development of the package itself. Adding a `resolve.alias` entry in `vite.config.ts` that points the package name to the local source path lets developers iterate on the shared package without publish cycles, while CI always uses the pinned npm version. Without this alias, any change to `frame-ui-components` during development would require either a local `npm pack` workflow or a `file:` reference — both of which re-introduce the problem we just solved.

Tags: `frame-ui-components`, `ci-cd`, `phase-1`

### Container-presenter decomposition as a fleet-wide practice, not a one-off refactor — *model-behavior-as-design*

The decomposition sweep across TripPlanner (`ItinerariesLibrary` 430→87 lines, `ImportAgentModal` 586→115 lines), BlogEngine (`PodcastResponder` 288→67 lines, `WorkingMemoryDashboard` 479→95 lines), and cv-builder (`BioDashboard` 833→373 lines) follows a consistent container-presenter split: extract state and side-effects into a custom hook, render logic into focused presenter components, and let the orchestrator file become a thin composition layer. The pattern matters beyond aesthetics — when a component is under 100 lines, the full component contract fits in one screen, making it far easier for Claude Code to propose targeted changes without hallucinating context that's off-screen.

Tags: `container-presenter`, `phase-1`, `cv-builder`, `BlogEngine`

### Awaiting watcher.close() in bead-store teardown to prevent test race conditions — *tooling-for-iteration*

`chokidar`'s `watcher.close()` is async — it returns a Promise that resolves once the OS inode watches are released. Discarding that Promise meant `afterEach` could delete the temp directory before chokidar finished, producing intermittent ENOENT errors that only appeared under load. The fix is a one-line `await` but the lesson is a general one: any cleanup function that touches the filesystem or OS resources should be treated as async until proven otherwise.

Tags: `ci-cd`, `core`

### Runtime type guard on Chrome extension onMessage listener instead of a cast — *security-as-emergent-ux*

The `chrome.runtime.onMessage` callback receives `unknown` at runtime — the Chrome types expose this as `any`, which makes it easy to accidentally narrow to a struct type without validation. Replacing `{ type: string }` with `unknown` and adding a type guard makes the actual runtime contract explicit and eliminates the 10 downstream type errors that were using the narrowed type as fact. This is the same pattern that prevents extension content-script bugs from surfacing silently in production.

Tags: `MrPlug`, `ci-cd`
