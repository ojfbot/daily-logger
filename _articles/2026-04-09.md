## The decisions

### Wait for shell render, not networkidle — *tooling-for-iteration*

Playwright's `networkidle` event fires when there are no pending network requests for 500ms — but Module Federation remotes load asynchronously *after* the host shell mounts. The shell mounts, renders its layout, then kicks off a fetch for each remote entry file; the remote component mounts only after that fetch completes. This means `networkidle` consistently fires during the gap between shell mount and remote mount. The fix in PR #15 waits for the `[data-frame-shell]` data attribute on the root layout container (rendered by `shell/src/ShellLayout.tsx`) instead. The wait logic is in `landing/e2e/screenshots.spec.ts`. The rejected alternative — increasing a static timeout — is fragile and environment-dependent; CI runners and local machines resolve Federation remotes at different speeds. This pattern will matter again when we add more remotes: every new Federation app extends the gap between `networkidle` and 'visually complete'. The `[data-frame-shell]` selector is coupled to the shell layout component — if that component is refactored in Phase 1, the screenshot spec breaks. That coupling is intentional and greppable; it's preferable to a timeout that silently becomes insufficient.

### Let packageManager field drive CI pnpm version — *tooling-for-iteration*

MrPlug's CI broke because `pnpm/action-setup` was hardcoded to `version: 9` while the repo's `package.json` declared `packageManager: pnpm@8.15.0`. Pnpm 9 cannot parse a v6 lockfile, so `pnpm install --frozen-lockfile` failed. The fix removes the explicit version from the GitHub Action YAML and lets it read `packageManager` from `package.json` — a single source of truth. This is the correct pattern fleet-wide: any repo that pins a pnpm version in CI risks this same drift when the lockfile format changes between major versions. The `/sweep` action below targets all remaining repos.

### Bot commits carry [skip ci] by convention — *tooling-for-iteration*

All four bot commits (article generation, editorial revision, stale comment cleanup) include `[skip ci]` in the commit message. Bot-generated changes are validated before commit via a Claude Opus structured diff check: Opus receives the proposed diff, confirms only comments/docs are touched, and returns a confidence score. If below threshold, the PR is flagged for human review. Running CI again on an already-validated change is redundant compute. The tradeoff: a bad bot commit won't be caught by CI. The mitigation: every bot PR goes through a human accept step before merge. This is a confidence heuristic plus human gate, not an automated test suite — the distinction matters, and we're honest about it.
