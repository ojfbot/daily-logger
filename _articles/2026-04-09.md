PR [#15](https://github.com/ojfbot/landing/pull/15) fixed preview screenshot capture so Playwright waits for the Frame shell to fully compose with Module Federation remotes before capturing — rather than firing on `networkidle`, which resolves before dynamic MF `import()` calls complete. The fix uses a DOM sentinel: a `data-mf-mounted` attribute (or equivalent stable marker) that each remote component renders once it has mounted inside the shell. Playwright's `waitForSelector` blocks on that attribute before triggering capture, ensuring the screenshot reflects the composed shell view a real user sees — nav chrome, shell wrapper, and all. The sentinel is the contract: any sub-app added to the screenshot pipeline must render it after mount. If a remote mounts but forgets the sentinel, Playwright will time out rather than silently capture the wrong state, making regressions loud instead of invisible. ## The decisions

### Wait for a DOM sentinel before capturing screenshots — not networkidle — *tooling-for-iteration*

The Playwright screenshot script was triggering on `networkidle`, which resolves as soon as the shell host's own network activity settles — before Module Federation's dynamic `import()` calls for remote bundles complete. The fix waits for a DOM sentinel: a stable attribute each remote component renders after it mounts inside the shell. Playwright's `waitForSelector` blocks on that attribute before capture fires. This approach does couple screenshot tooling to component implementation — every sub-app must honor the sentinel contract. The alternative, polling MF's internal container registry via `waitForFunction`, was considered but rejected: it reaches into MF internals that are subject to change across rspack/webpack versions and are harder to reason about in CI logs. The sentinel failure mode is explicit: if a remote forgets the attribute, Playwright times out loudly rather than silently capturing the wrong state. If the screenshots regress, start with the Playwright script in `landing/scripts/capture.ts` and verify each remote is rendering the sentinel after mount.

### Portfolio screenshots as a correctness signal, not aesthetics — *assistant-centric*

Showing apps inside the Frame shell is not a cosmetic preference — it is the entire thesis of the portfolio. A screenshot of cv-builder running standalone tells an interviewer nothing about Frame OS. A screenshot of cv-builder mounted inside the shell's navigation chrome is the argument. Getting this wrong silently was the worst kind of bug: it looked fine until you knew what you were looking for. The screenshot correctness signal matters because the shell is visually distinct — it renders a consistent nav bar and app-chrome wrapper around every remote — and that visual distinction is what makes the portfolio legible as a demonstration of an orchestration thesis, not just a collection of apps. The DOM sentinel fix is what ensures that signal is accurate on every deploy. Note: a decision record (ADR) for this wait strategy is not yet written. Given that it is the correctness signal for the entire portfolio CI pipeline, one should be.
