## The decisions

### ADR-0029: the prop-only boundary and why it matters for Module Federation

The most important architectural decision this week wasn't a feature — it was a constraint. ADR-0029 states: **every component in `packages/ui/` is a pure presenter. It accepts props. It does not import Redux hooks. It does not import the store.**

> **Why this constraint?**
> In a Module Federation architecture, the shell and each sub-app run in separate JavaScript module scopes. If a shared UI component in `@ojfbot/shell` imports `useSelector` or `useDispatch`, it will try to resolve the Redux store from *its own* module scope — not the host's. Depending on how `react-redux` is shared across remotes, this either silently uses a different store instance or throws at runtime. The prop-only boundary eliminates the ambiguity entirely: the component doesn't know a store exists.

The pattern this enforces is `Component.tsx` + `ComponentConnected.tsx`. The presenter (`Component.tsx`) is what goes into Storybook — it renders deterministically from props, with no async dependencies, no context providers required. The connected wrapper (`ComponentConnected.tsx`) lives in the host app and does exactly one thing: map store state to props and dispatch to callbacks.

> **What would break if we did it differently?**
> If you let UI components in the shared package import `useSelector`, you create a coupling between the component package's React/Redux version and the host's. Every time you upgrade `react-redux` in one place, you risk a version mismatch that shows up as a runtime `Cannot read properties of undefined` error — typically in production, after a deploy, not in development where federation is usually disabled.

The Storybook stories that landed this week ([shell] #42, [TripPlanner] #31, [BlogEngine] #32) are the *intended* executable proof of this constraint. A component that imports `useSelector` cannot be rendered in Storybook without a `<Provider>` wrapper around every story. A story that renders without a Provider is the test that ADR-0029's boundary is holding.

**But here is the honest gap:** that test is only running locally. There is no CI step that runs Storybook builds with a hard fail on a rogue `useSelector` import. The ADR documents the constraint. The stories demonstrate it. Nothing in the merge pipeline *enforces* it. The missing piece is an ESLint `no-restricted-imports` rule on `packages/ui/` banning `react-redux` at the lint level, paired with a CI Storybook build that would catch what lint misses. Until that exists, "the story will break before the PR merges" is a social contract, not a guaranteed gate.

**This demonstrates the *tooling for fast iteration* pillar** — one of the four core Frame OS / Samir Mody principles — in its current partial state. Storybook is doing useful work as a design contract, but the feedback loop is only as fast as the developer who remembers to check it locally. CI enforcement closes that loop.

### The NL instance spawning PR: what it's waiting for, and the answer to [shell] #5

[shell] #43 (`feat(shell): NL instance spawning + expanding header input`) has been open since Thursday. It extends `detectSpawnIntent` → `detectAction` in MetaOrchestrator — the ShellAgent's entry-point class that parses natural language intent into typed action objects for frame-agent dispatch — to handle both `spawn_instance` and `focus_instance` actions. It also passes the full instance list in request context so `frame-agent` can reference running apps by name. The expanding header input is the visible surface: when a user types a spawn intent, the input field expands inline rather than navigating away.

> **Why hasn't this merged?**
> [shell] #5 is an **open GitHub issue** (not a PR) tracking the store isolation architecture for multi-instance spawning. When two instances of cv-builder run simultaneously, their Redux slices need to be keyed by instance ID, not app type. #43 handles spawn intent detection and the UI surface but does not address slice keying. Merging #43 without a scope boundary means the feature appears to support multi-instance but breaks for exactly that case.

**The scope decision, stated explicitly:** [shell] #43 merges with a code comment and PR description update scoping it to **single-instance-per-app-type only**. Multi-instance Redux isolation is explicitly deferred to [shell] #5. This is the right call because the demo case (one cv-builder, one BlogEngine, one TripPlanner) is the immediate need, and the multi-instance refactor surface — keying every Redux slice by instance ID — is a meaningful change that deserves its own design pass. Estimate: not two hours. Closer to two days of careful slice surgery, which is why it should not be stapled to #43.

This decision was made here, Saturday, before the Monday merge window. It should be captured as an ADR or at minimum as a labeled comment in the PR before merge.
