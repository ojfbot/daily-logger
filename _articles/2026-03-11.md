## The decisions

### Decision 1: Rename cv-builder → Resume Builder now, not later

The repo slug stays `cv-builder` (GitHub URLs, CI environment variables, and ADR links remain stable); the npm package scope becomes `@resume-builder/*` and all display surfaces say "Resume Builder." This split identity — slug vs. display name — is documented explicitly in [core] `89698cc`. ADR-0018 ratifying this decision is not yet written; it appears in the suggested actions below.

**Why split slug from display name?** GitHub repo renames break clone URLs, CI environment variables, and any external links in ADRs or blog articles. The display rename — package scope, UI labels, Dockerfiles — costs one coordinated PR and zero broken links. Keeping the slug stable is the safer tradeoff at this stage.

The rename was coordinated with Gas Town Sprint 1 landing in cv-builder, so the module alias transition happened in the same PR window. Doing it separately would have created a period where Gas Town code shipped under the old namespace — a confusing diff history that would require a later cleanup commit. Coordinating the two PRs eliminates that window. This demonstrates **Tooling for fast iteration** (Samir pillar 2): the rename machinery was audited and applied in a single coordinated commit set rather than scattered across sprints.

### Decision 2: FrameBead `/api/beads` ships in cv-builder AND core-reader on the same day

Gas Town Sprint 1 implemented the same `GET /api/beads` interface in two repos. cv-builder returns `JobListing` objects mapped via `mapJobToBead()`; core-reader returns ADR manifests mapped via `parseADRsAsBeads()`. Both conform to `FrameBeadLike`.

**What does `FrameBeadLike` look like?** The interface is defined in the shared type package and cited by both mappers. The required fields include `id` (string, namespaced — e.g. `core-adr-0016` or a job listing ID), `type` (discriminant string), `status`, and `label`. The precise optional fields are in `ADR-0016` and the shared type file — link directly to `0016-framebead-work-primitive.md` for the canonical field list. If a required field is missing, TypeScript will catch it at build time; shape mismatches in optional fields surface at runtime when the shell tries to render a bead.

**Why ship both implementations simultaneously rather than proving one first?** The `FrameBeadLike` interface is the contract. If both implementations land on the same day against the same ADR-0016 spec, any deviation in the interface surface is caught immediately by comparison — two implementors are a better conformance test than one. If cv-builder had shipped alone and core-reader followed two weeks later, interface drift would be invisible until integration. This is **assistant-centric architecture** (Samir pillar 1) in the data layer: FrameBeads are the primitive that will let ShellAgent query heterogeneous app state through a uniform interface.

**What can the shell do with these endpoints right now?** Honestly: nothing automatically. The Mayor aggregation layer ([shell] #21) is unspecced and has no PR. Two `/api/beads` endpoints exist in production (core-reader confirmed, cv-builder deployment unconfirmed); nothing in the shell queries them yet. The activity feed — the UI surface that will display merged bead streams — is the payoff: a unified timeline in the shell showing active ADRs, live job applications, and future app states, queryable by ShellAgent via natural language. That surface doesn't exist yet; [shell] #21 is the prerequisite.

### Decision 3: Props-driven extraction for @ojfbot/shell — no Redux inside the package

`AppSwitcher` and `Header` are extracted into `packages/ui/` as pure, props-driven components. Redux state lives in `shell-app`; the extracted package receives only what it needs via props.

**What would break if we put Redux inside `@ojfbot/shell`?** Any app that imports `@ojfbot/shell` would need to install and configure the same Redux store. BlogEngine and TripPlanner each have their own stores. Forcing a shared store at the package level either merges three state trees (complex, error-prone) or requires three separate Provider instances that duplicate the shell's store shape. Props-driven extraction defers that decision entirely — the package is reusable without store coupling.

**What does this look like concretely?** `AppSwitcher` receives its registered apps list and active-app identifier as props; it calls no `useSelector` or `useDispatch` internally. `shell-app` owns the `appRegistrySlice` and passes derived values down. The `packages/ui/src/` file structure: `components/AppSwitcher.tsx`, `components/Header.tsx`, `index.ts` re-exporting both. When BlogEngine extracts its components, it follows the same pattern — any Redux-dependent state becomes a prop at the package boundary.

The Storybook hoisting fix (`c2fb056`) is the direct consequence of this extraction: Carbon Design System is now a shared dependency at the root, so Storybook and shell-app both resolve the same instance. Without `viteFinal` dedupe, two Carbon instances coexist and produce conflicting CSS tokens. This is a predictable extraction tax — it will recur when BlogEngine and TripPlanner extraction begins.

**When does [shell] #5 (Frame-wide Redux store strategy) become blocking?** It becomes blocking when the first BlogEngine or TripPlanner component needs to *write* to shell state rather than just receive it as props. A component that only reads data can be satisfied by props indefinitely. A component that dispatches actions — say, a TripPlanner "save itinerary to shell" button — needs access to the shell's dispatch, which is either passed as a prop callback (currently viable) or requires a shared store (the unresolved question). The trigger condition is the first cross-app write action, not the extraction pattern itself. That has not been attempted yet; [shell] #5 should be revisited when the first cross-app write action is specced.

### Decision 4: Vercel serverless bundle configuration for core-reader

Vercel's serverless functions only bundle files explicitly declared in `vercel.json`. The `decisions/adr/` directory — which `parseADRsAsBeads()` reads at runtime via `fs` — was not included in the initial deploy, so the endpoint returned empty results in production while working correctly locally.

**What did the broken version look like vs. the fixed version?**
- Broken: `vercel.json` had no `functions` block; the default bundle included only the function source.
- Fixed: `functions.includeFiles` declares `decisions/adr/**`, explicitly pulling those files into the serverless bundle at deploy time.

The one-line fix is straightforward; the diagnosis requires knowing that `vercel dev` gives the function access to the full local project tree, masking the constraint entirely. The rule generalises: **any file your serverless function reads via `fs` must be declared in `vercel.json`**. This is a dev/prod parity gap — `vercel dev` does not reproduce the production bundling constraint. A `vercel build` dry run in CI would catch this class of error before deploy. Whether that CI check now exists is not confirmed in the commit record — this is a live gap. Adding it and documenting the `fs` / `includeFiles` rule in CLAUDE.md are both open actions below.
