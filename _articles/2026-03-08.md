### [shell] #19 — multi-instance UI, session persistence, frame-agent env fix

This PR has been open since 2026-03-05 — the longest-lived unmerged change in the active set. It introduces the `AppInstance` hierarchy: `AppType → AppInstance → AppThread`. Before this PR, Frame tracks one instance of each app type. After it, the shell can hold three simultaneous cv-builder sessions and tell them apart.

The **Phase 4 roadmap item** — "multi-instance app launching from the chat bar" — has a hard dependency on this data model. You cannot route a ShellAgent natural language command to "the second trip planner" if the Redux store has no concept of a second trip planner. This PR is the prerequisite that unlocks Phase 4 entirely.

The **session persistence** piece is equally load-bearing: without it, a browser refresh destroys every open instance and the user rebuilds their workspace from scratch. That's not a minor annoyance — it breaks the "Frame as OS" metaphor at its most visible moment. What a user *sees* on return is the restored workspace rehydrating in place; there is no separate "restore" confirmation UI — sessions reload silently from persisted state. The visual representation of three simultaneous cv-builder sessions (what the shell actually renders under the new `AppInstance` hierarchy) is not yet documented in the PR; this is a gap worth closing before merge with at least a described layout or screenshot.

The **env fix** is the smallest change in the PR but the one most likely to cause a silent production incident: the specific frame-agent env var in question controls gateway authentication, and if it is undefined, the gateway silently falls back to no auth — accepting all requests and logging nothing unusual. There is not yet a documented test case that catches this before deployment. That test should be the first thing written Monday, before the compatibility shim, before the MF routing validation. What to grep for in a production incident: look for gateway requests succeeding with no `Authorization` header in the access logs.

The new surface area in [MrPlug] #25:

- **GitHub issue creation**: inspect any element on `*.jim.software`, trigger an AI-suggested badge, and a rich issue is created on the correct repo — with screenshots, acceptance criteria, and open-question fields.

- **MF-aware project routing**: the extension reads the Module Federation remote manifest at runtime to determine which repo owns a given UI element.

**On MF manifest reading mechanics**: MrPlug reads the `remoteEntry.js` manifest that Module Federation generates for each registered remote. This file is fetched at runtime from the host's known remote endpoints. The failure modes differ by mechanism: a network fetch for `remoteEntry.js` will fail with a 404 if the remote isn't deployed; a timing failure (remote not yet loaded when the manifest is read) will return an incomplete registry. Both failure modes need explicit handling in the PR before it exits draft. The manifest currently only covers registered remotes — any UI element owned by the `shell` host bundle itself (not a federated remote) needs a fallback rule. That fallback rule is not yet specified in the PR; the current options are (a) a modal asking the user to pick a repo, or (b) a silent default to `shell`. Silent defaults in write paths are exactly the failure mode the MF routing was designed to prevent — option (a) is the right answer, and it should be written down as a decision in the PR before promotion.

## The decisions

### Why MF-aware routing is the right abstraction for the issue-creation write path

The naive implementation of "create a GitHub issue from MrPlug" is: hardcode the repo, or let the user pick from a dropdown. Both options fail in a multi-remote Module Federation topology. An element rendered on `frame.jim.software` might belong to `cv-builder`'s remote bundle, `shell`'s host bundle, or eventually `BlogEngine`'s remote — and the user has no reliable way to know which, nor should they need to.

The Phase 5B approach reads the MF remote manifest at runtime. The manifest already exists — it's how Module Federation wires remotes to the host at load time. Repurposing it as a routing oracle for the write path is a zero-new-infrastructure solution. The tradeoff is narrow but real: the manifest only covers registered remotes. Any UI element that isn't federated — something owned by `shell` itself — needs a fallback rule. That fallback rule is currently unspecified in the PR. The right answer is an explicit user-facing prompt (not a silent default to `shell`), because silent defaults in write paths are exactly the failure mode this routing was designed to prevent.

**What breaks if you skip MF-aware routing?** Issues get filed against the wrong repo. A bug in cv-builder's Resume Generator panel gets opened on `shell`. The CI relay triggers the wrong Claude Code workflow. Both failures are silent — the issue and the workflow run successfully; they're just in the wrong place. Silent misrouting is harder to debug than a loud failure. That's the sentence that justifies the routing complexity to anyone who would reach for a dropdown instead.

**Blast radius of a routing failure on the Claude Code relay**: a misrouted issue is recoverable with a close and reopen. A Claude Code workflow triggered against the wrong repo depends on what that workflow does — if it creates branches, opens PRs, or modifies files, the unwind cost is non-trivial. That analysis belongs in the [MrPlug] #25 PR description before it exits draft.

This is **Pillar 4 — Security as emergent UX**: the routing logic isn't a security feature in the traditional sense, but it enforces a trust boundary (writes go to exactly the correct repo, not an approximate one) through a UX mechanism — automatic, invisible, correct-by-construction when the routing works and explicitly user-confirmed when it falls back.

### Why [shell] #19 has been open for three days without merging — and what the shim decision actually is

The multi-instance PR is not stalled because it's broken. It's stalled because it's a store-shape change. The `AppInstance` hierarchy touches Redux state that every sub-app reads. The options are:

- **Option A**: merge the new shape and simultaneously update all sub-app selectors. Atomic in a monorepo; impossible in a multi-repo Module Federation topology where each remote deploys independently.
- **Option B**: merge the new shape with backward-compatible derived selectors that project the old `AppType`-keyed shape out of the new `AppInstance` structure — keeping existing sub-app selectors working for one release cycle, then cutting the shim once remotes have updated.

Option B is the correct sequencing. "One release cycle" in this topology means: until cv-builder, BlogEngine, and TripPlanner each have a deployed version that reads `AppInstance`-keyed state. The coordination mechanism is a feature flag or a versioned selector contract documented in the shell repo — this does not exist yet and needs to be written as part of the PR.

**What "under review" meant, and what it should mean instead**: this decision is Jim's to make. It doesn't require sign-off from external reviewers. It requires writing the backward-compatible selector shim (derived selectors that wrap `AppInstance` state and return the old `AppType`-keyed shape), pushing it to the PR branch, and documenting the shim-cutting contract. That is the concrete work. The decision is already made above — what remains is execution.

The practical shape of the shim: add derived selectors in the shell store that accept the new `AppInstance[]` state and return a `Record<AppType, AppInstance>` map keyed by type (for sub-apps that only care about "is an instance of this type open"). Selectors that need per-instance granularity get updated in the same PR. Selectors in cv-builder, BlogEngine, and TripPlanner that read only type-level state continue working unchanged until those remotes are updated.

**What to look at first in [shell] #19**: the file that owns the store shape is the Redux slice for `instances` (exact filename not confirmed in this article — locate it in the shell package's `store/` directory). Every selector that currently reads `AppType`-keyed state is a compatibility target. The `/investigate` action item below will enumerate them.

This is **Pillar 1 — Assistant-centric architecture**: the multi-instance data model is not a UI nicety. It's the data primitive that lets ShellAgent refer to "the second trip planner" as a discrete entity. The store shape is the agent's world model. Getting it right — and getting the backward-compatibility layer right — is what makes Phase 4 (natural language multi-instance routing from the chat bar) buildable at all.
