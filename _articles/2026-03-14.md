This PR proves the `@ojfbot/shell` package boundary by replacing shell-app's direct local component usage with connected wrappers that import from `packages/ui`. `HeaderConnected` resolves Redux state and env vars before handing clean props to the extracted `Header`. `HomeScreenConnected` does the same for the tile grid.

*Why does the merge order matter?* `Header.tsx` and `HomeScreen.tsx` exist in `packages/ui`, but they're inert until shell-app actually imports and renders them from the package. Until #31 merges, Phase 1 is "extracted in isolation" — not "extraction proven at the seam." Module Federation boundaries are where silent failures live. The specific failure mode: a component that compiles cleanly and passes `pnpm build` in `packages/ui` can still produce a blank render or a stale cached chunk when imported across an MF boundary at runtime. `pnpm build` passing is a compile-time precondition, not the seam test. The actual seam test is loading the shell in a browser and confirming `HeaderConnected` renders with correct Redux-resolved props — not a blank component, not a cached pre-extraction chunk.

**[shell] #32 — Cross-domain fan-out + isolated synthesis in MetaOrchestratorAgent**

This is the engine of the hero demo. It wires up `MetaOrchestratorAgent` to fan out a natural-language query across multiple sub-app agents in parallel, then synthesize isolated per-domain responses into a single coherent answer.

**The hero demo query:** `"I'm applying for Berlin jobs, how does this affect my trip plan?"` — cv-builder's job analysis agent and TripPlanner's itinerary agent both fire, neither sees the other's response until synthesis.

*Why isolated synthesis?* If both agents share a context window during execution, each agent's reasoning is contaminated by what the other found — and this operates at the token-distribution level, not just the output level. A language model attending to Berlin-trip tokens from TripPlanner will weight Berlin differently in the job analysis response even before it reasons explicitly about the overlap. Isolation preserves the integrity of each domain's signal. Synthesis happens after both return — a reduce step, not a shared scratchpad.

This is the **assistant-centric architecture** pillar in practice. For a user, the difference is concrete: an assistant-centric architecture means the shell *coordinates specialists* rather than routing to a single handler. The Berlin-jobs query doesn't go to whichever sub-app is cheapest or most recently used — it fans out to the domain experts, preserves their independence, and synthesizes a response that neither agent could have produced alone. ShellAgent owns the synthesis contract. The same bet Dia makes when it decides whether to call one web agent or three.

*On the synthesis layer:* The article states #32 "depends on `POST /api/resumption` for the synthesis layer." This needs clarification. `POST /api/resumption` was described when it merged ([shell] #33) as generating a contextual conversation opener from existing history — a thread-resumption feature. That's a different job than synthesizing cross-domain fan-out results. If #32 is repurposing that endpoint for synthesis, that's a significant architectural decision that deserves an explicit sentence in the PR description. If #32 uses a separate synthesis endpoint, this article has conflated two things. **This ambiguity should be resolved in the #32 PR review before merge.**

*On the ADR:* The isolated synthesis pattern is the most architecturally significant decision currently open in this project. The written rationale — agents don't share context windows, synthesis is a reduce step, contamination operates at the token-distribution level — should exist as an ADR *before* #32 merges, not as a post-merge to-do. The alternatives considered (shared scratchpad, sequential chaining, single concatenated prompt) and why they were rejected should be on record. The reasoning in this article is the starting point; the ADR action item below names where it goes.

*On token cost:* The hero demo's showcase feature is also the project's highest-cost operation. Parallel LLM calls across two sub-app agents, plus a synthesis step, means at minimum three model calls per query. The current architecture has no stated token budget per hero-demo query and no stated behavior at scale (100 queries/day vs. 1,000). This is not a blocker for the Monday merge, but it is the first question a technical evaluator will ask when they run the demo. Own that number before they do.
