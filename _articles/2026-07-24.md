## The decisions

### Silicon-empires: stacked PRs with deliberate golden re-bakes as a first-class pattern — *tooling-for-iteration*

The claim system landed as four sequential stacked PRs (#22→#25), each with an explicit re-bake of snapshot goldens when serialization changed. Re-baking goldens is framed as a deliberate act in each PR description — not a failure signal, but evidence that structural change propagated correctly into the test oracle. Mechanically: the re-bake command updates the stored serialization snapshots; the verification step is diffing the new goldens against the previous ones and confirming every change maps to a known structural edit. A re-bake with unexplained diffs is a regression signal; a re-bake with fully-explained diffs is a green light. The alternative — freezing the goldens and patching tests to accept drift — would mask the serialization impact of each structural change and make the stacked PR sequence unreviewable.

### Playlog born from the first human playthrough — telemetry closes the loop for agents — *model-behavior-as-design*

PR #27 (S38) added deterministic session replay and experience telemetry directly because the first human playthrough on 2026-07-22 found the game 'a little overwhelming and the machine won.' What happened: the player attempted to build out a crew and stake multiple claims in the first few turns; the AI opponent had already seeded ACTIVE claims across high-yield territory, and the contest mechanic resolved against the player before the DECAY phase gave any recovery window. The game was not broken — but it was tuned for agents, not humans with a first-session learning curve. The playlog captures the full session as a deterministic replay hash plus structured telemetry events (claim staked, contest triggered, lifecycle transition, score delta). Without it, agent-driven iteration has no ground truth to optimize against. This is the first replay artifact; whether the replay hash round-trips correctly against the 2026-07-22 session is explicitly unverified — the /validate action below is the first run, not a confirmation of a known-good state.

### Switchboard S1: byte-for-byte SSE passthrough as the consumer-adoption contract — *assistant-centric*

The FastAPI proxy forwards Anthropic SSE chunks without buffering or re-framing, so any existing SDK consumer can switch by base-URL swap alone. Mechanically: the proxy opens a streaming HTTP connection to Anthropic, reads SSE frames off the wire, and writes them to the consumer response without accumulating. If the upstream connection drops mid-chunk, the consumer sees a truncated SSE stream and an abrupt close — the same behavior they would see calling Anthropic directly. There is no synthetic error envelope in S1. This is the minimal interface that avoids lock-in while enabling per-app budgets and token-bucket rate limits to be layered on top in S2+. The alternative — buffering full responses before forwarding — would have enabled response inspection and retry logic, but would have broken streaming UX and forced SDK changes on consumers. That tradeoff is deferred to a future slice if needed. S2 (provider adapter layer) is marked ready in the roadmap, meaning the next slice has a filed northstar; acceptance criteria will be written in the /scaffold session, not assumed to already exist.

### Mirrorworld: real USGS elevation data as the first rendered artifact, not synthetic terrain — *tooling-for-iteration*

S2 chose a real 3DEP DEM tile (Mount Rainier) over synthetic heightfield data for the first rendered scene. Recognizable real-world relief — glacial valleys, Little Tahoma — is the concrete proof that the data pipeline (USGS → local tile → three.js heightfield) is wired correctly end-to-end. Synthetic terrain would pass the renderer test without proving the data ingestion path. The alternative was procedural noise generation (e.g. Perlin/simplex heightfield): zero infrastructure cost, visually convincing, but it would have validated only the three.js rendering layer, leaving the USGS fetch, tile projection, and DEM-to-heightfield transform untested until a later slice. Choosing real data first means a broken ingestion path fails immediately rather than hiding behind a passing renderer. Mirrorworld is a geospatial app running inside the Frame OS shell — the long-term vision is navigable real-world terrain as an interactive surface. The S2 artifact is a local dev render; no public URL exists yet.
