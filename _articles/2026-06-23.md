## The decisions

### Stacked PR strategy for morning-cockpit redesign: seven slices instead of one branch — *tooling-for-iteration*

The redesign is decomposed into seven sequentially-dependent PRs (#1–#7) plus an independent docs PR (#8). Each slice targets a named base branch so reviewers see exactly what changed in that layer alone — tokens, then pods, then briefing console, then briefing generator, then fleet/delivery read-models, then critical-path board, then chat rail polish. The alternative — one giant feature branch — would make bisection on a regression nearly impossible. If a visual defect appeared in the chat rail, there'd be no way to confirm whether it originated in the token layer or the pod restyle without manually reverting an undivided branch. The slice boundary makes that question answerable in minutes.

The PR/slice numbering mismatch (PR #6 must merge after PR #7) is a one-time artifact of the order in which PRs were opened, not a systematic risk in the stacked-PR workflow. The authoritative sequencing signal is the base branch, not the PR number. PR #6's base is `feat/cockpit-redesign-s5-critical-path` — which means #7 must land first. Going forward, the base-branch name is the source of truth; PR numbers should be treated as identifiers, not order indicators. The roadmap table below is sorted by merge order, not PR number, to make this unambiguous.
