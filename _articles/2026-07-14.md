## The decisions

### Delivery pane as a read-only projection, not a mutable editor — *tooling-for-iteration*

The morning-cockpit Delivery pane renders northstar gaps, slice pipeline state, and movement history through a dedicated `/api/delivery` endpoint — it is explicitly read-only. This keeps the cockpit in its designed role as a consumption surface for the delivery read-model, not a write path. If the pane were writable, mutations would bypass the canonical record-movement tooling and create divergence between the `status.jsonl` ledger and whatever the UI has locally committed. The `/api/delivery` endpoint is live; the pane is rendering in the deployed cockpit as of Monday's merge.

### Two-commit strategy for the S5 odometer turn: canonical file + ledger append as separate PRs — *tooling-for-iteration*

The S5 northstar movement was split across two PRs: #26 (Delivery pane, containing the movement proposal) and #27 (the canonical-file application via `record-movement.mjs --apply`). This separation keeps proposal and acceptance in distinct, independently-reviewable commits. Had both landed in one PR, a failed merge of the UI work would have blocked the score update, and vice versa — decoupling them means either can be reverted without clobbering the other. The failure mode this guards against is real: PR #26 touched React component code and routing; PR #27 touched only `status.jsonl` and the northstar registry file. These are different risk profiles and should be independently revertable. One open question this design defers: the two PRs together constitute a distributed transaction with no explicit rollback. If #26 merges and #27 fails, the ledger is in a known-incomplete state. There is currently no compensating mechanism documented — that gap should be named in an ADR.

### OPAV S22(a): capture-quality verifier excludes maintenance traps by design — *model-behavior-as-design*

OPAV — Observe, Propose, Act, Verify — is the feedback loop that measures whether the assistant is meaningfully improving over time by tracking skill invocations: moments where a catalogued skill was genuinely used in a session. The S22(a) verifier addresses two distinct correctness properties. Path-independence: the capture-quality signal must be identical whether a skill was invoked via the CLI recorder path or the Stop-hook path — before this fix, the two paths could disagree, making the aggregated count meaningless. Maintenance-trap exclusion: routine file edits (any write to a tracked path, triggered by the Stop-hook) were registering as `skill:acted` events because the hook fires on file writes generically. This inflated the acted denominator — the count of sessions where a skill was 'used' — making the suggestion funnel appear healthier than it was. The verifier proves equivalence between paths via an assertion in the commit, and excludes maintenance-trap writes by pattern-matching against a defined set of non-skill write signatures. Both properties are prerequisites before the funnel numbers can be treated as signal rather than noise.

### End-to-end skill-use capture wiring: honest PR audit trail as the acceptance criterion — *model-behavior-as-design*

The skill-use capture end-to-end pass (`bf19d12`) treats 'honest PR audit trail' as its single acceptance criterion — inline reads must be visible in the log, and the acted path must be invocable, but only when a genuine skill invocation has occurred. This framing matters because the instrumentation is self-referential: it measures the very behavior that determines whether the OPAV loop is working. A dishonest signal here corrupts every downstream metric. 'Downstream metric' is specific: the suggestion-funnel ratio (skills proposed / skills acted) drives the weekly OPAV review, which determines which skills get promoted, deprecated, or rewritten. If the acted denominator is inflated by maintenance-trap writes, low-quality skills appear to be working and never get pruned. The acceptance criterion of 'honest PR audit trail' means: the commit itself must demonstrate a real invocation was captured, not a synthetic one.

### Orphaned northstar entries resolved by pickup PRs rather than backfill scripts — *tooling-for-iteration*

The BlogEngine and shell northstar entries were voice-confirmed on 2026-06-28 but orphaned — never committed to the registry. Monday's resolution was a pair of pickup PRs (BlogEngine #58 and the shell commit) rather than a retroactive backfill script. Pickup PRs preserve the exact confirmation date in the commit message and produce a clear audit trail in the registry history; a backfill script would obscure when the confirmation actually happened, making the registry history misleading for anyone reconstructing the project timeline. The pattern of confirmed things sitting unrecorded appears more than once this week — l1-blogengine and l1-shell orphaned for 16 days, BlogEngine #57 open 33 days. This is not a coincidence. The project is accumulating a class of 'confirmed but not committed' debt that pickup PRs resolve individually but don't prevent structurally.
