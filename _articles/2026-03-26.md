## The decisions

### Decision 1: Remove automatic LLM calls — explicit over ambient

The most consequential decision in Wednesday's work is the one that *removed* behaviour rather than adding it.

GroupThink's original design called LLM endpoints automatically on `chrome.tabs.onCreated`, `onUpdated`, and `onActivated` events. The intent was an always-current grouping state — tabs reorganise themselves as you browse.

**Why we removed it:** Three problems compound. First, cost — every navigation event becomes a billable API call with no user intent behind it. Second, latency — the Claude response time is 1–3 seconds; a tab event that fires while you're mid-keystroke produces a visible regroup that fights the user's current action. Third, trust — silent background mutations violate Pillar 4 (**security as emergent UX**). The user never consented to the model reorganising their browser state; they just opened a tab.

**What would break if we kept it:** Users would experience involuntary tab reorganisation mid-session, every `env.json` race condition would surface in production (since calls fire before config is settled), and token costs would scale with browsing volume rather than intent volume.

**What the UX looks like now:** On install, GroupThink presents an empty state. Tab grouping does not occur until the user explicitly triggers it via the Refresh button, specificity slider, or chat input. This is the correct first-session experience — the model acts when asked, not when opportunity arises. The open question (not yet answered by Wednesday's commits) is whether there is a sufficient empty-state prompt or onboarding nudge. That UX gap should be surfaced in the next GroupThink sprint.

> **Why does this matter architecturally?**
> The distinction between *ambient* and *explicit* model invocation is one of the core design questions in any AI-native product. Ambient invocation (always watching, always acting) feels powerful in a demo and hostile in daily use. Explicit invocation (user presses a button, model responds) feels slower in a demo and trustworthy in daily use. GroupThink's original design made the demo-friendly choice. Wednesday's fix makes the daily-use choice.
>
> The Dia parallel is real but worth being precise about: Dia operates at the browser chrome layer with persistent session context; GroupThink is a tab organiser with no persistent state between sessions. The invocation *philosophy* is the same — the model acts when asked — but the product constraint differs. GroupThink has no session memory to make ambient invocation safe even in principle. Dia at least has the context to know whether a background action would feel continuous or intrusive. The architectural claim here is narrow: both products converge on explicit invocation, and the reasons overlap (cost, latency, trust), even though the full design space is different.

Until a written ADR is linked from the GroupThink repo, the rationale is one contributor turnover away from being lost. That is a gap, named plainly.

### Decision 2: Per-method token limits rather than a global cap

Commit `9e3d2ef` assigned separate token budgets to each LLM invocation type — grouping, chat, label generation — rather than a single shared ceiling.

**Why per-method:** These calls have different output shapes. The grouping call produces a JSON structure whose size scales with tab count. The chat call produces prose whose length should feel conversational. The label call produces a 2–4 word string. A global cap either starves the grouping call on large tab sets or over-allocates for label generation. Per-method limits let each call be tuned to its actual output contract.

**The specific limit values are not surfaced in this article** — that is a gap. The values live in commit `9e3d2ef`. They should be extracted into a named config constant (e.g. `TOKEN_BUDGETS` in a dedicated file) so that anyone debugging a truncated grouping response on a large tab set can find the ceiling they're bumping against without reading a diff. That extraction is the right follow-on to this commit.

> **What does this mean in practice?**
> This is Pillar 3 — **model behavior as design discipline** — applied at the budget level. Token limits are not an infrastructure concern; they are a product decision about what each model interaction is allowed to say. Setting them per-method means you can reason about the cost and quality of each interaction independently, and version those limits as first-class configuration.

**On cost:** Wednesday's changes — removing ambient calls and capping per-method token spend — are a cost-control architecture. The article has not closed the loop with a number. What a typical session costs under the explicit model (e.g. five Refresh calls and three chat queries) is not yet benchmarked. That benchmark should be the first output of the `/investigate GroupThink background.ts` work, because per-method limits are only correctly tuned if you know what "correctly tuned" costs.

### Decision 3: Step 7.2 — SYSTEM_PROMPT registration as a required checklist item

The `/scaffold-app` checklist previously ended at "add repo to `REPOS` sweep array." That was sufficient for context *collection* but not for context *understanding*. The article generator sweeps repos to pull commits and PRs, but the system prompt is where it learns what each repo *is* — its purpose, port, relationship to Frame, and role in the pitch.

**The fix is structural, not reactive:** rather than patching the omission per-repo, we added Step 7.2 to the canonical checklist so every future repo registration is complete by definition. This is Pillar 2 — **tooling for fast iteration** — applied to the tooling itself: the scaffolding process is a product that must be kept correct.

**Why GroupThink work happened before the Phase 1 npm publish gate:** GroupThink had active UX regressions (black screen, ambient LLM calls, cold-start crash) that would have made it unreliable in a demo context. The stability sprint was the right call because a broken sub-app contaminates the Frame pitch regardless of whether the shell is published. The npm publish gate requires an uninterrupted planning block; the GroupThink fixes were addressable in a focused sprint. That said, the Phase 1 gate has been "explicitly unstarted" for three sprint reports — naming the reason for Wednesday's sequencing does not resolve that debt. It is the highest-priority item in "What's next," ordered first.
